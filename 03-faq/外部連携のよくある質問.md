# よくある質問: 外部連携

> **2026-07-02更新**: Shopifyは接続済みストアの詳細画面、Shopify側アプリ表示、ロケーション接続、商品/在庫一括同期の認証情報エラー、Shopify注文 `#1008` / `#1009` の流入まで実機確認済みです。アクセストークン9種が未設定でも、ShopifyからSQへの注文作成・顧客作成/更新・返金・キャンセル受信は動作確認済みです。ただし未連携商品では出荷/返品/売上実績化は反映されず、SQからShopifyへ送る商品・在庫・価格などの同期と、正式連携SKUでの出荷/返品反映は認証情報設定後の再検証が必要です。

---

## Q. SQはどのサービスと連携できますか？

A. SQから直接設定できる連携先は以下のとおりです。

**チャネル連携（サイドナビの「チャネル」セクション）**

| 連携先 | 主な用途 |
|:--|:--|
| Shopify | ECサイトの注文・商品・在庫データを同期する |
| OmnibusCore | 商品同期・在庫連携・下書き注文の流入 |
| スマレジ | POS（実店舗）との商品・在庫・注文データを同期する |
| リテールポータル | 店舗スタッフ向けポータルとの接続・下書き注文管理 |

**外部サービス連携（設定 > 外部連携グループ）**

| 連携先 | 主な用途 |
|:--|:--|
| ロジザード（ロジザードZERO） | WMS（倉庫管理システム）との連携 |
| Recustomer | 返品・交換フローの外部委託 |
| アプリ（APIキー） | SQ Admin APIへのアクセス権限を持つアプリを作成する |

## Q. スマレジ連携を設定するための前提条件は何ですか？

A. スマレジ連携を利用するには、以下の準備が必要です。

1. **スマレジ側でアプリをインストールする**（スマレジ連携フォームの上部に「スマレジ連携を利用するには、スマレジにアプリをインストールする必要があります。」というメッセージが表示されます。2026-06-28実機では、SQフォーム内にスマレジ側インストール手順への外部リンクは確認できません）
2. **スマレジの契約IDを用意する**（スマレジ連携フォームの必須項目です）
3. **SQ側でテナントが1件以上登録されていること**（設定 > テナント）

**スマレジ連携作成フォームの必須項目**

| 項目 | 必須/任意 |
|:--|:--|
| 名前 | 必須 |
| テナント | 必須 |
| 契約ID | 必須 |
| プラン | 必須（デフォルト: リテールビジネス） |

プランの選択肢は「スタンダード / プレミアム / プレミアムプラス / フードビジネス / リテールビジネス」の5種類です。

関連: [スマレジ連携](../01-by-feature/スマレジ連携.md)

## Q. スマレジ連携の在庫同期の方向は選べますか？

A. はい、3つの選択肢から選べます。

| 選択肢 | 内容 |
|:--|:--|
| スマレジからSQへ在庫を同期する（デフォルト） | スマレジ側の在庫数をSQに反映する |
| SQからスマレジへ在庫を同期する | SQ側の在庫数をスマレジに反映する |
| 在庫を同期しない | 在庫の双方向同期を行わない |

スマレジ連携の作成フォームおよび詳細画面の「在庫設定」セクションにあるコンボボックスで設定します。

商品連携設定では「カタログ」を選択します。外部システムへ在庫数を出す対象商品を指定する用途では、カタログを商品グループとして作成し、スマレジ連携などの外部連携設定側で参照します。CSVの在庫エクスポートとは別物です。

2026-07-02に設定済みスマレジ連携 `TEST-SQ` では、外部アクセストークン設定済み表示とスマレジ店舗候補の取得までは確認できました。スマレジPOS管理画面側では店舗1件、商品84件、取引履歴0件、会員0件、アプリ設定0件を確認しています。ただし、SQ側 `TEST-SQ` の店舗設定は0件、商品管理/顧客管理も0件のため、スマレジ側84商品がSQ連携で作成されたものとは断定しません。商品・在庫・注文・顧客がスマレジ側またはSQ側へ実際に同期されるタイミング、頻度、エラー時の挙動は未確認です。スマレジの在庫一覧・受注管理・アプリマーケットは別クライアントの追加ログインが必要でした。

## Q. Shopify連携で事前に必要な準備は何ですか？

A. SQ側のマスタに加えて、Shopifyストア側にSQコネクターアプリ（SQ Connect）を先にインストールしておく必要があります。

| 必要なもの | 説明 |
|:--|:--|
| SQ内のテナント | Shopify連携フォームの必須項目 |
| カタログ | Shopify連携で参照する対象商品グループ |
| ロケーショングループ | Shopify注文の在庫引当元 |
| Shopifyショップドメイン | `shop.myshopify.com` 形式 |
| SQ Connect | Shopify側にインストールするコネクターアプリ |

SQ Connectが未インストールの状態でSQ側の「連携する」を実行すると、連携エラーになります。先にShopify側でアプリをインストールしてください。

また、公開アプリのSQ Connectは本番SQ（`sq.stackservice.com`）向けです。staging環境（`sqstackstaging.com`）へ接続する場合は、staging向けの別アプリとインストールリンクが必要です。

関連: [Shopify連携](../01-by-feature/Shopify連携.md) / [販売チャネルを接続する](../02-by-task/販売チャネルを接続する.md)

## Q. Shopifyに接続済みなのに、商品や在庫の同期が失敗するのはなぜですか？

A. Shopifyアプリのインストールとストア接続だけでは、商品・在庫などの同期用認証情報は揃いません。SQのShopify連携詳細にある `アクセストークン` タブで、データ種別ごとにクライアントID/シークレットを設定する必要があります。

2026-07-02の実機確認では、アクセストークンタブに次の9種類があり、すべて未設定でした。

| 種別 |
|:--|
| 商品 |
| 在庫（絶対値） |
| 在庫（差分値） |
| 翻訳 |
| 注文 |
| 出荷指示 |
| ディスカウント |
| マーケット |
| リカバリー操作 |

実際に `一括同期 > 商品情報を一括で同期する` を実行すると、内部エラーは `ProductSet 用の認証情報が設定されていません` でした。`一括同期 > 在庫を一括で同期する` は、ロケーションを選択して実行すると `InventorySet 用の認証情報が設定されていません` で停止しました。

この失敗時、Shopify側の商品一覧は17件のままで、SQの `UNIQLO` カタログ対象5商品は作成されていませんでした。つまり、認証情報エラーではShopify側へ部分反映されない状態を確認しています。

## Q. Shopifyのアクセストークンが未設定でも、注文や顧客はSQに入りますか？

A. 2026-06-29〜2026-07-02の実機確認では、入りました。

Shopify連携詳細の `アクセストークン` タブにある9種すべてが未設定でも、Shopify注文 `#1001` / `#1002` / `#1003` / `#1008` / `#1009` はSQ注文として取り込まれました。Shopify顧客 `sq-sync-test-20260629-1436@example.com` の作成とタグ更新もSQ購買顧客へ反映されました。Shopify上のキャンセル/返金はSQ注文のキャンセル済み・返金済みに反映されます。ただし、2026-07-02の `#1008` 返金は注文情報内の決済履歴には表示されましたが、注文内 `返金履歴` タブは空でした。

一方で、Shopify側に同じSKU文字列の商品を手作成しても、SQの既存SKUには自動紐づきしませんでした。SQ注文行は `新規` 表示になり、Shopifyで発送済みにしてもSQ注文は `未出荷` のまま、出荷指示も作成されませんでした。2026-07-02に `#1009-R1` のShopify返品を作成しても、SQ返品一覧には表示されませんでした。正式に連携済みSKUとして紐づけた場合の出荷/返品反映は、商品・在庫系の認証情報設定後に再検証が必要です。

つまり、現時点の切り分けは次のとおりです。

| 種別 | 状態 |
|:--|:--|
| Shopify→SQの注文・顧客・返金・キャンセル受信 | アクセストークン未設定でも実機確認済み。返金は決済状況/注文情報内の決済履歴と、返金履歴タブの表示を分けて見る |
| Shopify→SQの出荷・返品受信 | 未連携商品では未反映。正式連携SKUで再検証が必要 |
| SQ→Shopifyの商品・在庫・価格などの送信 | アクセストークン未設定では認証情報エラーで停止 |

## Q. Shopify連携のデータ連携チェックをOFFにすると同期は止まりますか？

A. 項目によって挙動が違いました。2026-06-29の実機確認では、注文・顧客・キャンセルはOFFで反映が止まりましたが、返金はOFFでもSQへ反映されました。

| OFFにした項目 | 実機結果 |
|:--|:--|
| 注文データの同期を有効にする | Shopify注文 `#1004` はSQに取り込まれなかった |
| 顧客データの同期を有効にする | Shopify顧客はSQ購買顧客に取り込まれなかった |
| 返金操作を連携する | OFFでもShopify返金後にSQ注文 `#1005` が `返金済み` になった |
| キャンセル操作を連携する | Shopify注文 `#1006` をキャンセルしてもSQでは `支払い済み` のままだった |
| 出荷操作を連携する | Shopify注文 `#1007` / `#1008` を発送済みにしてもSQでは `未出荷` のまま。ただし正式連携SKUなしの条件付き |
| 返品操作を連携する | Shopify返品 `#1007-R1` / `#1009-R1` はSQ返品一覧に出なかった。ただし正式連携SKUなしの条件付き |

検証後、6チェックはすべてONへ戻しています。返金OFFは「必ず返金を止める設定」とは案内せず、抑止が必要な場合は現行挙動をStack社へ確認してください。

## Q. Shopify側ではSQ連携はどう見えますか？

A. 2026-06-29の検証環境では、Shopify管理画面のインストール済みアプリに `erp-stg` が表示され、埋め込みアプリの表示名は `SQ (エス・キュー)` でした。

埋め込みアプリには、連携対象として `商品データ`、`価格データ`、`在庫データ`、`注文データ`、`顧客データ` が表示されます。データ連携状況は `成功 完了 / 正常`、カタログは `UNIQLO - 5点の商品が連携されます。` と表示されました。

ただし、このShopify側画面には、SQの `アクセストークン` タブへ入力するクライアントID/シークレットの発行・表示導線は確認できませんでした。

## Q. Shopify連携の「会員証バーコードのフォーマット」とはどういう設定ですか？

A. Shopify連携の設定フォームにある任意項目で、**Shopifyから流入する購買顧客の会員証バーコードをどの形式で生成するか**を指定します。

| 選択肢 | 内容 |
|:--|:--|
| Shopify ID（数値部分）（デフォルト） | ShopifyのユーザーIDの数値部分をバーコードとして使用する |
| JAN-13コード（モジュラス10/ウェイト3方式） | JAN-13規格のバーコードを生成する |

設定はShopify連携の作成フォーム（ストア名・ショップドメイン・テナント・カタログ・ロケーショングループなどの必須項目と合わせて設定します）にあります。

関連: [Shopify連携](../01-by-feature/Shopify連携.md)

## Q. 連携を設定したあと、在庫や注文はいつ同期されますか？

A. 開発元の説明では、連携先ごとに次の性質で同期されます。

| 連携先 | 主な連携データ | 連携の性質 |
|:--|:--|:--|
| Shopify | 注文・在庫・商品・顧客 | リアルタイム |
| WMS（倉庫管理システム） | 出荷指示・発送実績・入庫 | バッチ / リアルタイム |
| POS（店舗） | 売上実績・店舗在庫 | リアルタイム |
| 既存ERP（基幹システム） | 会計仕訳・原価・マスタ | 日次バッチ など |

> この表は開発元の製品説明に基づくものです。具体的な反映タイミングや手動トリガーの有無は、接続環境での確認ができ次第このFAQを更新します。
