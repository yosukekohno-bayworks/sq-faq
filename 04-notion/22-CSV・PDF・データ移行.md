# 22. CSV・PDF・データ移行

SQの「CSVインポート」「CSVエクスポート」「PDFエクスポート」の3メニューを取り扱うエリアです。商品・在庫・価格・ポイント・売上実績などを**一括で登録・更新・出力**する機能で、手作業では時間がかかる大量データの取込みと取り出しを効率化します。このエリアでは、**何が一括処理でき、インポートが失敗したときにどう確認するか**を整理して説明します。

## このエリアで学べること

- CSVインポートで一括登録・一括更新できるデータの対象（9グループ・22カテゴリ）
- CSVエクスポートでCSV出力できるデータの対象（5グループ・9カテゴリ）
- PDFエクスポートの対象（納品書）と現在の実装状況
- カテゴリごとに異なるインポートフォームの構成（ファイルのみ／ラジオ選択／ルール選択／検索型選択）
- インポートが「検証 → 実行」の2段階で進むことと、失敗時の確認方法
- 実行履歴・ダウンロードの確認方法

## 機能概要

| メニュー | 画面URL | 内容 | 状態 |
|:--|:--|:--|:--|
| CSVインポート | `/admin/csv_import` | CSVでデータを一括登録・更新（9グループ・22カテゴリ） | 実装済み |
| CSVエクスポート | `/admin/csv_export` | データをCSVで出力（5グループ・9カテゴリ） | 実装済み |
| PDFエクスポート | `/admin/pdf_export` | 出荷指示の納品書PDFカテゴリを表示 | 納品書カテゴリページは履歴画面。ページ自体に新規作成/任意生成ボタンは確認できず、作成直URLは存在しない画面。出荷一覧の一括操作 `アクション > 納品書（PDF）` から履歴作成までは確認済み |

- 3メニューともサイドナビの専用エントリから開きます。ホーム（`/admin`）のクイックリンクからも遷移可能です（`CSVでデータをインポートする` / `CSVでデータをエクスポートする` / `PDFでデータをエクスポートする`）。
- CSVインポートは**登録だけでなく更新**にも使います。既存データを上書きする場合は、テンプレートに沿ったCSVを新しく取り込みます。
- インポートは保存後、**「検証」→「実行」の2段階**で処理されます。検証でエラー行があると、実行前に失敗内容を確認できます。
- CSVエクスポートには「エクスポート」という名前のボタンはなく、**各カテゴリ行のリンク**をクリックして対象カテゴリの画面に遷移します。
- CSVエクスポートの「在庫」カテゴリは、外部連携の在庫同期とは別機能です。実機上、在庫CSVエクスポートの条件はロケーション指定のみで、カタログ指定はありません。
- PDFエクスポートは現在「納品書」の1カテゴリのみです。カテゴリページ（`/admin/pdf_export/pdf_export_operation_packing_slips`）は表示されますが、このページ自体に新規作成/任意生成ボタンは確認できません。新規作成画面（`/create`）は存在しない画面です。2026-07-01に、出荷一覧で対象行をチェックして `アクション > 納品書（PDF）` を実行すると、このカテゴリページに履歴行が作成されることを確認しました。ただし `#IO-1047` / `#IO-1046` はどちらも `重大 / 失敗` になり、実ファイルはダウンロードできませんでした。

#### 3メニューの関係とインポートの状態遷移

```
┌─────────────────────────────────────────────────────────────┐
│  CSVインポート        CSVエクスポート      PDFエクスポート   │
│  （取り込む）         （取り出す）         （取り出す/PDF）   │
└─────────────────────────────────────────────────────────────┘

CSVインポートの処理フロー:
  保存する → 検証(実行中) → 検証完了 ─┬─ 実行する → 実行完了（反映）
                                       │              └─ 完了後は実行ボタンがdisabledで残る
                                       │
                                       └─ 検証失敗 → 失敗詳細画面で原因確認 → CSV修正して再取り込み

CSVエクスポート / ヤマトB2クラウド:
  エクスポート開始 → 処理中 → 完了 → ダウンロードリンク
  （ヤマトB2は出荷管理画面起点。画面文言上は完了後メール通知）
```

## 画面・項目の説明

> **フォームに出てくる用語のメモ**（初出のみ）
> - **テナント（事業部＝運用ルール分離単位。詳細は 03.組織・通知）**: CSVエクスポート等のフォームで選択する事業区分。テナント0件状態は既存マスタ削除が必要になるため検証対象外です。候補が出ない場合は先に「設定 > テナント」を確認してください。
> - **ロケーション**: 在庫の置き場所。場所種別は「倉庫」か「店舗」の2種のみ。倉庫は在庫保管・発注入荷先・出荷元・リテールポータル連携の在庫ロケーション、店舗はリテールポータル連携の店舗ロケーション・店舗受取の受取ロケーションとして扱う。手動の在庫依頼・移動伝票では倉庫/店舗の両方が関与するため、画面ごとの選択肢は04.基本マスタで確認する。
> - **SKU（商品バリエーション）**: 商品の色・サイズ等のバリエーション1つ1つ。在庫・価格はSKU単位で持つ。
> - **販売価格ルール／予約販売ルール／販売閾値ルール**: いずれも販売設定で**事前作成が必要**なルール。CSVインポート・エクスポートでは、そのドロップダウンから既存ルールを1つ選ぶ（ルール自体はここでは作れない）。

> **2026-06-29更新**: Shopifyは接続済みストアで注文・顧客・返金・キャンセル受信まで確認済みです。ただし、CSV/PDFの各インポート・エクスポートがShopify連携データをどう出力/取込するか、SQ→Shopify送信と連動するかは個別検証が必要です。

#### テンプレートの有無

テンプレートがあるカテゴリでは、一覧画面に `テンプレート` ボタンが表示されます。テンプレートは Google スプレッドシート形式で別タブで開きます。

- テンプレートがないカテゴリ（6種）: 販売価格 (通常) / 原価 / 販売閾値 / DHL / キャンペーン対象商品 / 基準ランク

> **カタログ商品テンプレート**: 2026-06-28実機確認時点の列は `product_code,command` です。`product_code` には既存商品の商品コード、`command` にはカタログへ追加する場合 `NEW`、カタログから削除する場合 `DELETE` を入れます。

#### インポート実行フォームのパターン

カテゴリによってフォーム構成が異なります。

| パターン | 構成 | 該当カテゴリ |
|:--|:--|:--|
| A：ファイルのみ | CSVファイルをアップロードして `保存する` | 商品 / 商品バリエーション / 商品メタフィールド / 商品バリエーションメタフィールド / 原価 / 在庫移動伝票 / ヤマトB2クラウド / DHL / 管理ユーザー / ロケーション |
| B：ラジオ選択 + ファイル | 処理方法をラジオで選んでからCSVアップロード | 商品画像 / 商品バリエーション画像 / 販売可能在庫 |
| C：ルール選択 + ファイル | ルールをドロップダウンで選んでからCSVアップロード | 販売価格 (通常) / 販売価格 (セール) / 販売閾値 / 予約販売 / 基準ランク |
| D：検索型選択 + ファイル | 検索＋ダイアログで対象を選んでからCSVアップロード | キャンペーン対象商品 |
| E：カタログ選択 + ファイル | カタログをドロップダウンで選んでからCSVアップロード | カタログ |

##### パターンBの選択肢

| カテゴリ | ラジオボタンの選択肢（デフォルト → 変更可） |
|:--|:--|
| 商品画像 | 「画像を追加する」（デフォルト） / 「画像を上書きする」 |
| 商品バリエーション画像 | 「画像を追加する」（デフォルト） / 「画像を上書きする」 |
| 販売可能在庫 | 「絶対値で反映する」（デフォルト） / 「差分値で反映する」 |

> **画像の追加／上書きの違い**: 「画像を追加する」＝既存画像を残したまま新しい画像を追加。「画像を上書きする」＝当該商品/SKUの既存画像を削除して置き換え（分析記録に基づく）。商品画像を一新したいとき以外は追加（デフォルト）を使います。複数画像時の一部上書きなど詳細な置換範囲は追加確認。

> 販売可能在庫の画面説明原文: 「現在の在庫数が10のとき、CSVで20を指定すると、絶対値で反映した場合の在庫数は20になります。差分値で反映した場合の在庫数は30になります。」

##### パターンCのフィールドラベル

| カテゴリ | フィールドラベル |
|:--|:--|
| 販売価格 (通常) | 販売価格ルール |
| 販売価格 (セール) | 販売価格ルール |
| 販売閾値 | 販売閾値ルール |
| 予約販売 | 予約販売ルール |
| 基準ランク | テナント |

- **各ルールの運用上の意味**: いずれも販売設定で事前作成したルールを1つ選び、そのルール配下のSKUへCSV値を反映させます。
  - 販売価格ルール = 通常/セール価格をSKU別に束ねる設定（事前作成必須）
  - 販売閾値ルール = 在庫の閾値制御の設定
  - 予約販売ルール = 在庫0でも受注を続ける設定
  - 基準ランクのテナント = 基準ランクはテナント単位で顧客に設定されるため、対象テナントを指定します

#### ポイント一括加算・一括減算フォームの固有項目

![ポイント一括加算のインポートフォーム（テナント・顧客IDの種別）](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/15-csv-import-point-plus-create.png)

| カテゴリ | 項目（UIラベル） | 説明 | 選択肢 |
|:--|:--|:--|:--|
| 一括加算 | テナント | 加算対象のテナントを選択する | セレクトボックス（プレースホルダ「選択してください」） |
| 一括加算 | 顧客IDの種別 | CSV内の顧客識別子の種類を指定する | 「SQが発番した顧客ID」 / 「外部システムが発番した顧客ID」 |
| 一括減算 | テナント | 減算対象のテナントを選択する | セレクトボックス |

- **顧客IDの種別の使い分け**: 一括加算では、CSVの顧客ID列にSQ内部IDを書くなら「SQが発番」、Shopify等の外部IDを書くなら「外部システムが発番」を選びます。2026-07-01実機確認では、SQ顧客IDで一括加算に成功しました。
- 一括加算の `テナント` と `顧客IDの種別` は必須マーク（*）が表示されませんが、未選択で `保存する` を押すとバリデーションエラーが出ます（実質必須）。
- 一括減算フォームに「顧客IDの種別」はありません。空保存時は `テナントを選択してください`、テナント選択済みでファイル未選択なら `ファイルを選択してください` と表示されます。2026-07-01実機確認では、SQ顧客IDは検証失敗し、同じ顧客の外部IDで一括減算に成功しました。

## 主な操作手順

### CSVをインポートする（基本の流れ）

1. サイドナビの「CSVインポート」を開く（`/admin/csv_import`）。
2. 取り込みたいカテゴリの行リンクをクリックし、カテゴリの一覧画面に遷移する。
3. 必要に応じて `テンプレート` ボタンを押し、Googleスプレッドシート形式のテンプレートを開いてCSV形式を確認する。
4. 一覧画面の `新規インポート` を押し、新規インポートフォームを開く。
5. カテゴリに応じた追加項目（ラジオ／ルール／検索型／テナントなど）を選ぶ。
6. CSVファイルをアップロードし、`保存する` を押す。
7. 詳細画面に遷移する。検証ステータスが「実行中」のあいだは `実行する` ボタンが無効。
8. 検証ステータスが「完了」になったら `実行する` を押す。
9. 確認ダイアログ「CSVの取り込み処理を実行しますか？」の内容を確認し、`実行する` を押す。
10. 実行ステータスが「完了」になり、データに反映される。完了後の `実行する` はdisabledで残り、再実行できない。

### インポート失敗時の確認方法

1. カテゴリの一覧画面で、対象の実行履歴行をクリックする。
2. 詳細画面の `検証失敗` の件数リンクをクリックする。
3. 失敗詳細画面（見出し「検証失敗」）で、エラーとなった行と理由を確認する。
4. CSVを修正し、新規インポートで再度取り込む。

### ポイントをCSVで一括加算・減算する

#### 一括加算

一括加算のフォームは `テナント`、`顧客IDの種別`、CSVファイルを指定します。`顧客IDの種別` は `SQが発番した顧客ID` / `外部システムが発番した顧客ID` の2択です。CSVヘッダーは `customer_id,title,points,available_date,expires_date,note` で、`expires_date` は必須です。

2026-07-01実機確認では、`ユニクロ`、`SQが発番した顧客ID`、顧客 `検証顧客ユニクロ095318FAQ` のSQ顧客IDで、`FAQ_POINT_PLUS_20260701` を5ポイント加算しました。検証結果は `成功 成功`、`検証成功 (1件)`、実行結果は `成功 成功`、`実行成功 (1件)` でした。確認ダイアログ本文は `1件のポイント付与を実行します。この操作は巻き戻すことができません。` です。顧客詳細の保有ポイントは `0` から `5` になり、履歴には `FAQ_POINT_PLUS_20260701 / 5 / 2026年07月01日 21:08 / 失効日 2026年07月31日` が表示されました。

別テナント所属の顧客IDを `ユニクロ` で指定した場合、検証は成功しても実行で `tenantID or purchasingCustomerID is incorrect` になりました。選択テナントと顧客の所属テナントは一致させる必要があります。

#### 一括減算

一括減算のフォームは `テナント` とCSVファイルのみで、`顧客IDの種別` はありません。CSVヘッダーは `customer_id,title,points,note` です。`points` は減算したい値を正数で入れます。

2026-07-01実機確認では、同じ顧客のSQ顧客IDを `customer_id` に入れると検証失敗し、失敗詳細に `customer_id: 指定された顧客IDの顧客が存在しません` と表示されました。同じ顧客の外部ID `FAQ_E2E_UNIQLO_CUSTOMER_20260628095318` では検証結果 `成功 成功`、`検証成功 (1件)`、実行結果 `成功 成功`、`実行成功 (1件)` になりました。確認ダイアログ本文は `1件のポイント減算を実行します。この操作は巻き戻すことができません。` です。顧客詳細の保有ポイントは `5` から `3` になり、履歴には `FAQ_POINT_MINUS_20260701 / -2 / 2026年07月01日 21:17` が表示されました。

> 在庫系の反映誤りは、インポートの再実行だけでなく在庫調整伝票での補正も選択肢になります。

### CSVで在庫を一括更新する（販売可能在庫の例）

1. 「CSVインポート」> 在庫グループ > `販売可能在庫` を開く。
2. `新規インポート` を押す。
3. 処理方法を選ぶ:
   - 「絶対値で反映する」（デフォルト）: CSVの値をそのまま在庫数にする
   - 「差分値で反映する」: 現在の在庫数にCSVの値を足す／引く
4. CSVファイルをアップロードし、`保存する` を押す。
5. 検証完了後に `実行する` を押し、確認ダイアログで `実行する` を確定する。
6. 在庫詳細（SKU詳細）の販売可能数・手持ち数に反映される。

### CSVをエクスポートする（基本の流れ）

1. サイドナビの「CSVエクスポート」を開く（`/admin/csv_export`）。
2. 出力したいカテゴリの行リンクをクリックする。
3. `エクスポートを開始する` 画面で、カテゴリに応じた条件を入力する（テナント・開始日時・終了日時／ロケーション／ルール／ディスカウントなど）。
4. `エクスポートを開始する` を押す。
5. 一覧の実行履歴から `ダウンロード` リンクを開き、CSVを取得する。

> 「エクスポート」という名前のボタンはありません。カテゴリの行リンクから遷移してください。
