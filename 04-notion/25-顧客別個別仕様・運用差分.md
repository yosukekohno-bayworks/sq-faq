# 25. 顧客別個別仕様・運用差分

> このページはWBS-25エリアの第25エリアです。SQの「標準機能」と、CHAMPION／MICHAEL KORSなど個別顧客での「連携・運用の差分」を混ぜずに説明できるようにするのが目標です。

## このエリアで学べること

- SQ標準機能と顧客別運用を分けて説明できる
- MICHAEL KORS固有のシステム構成（Shopify／Logizard／OmnibusCore／MK基幹／Retail Portal）を把握できる
- CHAMPION固有のシステム構成（AS400／Logisteed WMS／ReTELA POS／Shopify／OmnibusCore）を把握できる
- 顧客別に見るべき「正のシステム」が業務ごとに違うことを理解できる
- SQ標準の在庫依頼・移動伝票・出荷完了が、顧客別運用ではどう意味が変わるか説明できる

---

## 機能概要

このエリアは **SQ管理画面上の機能** ではなく、**実プロジェクトごとのシステム構成と運用差分** を整理するエリアです。SQ標準FAQ（エリア1〜24）は「SQ管理画面に存在する機能」を説明しますが、実運用では下記が絡みます。

- 外部連携システム（WMS／POS／基幹／モール中継）
- 顧客別のデータ発生源（どこがマスタか、どこが実績か）
- 顧客別の運用判断（承認フロー・除外タグ・バッチ時間）

> **根拠**: このエリアは `CUSTOMER-DOCS-FAQ-REVIEW-2026-06-16.md` のBacklog目視結果に基づきます。SQ実機（staging）上で直接確認できる機能は標準FAQを参照してください。顧客別の実際の連携挙動は**外部接続が必要なため未確認**です。

### 対象顧客

| 顧客 | 主な基幹 | WMS | POS | EC | モール中継 |
|:--|:--|:--|:--|:--|:--|
| MICHAEL KORS | MK基幹（CRM／SA） | Logizard（EC倉庫） | リテールポータル | Shopify | OmnibusCore |
| CHAMPION | AS400 | Logisteed | ReTELA | Shopify | OmnibusCore（3rd・預託・客注） |

## 画面・項目の説明

このエリアに該当する **専用のSQ管理画面は存在しません**。顧客別の運用差分は、主に下記の既存SQ機能と外部システムの組み合わせで表れます。

### SQ側で顧客別設定に関わる機能（既存）

| SQ機能 | URL | 顧客別運用での役割 |
|:--|:--|:--|
| ロケーション管理 | `/admin/settings/locations` | 倉庫コード・店舗コード（MK: `77910/77901/77903/77940` 等）を設定 |
| 在庫管理 | `/admin/inventory_items` | 顧客別の在庫区分・数量の最終的な「見える化」場所 |
| 在庫依頼 | `/admin/inventory_requests` | 標準は取り寄せ依頼。MKではEC注文確定時の割当依頼として意味が変わる |
| 移動伝票 | `/admin/inventory_moves` | 店舗・BtoB・倉庫間の集約移動で顧客別運用の核心 |
| 出荷管理 | `/admin/outbound_orders` | 標準の出荷実績登録。MKはLogizard実績・CHAMPIONはWMS CSVが正になる場合がある |
| 会計（売上実績） | `/admin/sale_change_line_items` | 標準は会計CSV。MKはMK基幹SAへ日次連携 |
| 顧客管理 | `/admin/customers` | 標準はチャネル流入。MKはCRM日次連携・CHAMPIONはPOS/Shopify/SQ統合 |

## 主な操作手順

### 1. SQ標準と顧客別運用を分けて説明する

1. 説明したい業務（在庫・注文・顧客・売上など）を1つ選ぶ
2. **SQ標準FAQ（エリア1〜24）** で管理画面上の操作を説明する
3. 顧客別の連携先・データ発生源・運用判断を **本エリア** で追加する
4. 同名の言葉（在庫依頼・移動伝票・出荷完了など）が顧客別で意味が変わる場合は、用語対照表を併記する

### 2. MICHAEL KORSのシステム構成を把握する

| システム | 役割 |
|:--|:--|
| Shopify | EC storefront/admin。注文・返金・商品表示・顧客接点 |
| SQ | 商品・在庫依頼・移動伝票・販売閾値・顧客統合・OmnibusCore/Logizard連携設定 |
| SQ Retail Portal | 店舗側の在庫リクエスト確認・承認・店舗出荷操作 |
| Logizard | EC倉庫/WMS。出荷依頼・入荷予定・出荷実績・追跡番号 |
| MK基幹 CRM | 顧客データ連携先 |
| MK基幹 SA | 売上実績連携先 |
| OmnibusCore | 外部モールの注文・商品・在庫・売上/出荷の中継 |

### 3. MICHAEL KORSの主なデータ流れ

| 流れ | 頻度・条件 |
|:--|:--|
| 顧客連携 | SQ Proxy Server → MK基幹CRM。毎日19:00。前日19:00:00〜当日18:59:59の作成/更新が対象 |
| 売上連携 | SQ Proxy Server → MK基幹SA。毎日19:00。出荷/返品完了が対象 |
| 倉庫在庫連携 | `77910/77901/77903` はWMS等から取り込み。安全在庫を差し引いた値がSQ/Shopify側の売れる数になる |
| 店舗在庫連携 | Full-price/outlet店舗は30分ごと。ゼロ在庫レコードが出ない問題に注意 |
| Logizard出荷依頼 | SQ → Logizard。約15分間隔の前提。追跡番号がないと出荷完了/Shopify通知が進まない |

### 4. CHAMPIONのシステム構成を把握する

| システム | 役割 |
|:--|:--|
| AS400 | 基幹。商品・在庫・顧客・注文・EC注文などの連携元/連携先 |
| SQ | OMS/商品管理/在庫管理/顧客基盤。各システムのHub |
| WMS（Logisteed） | 出荷指図CSV・出荷実績CSV |
| 店舗POS（ReTELA） | 顧客更新・ポイント照会・ポイント更新・新規顧客登録・店舗注文 |
| Shopify | ECサイト。SQ SDK等で注文/顧客/ポイント情報と連動 |
| OmnibusCore | 3rdモール・預託在庫・客注在庫補充 |

### 5. CHAMPIONの主な運用論点

| 領域 | 内容 |
|:--|:--|
| CRM | 会員ランク・ポイント算出・バースデー特典・初回購入特典 |
| POS連携 | ポイント照会・ポイント更新・新規顧客登録・店舗注文CSV |
| 顧客統合 | ShopifyではなくSQから統合。統合前に `sq:account-disabled` を付与 |
| 出荷指示 | AS400/WMS/SQの状態を見ながら作業 |
| 3rd運用 | 注文キャンセル・住所変更・物流管理・出荷指示の手順が別管理 |

## 用語対照表（標準SQ vs 顧客別運用）

FAQで一番混乱しやすいのは、同じ言葉が標準SQと顧客別運用で違う意味を持つことです。

| 用語 | 標準SQ | MICHAEL KORS | CHAMPION |
|:--|:--|:--|:--|
| 在庫依頼 | 管理画面で作る取り寄せ依頼 | EC注文確定時に倉庫/店舗へ割当依頼が発生 | 仕様書上の中心ではなく、AS400/WMS/POS/3rd連携が主 |
| 移動伝票 | 出荷指示/入荷指示を自動生成する伝票 | 店舗/BtoB/倉庫からLogizard EC倉庫へ集約する運用で重要 | 預託在庫/客注在庫補充ではAS400/WMS/OmnibusCore連携と絡む |
| 出荷完了 | SQの出荷実績登録後 | Logizard出荷実績と追跡番号が重要。Shopify通知にも影響 | WMS出荷実績CSV・AS400反映・Shopify側扱いに注意 |
| 顧客 | チャネル経由で流入 | MK基幹CRMへ日次連携。除外タグあり | POS/Shopify/SQの統合とポイント参照が重要 |
| 売上実績 | 会計/CSVとして扱う | MK基幹SAへ日次連携。返品も対象 | AS400/POS/Shopify/3rd側の扱いを分ける必要あり |

## よくある問い（顧客別FAQに必ず入れるべき問い）

### MICHAEL KORS

| 問い | 回答方針 |
|:--|:--|
| EC倉庫に在庫が足りない注文はどうなるか | BtoB倉庫・店舗倉庫・店舗在庫から取り寄せる。SQの在庫依頼/移動伝票とLogizard入荷後に出荷可能になる |
| 店舗は何を見るのか | SQ本体ではなくSQ Retail Portalを見る。自店舗分の依頼/注文だけを扱う |
| 店舗が承認しない場合はどうするか | HQが毎朝確認し、必要に応じて依頼終了・Shopify注文編集・返金を行う |
| 安全在庫はどこで効くか | 外部在庫取り込み時に差し引かれる。管理画面で直接入れた在庫には即時反映されない |
| ゼロ在庫レコード問題とは何か | 店舗在庫ファイルでゼロSKUが連携されない場合、SQ側に古い在庫が残るリスクがある |
| ZOZO/外部モールは通常ECと同じか | 同じではない。OmnibusCore・梱包番号/ケース番号・Logizard・出荷実績連携の論点がある |

### CHAMPION

| 問い | 回答方針 |
|:--|:--|
| SQだけ見れば運用できるか | できない。AS400・WMS・POS・Shopify・OmnibusCoreのどこが正かを業務別に分ける |
| ポイント/会員ランクはSQ標準だけか | 標準機能に加え、ReTELA POSやShopify自動ディスカウント連携の運用がある |
| 顧客統合はどこで行うか | SQから行う。Shopifyの統合機能は使わない |
| 3rdモールの注文変更はどうするか | 3rd運用マニュアルに従う。通常SQ標準の注文操作として扱わない |
| 古い/不採用手順はどう扱うか | `使用不可` やゴミ箱配下の資料はFAQ本文に混ぜず、履歴または注意として隔離する |

## 注意点・制約

- **実機確認範囲の限界**: このエリアの顧客別運用（WMS/POS/基幹連携の実挙動）は、staging環境では確認できません。外部連携が必要なため、すべて「未確認」として扱います（`VERIFIED-FACTS-ONLY-2026-06-19.md` 参照）
- **Shopify連携の前提**: Shopifyストア側にSQコネクターアプリ（SQ Connect）のインストールが必要。未インストール状態で「連携する」を実行するとエラーになります（constraints.md 該当項目）
- **staging環境の制約**: 公開アプリ「SQ Connect」は本番SQ専用。stagingへは別アプリとStack社からのインストールリンクが必要です
- **Backlog目視範囲**: MICHAEL KORSの2025-06-30〜2025-12-11ページはリンク集のみ確認。リンク先課題は未確認。2026-03-27の外部Notionは本文確認できていません
- **古い資料の混入リスク**: Backlogに残る `使用不可`・DRAFT・ゴミ箱配下の資料はFAQ本文に混ぜず、履歴として隔離してください
