# 12. 在庫状態・在庫数

> このページは標準24分類の第12分類です。SQの在庫データがどのような区分で構成され、注文・取置・移動の各操作でどう変化するかを理解するのが目標です。在庫管理はSQの中核機能で、発注・出荷・販売ルールの判定すべてが在庫数に依存します。

> **2026-07-01更新**: Shopifyは接続済みストアのロケーション接続まで確認済みです。ただし、SQ→Shopifyの商品/在庫同期はアクセストークン未設定で停止しており、リスティング上の連携済み商品数は0、連携済みバリエーションも0件でした。Shopifyで架空注文を作るだけではSQ既存SKUの在庫引当検証にならず、Shopify注文で在庫変動を見るには、先に商品/SKUが正式に連携済みである必要があります。本ページでは管理画面内と内部GraphQL注文の在庫挙動を中心に扱います。

## このエリアで学べること

- 在庫の9区分（販売可能・引当済み・取置中・破損・検品・予備・手持ち・積送中・入荷予定）の意味を説明できる
- 「手持ち」の計算式と、積送中・入荷予定が手持ちに含まれない理由が分かる
- 注文の引当・取置伝票の作成・移動伝票の作成で、それぞれどの区分が増減するか説明できる
- 一覧画面とSKU詳細画面の列構成の違いを把握できる
- ロケーション別に在庫数を確認・編集できる

## 機能概要

| 項目 | 内容 |
|:--|:--|
| 機能名 | 在庫管理 |
| 画面URL（一覧） | `/admin/inventory_items` |
| 画面URL（SKU詳細） | `/admin/inventory_items/[id]` |
| 画面URL（調整履歴） | `/admin/inventory_items/[id]/history` |
| 何ができるか | SKU（商品バリエーション）×ロケーション単位で在庫数を確認する。販売可能数の直接編集、在庫振替、調整履歴の確認ができる |

在庫管理画面では、商品ごと・ロケーション（店舗・倉庫）ごとに在庫数を確認します。一覧画面では主要4区分（販売可能・引当済み・取置中・手持ち）を、SKU詳細画面では9区分すべてを確認できます。

### 基礎用語

- **SKU（商品バリエーション）**: 商品の色・サイズ等のバリエーション1つ1つ。在庫数・価格はSKU単位で持つ最小管理単位。本画面では品番（SKU）列で識別する
- **ロケーション**: 在庫の置き場所。場所種別は「**倉庫**」（在庫保管・発注入荷先・出荷元・リテールポータル連携の在庫ロケーション）と「**店舗**」（リテールポータル連携の店舗ロケーション・店舗受取の受取ロケーション）の2種のみ。手動の在庫依頼・移動伝票では倉庫/店舗の両方が移動元・移動先として関与するため、画面ごとの選択肢は04.基本マスタを参照
- **論理ロケーション**: EC販売用など、物理的に存在しない仮想のロケーション。販売可能数はSKU詳細画面の鉛筆アイコンで設定する（詳細は後述「販売可能数を編集する」）

> 倉庫と店舗では登場する画面・操作が異なります。リテールポータル連携では店舗種別が「店舗ロケーション」、倉庫種別が「在庫ロケーション」として分かれて表示されます。店舗受取のロケーション選択肢は店舗種別のみです。タブ（すべて / 店舗 / 倉庫）で種別を絞り込めます。

### 在庫詳細画面（/admin/inventory_items/[id]）

特定のSKUの在庫数をロケーション別に確認・操作する画面です。

#### ヘッダーボタン

| ボタン（UIラベル） | 動作 |
|:--|:--|
| 調整履歴 | 変更履歴ページ（`/admin/inventory_items/[id]/history`）へ遷移する |
| 商品管理 | バリエーション詳細画面へ遷移する |
| その他のアクション ▼ | 「在庫振替を行う」メニューが展開する |

タブは「**すべて / 店舗 / 倉庫**」の3種で、ロケーション種別を絞り込めます。

#### 在庫数量の9区分

> **2026-06-14更新**: 区分名が「確定済み→引当済み」「確保済み→取置中」に変わり、「積送中」「入荷予定」が新設されました。

| 列名 | 意味 | 手持ちに含まれるか |
|:--|:--|:--|
| ロケーション | ロケーション名 | — |
| 販売可能 | 販売できる在庫数（鉛筆アイコン付き） | 含まれる |
| 引当済み | 注文に引き当て済みの在庫数（旧「確定済み」）。2026-06-28追加検証では、DECREMENT注文作成時に `販売可能 1→0`、`引当済み 0→1` へ移った | 含まれる |
| 取置中 | 取置・移動用に確保された在庫数（旧「確保済み」） | 含まれる |
| 破損 | 破損品の在庫数。管理画面からは直接編集不可（2026-06-19実機確認） | 含まれる |
| 検品 | 入荷時の検品などで一時的に入る区分（想定）。管理画面からは直接編集不可 | 含まれる |
| 予備 | 用途は資料に明記なし（販売不可枠の内訳の一つと推定）。管理画面からは直接編集不可 | 含まれる |
| 手持ち | 実在庫合計（上記6区分の和） | — |
| 積送中 | 出荷済み・入荷前の輸送中の数（移動先で計上）。移動伝票を作成すると移動先ロケーションに立ち、物理的に手元に無いため手持ち外。到着（入荷実績登録）で販売可能・手持ちに振り替わる | 含まれない |
| 入荷予定 | 移動伝票作成〜出荷前の入荷予定数（移動先で計上）。移動伝票作成で移動先に立ち、まだ手元に無いため手持ち外。出荷実績登録で積送中へ、入荷実績登録で販売可能・手持ちへ | 含まれない |

> **破損・検品・予備の扱い（2026-06-28再確認）**: 管理画面の「その他のアクション」には「在庫振替を行う」の1メニューのみで、これら3区分に在庫を入れる専用の操作UIはありません。調整伝票の「理由」選択肢も `廃棄` / `紛失` / `在庫差異` / `その他` の4種のみです。WMS（ロジザードZERO）連携経由で設定される可能性は開発元確認が必要です。表の数字は読み取り専用とお考えください。

**手持ちの計算式**（積送中・入荷予定は含まない）

```
手持ち = 販売可能 + 引当済み + 取置中 + 破損 + 検品 + 予備
```

### 「販売可能在庫数を編集する」ダイアログ

在庫詳細の「販売可能」列にある鉛筆アイコン（aria-label: 「販売可能数を編集」）をクリックすると開きます。EC販売用など物理的に存在しない仮想のロケーション（**論理ロケーション**）の販売可能数も、同じ鉛筆アイコンで設定します。

| 項目（UIラベル） | 必須/任意 | 内容 |
|:--|:--|:--|
| 理由 | **必須** | テキスト自由入力。在庫変更の監査用メモで、調整履歴画面の「アクティビティ」列に表示される。後で「誰がなぜ変えたか」が分かるように書く（例: 「棚卸修正」「EC枠設定」）。空のまま「保存する」を押すと「理由を入力してください」と表示されて保存できない |
| 数量 | — | 数値入力（初期値 0、単位「個」）。**絶対値指定（加算ではない）**。例: 現在5の状態で10にしたい→10を入力（5に加算して10、ではない）。負数も保存できるが実在庫と合わなくなるので補正目的に限定 |
| キャンセル / 保存する | — | 「保存する」で確定 |

**保存したときの挙動（実機確認済み）:**

- 入力した数値が**そのまま新しい「販売可能」数になる**（絶対値の上書き。加算ではない）。例: 販売可能1の状態で数量9を入力 → 販売可能は9になる
- 手持ちも差分だけ増減する（1→9なら手持ちも+8）
- 負数も保存できる（例: 数量 -1 で販売可能・手持ちが -1 になる）。実機確認では `483457-10-L` で 95→1→-1→95 を実施
- 調整伝票は作成されない（調整伝票とは別経路の在庫変更）
- 変更は「調整履歴」に記録される

## 主な操作手順

### 手順1: ロケーションを切り替えて在庫を確認する

1. 在庫管理一覧（`/admin/inventory_items`）を開く
2. ヘッダーのロケーション名ボタン（例: 「**物流倉庫 ▼**」）をクリックする
3. 「**ロケーションを選択する**」モーダルが開く
4. タブ（すべて / 店舗 / 倉庫）で種別を絞り込む
5. 目的のロケーション行を選択し「**選択する**」を押す
6. 一覧が選択したロケーションの在庫に切り替わる

### 手順2: SKUの在庫内訳（9区分）を確認する

1. 在庫管理一覧で対象SKUの行をクリックする
2. 在庫詳細画面（`/admin/inventory_items/[id]`）へ遷移する
3. タブ（すべて / 店舗 / 倉庫）でロケーション種別を絞り込む
4. 各ロケーション行の販売可能・引当済み・取置中・破損・検品・予備・手持ち・積送中・入荷予定を確認する

### 手順3: 販売可能数を直接編集する

1. 在庫詳細画面で対象ロケーション行の「販売可能」列にある鉛筆アイコンをクリックする
2. 「**販売可能在庫数を編集する**」ダイアログが開く
3. 「理由」に変更理由を入力する（必須）
4. 「数量」に新しい販売可能数を入力する（絶対値。加算ではない）
5. 「**保存する**」を押す
6. 販売可能と手持ちが更新される。変更は「調整履歴」に記録される

### 手順4: 在庫変更の履歴を確認する

1. 在庫詳細画面で「**調整履歴**」ボタンを押す
2. 調整履歴画面（`/admin/inventory_items/[id]/history`）へ遷移する
3. 必要に応じてロケーションを選択して絞り込む
4. 日時・アクティビティ・各区分の差分を確認する

## 操作ごとの在庫の変化（実機検証済み 2026-06-14〜2026-07-01）

各操作が在庫区分に与える影響をまとめた表です。在庫は操作のタイミングで以下のように変化します。

| 操作 | 移動元の変化 | 移動先の変化 |
|:--|:--|:--|
| 在庫依頼を作成する（確認待ち状態） | 変化なし | — |
| 在庫を引当てる（在庫依頼が確保済み状態になる） | 変化なし（数字はまだ動かない） | — |
| 移動伝票を作成する | 販売可能 −n、取置中 +n（手持ちは不変） | 入荷予定 +n |
| 出荷実績を登録する | 取置中 −n、手持ち −n | 入荷予定 −n → 積送中 +n |
| 入荷実績を登録する | — | 積送中または入荷予定 −n → 販売可能 +n・手持ち +n。出荷前でも入荷登録は可能 |
| 移動伝票をキャンセルする | 取置中 −n → 販売可能 +n（元に戻る） | — |
| 取置伝票を作成する | 販売可能 −n、取置中 +n | — |
| 取置を処理済みにマークする | 取置中 −n → 販売可能 +n（解放） | — |
| 調整伝票を実行する | 増減数がそのまま反映 | — |
| 鉛筆で販売可能数を編集する | 「販売可能」を入力値そのものに設定（絶対値）。手持ちは差分だけ増減 | — |
| 注文を作成する（SQ内部 `inventoryBehavior=DECREMENT`、出荷必要） | 販売可能 −n、引当済み +n（手持ちは不変） | — |
| 注文由来の出荷指示を出荷完了する | 引当済み −n、手持ち −n（数秒後に反映。販売可能は注文作成時点で減っているため追加では減らない） | — |
| 在庫振替を行う | 選択ロケーションの振替元SKUの販売可能・手持ちが全量減る | 同一商品の振替先SKUの販売可能・手持ちが同量増える |

> **2026-06-22〜28追加確認:** 新規GU SKU `TEST_E2E_20260622_GU_1905_NAVY_M` でも、在庫詳細テーブルにロケーション別の `販売可能` と `手持ち` が表示されました。後続確認では `TEST_E2E_20260622_GU倉庫_ON_1905` が販売可能5/手持ち5、`TEST_E2E_20260622_GU店舗_OFF_1905` が販売可能1/手持ち1でした。調整伝票 `#IA-1011` の `+2` 実行後、`ON_1905` は販売可能7/手持ち7になりました。さらに2026-06-27の `#IA-1013` で同じロケーション・SKUに `+2` を実行し、在庫一覧上で販売可能7→9、手持ち7→9、引当済み0・取置中0のまま変化しないことを確認しました。同日 `#IR-1009` の取置数量1では、保存直後に販売可能9→8・取置中0→1・手持ち9のまま、処理済み後に販売可能8→9・取置中1→0・手持ち9のまま戻ることを確認しました。2026-06-28のDECREMENT注文 `FAQ-REMAINING-DECREMENT-20260628135953` では、同SKU/同倉庫が `販売可能 1→0`、`引当済み 0→1`、`手持ち 1→1`、`売可 1→0` になり、注文引当は手持ちを変えず販売可能から引当済みへ振り替えることを確認しました。

### 移動伝票1件のライフサイクル（移動元と移動先を並べて時系列で追う）

数量 n の移動伝票1件を作成→出荷→入荷するまで、移動元と移動先の9区分が同時にどう動くかを示します。

| ステップ | 移動元の状態 | 移動先の状態 |
|:--|:--|:--|
| 1. 移動伝票を作成 | 販売可能 −n / 取置中 +n（手持ち不変） | 入荷予定 +n |
| 2. 出荷実績を登録 | 取置中 −n / 手持ち −n | 入荷予定 −n → 積送中 +n |
| 3. 入荷実績を登録 | （確定済み。変化なし） | 積送中 −n → 販売可能 +n / 手持ち +n |

> 出荷前に移動先で入荷実績を登録することも可能です（ステップ2を飛ばして3へ）。その場合は入荷予定から直接販売可能・手持ちへ遷移します。キャンセルはステップ1の後なら「取置中 −n → 販売可能 +n」で元に戻ります。

## 注意点・制約

- **手持ちはマイナスになり得る**: 在庫が0のロケーションから移動伝票作成・出荷実績登録をしても、SQはブロックしません。実在庫を超えた出荷登録をすると在庫数がマイナスになります（2026-06-18実機確認 `#IM-1016` / `#IO-1016`）。修正は調整伝票や移動伝票で補充してください。<!-- 実機確認済み -->
- **破損・検品・予備区分への管理画面振替UIなし**: 管理画面の「その他のアクション」は「在庫振替を行う」の1メニューのみで、破損・検品・予備専用の操作メニューはありません。調整伝票の「理由」選択肢（廃棄・紛失・在庫差異・その他）にもこれらの区分は存在しません。WMS（ロジザードZERO）連携経由の可能性は開発元確認が必要です <!-- 連携実接続対象外: WMS連携〈ロジザード等〉経由での設定なのか、開発元への確認が必要 -->
- **「在庫振替を行う」は同一商品内の別バリエーション限定**: 別商品への振替はできません。振替先バリエーションのドロップダウンには同一商品の他バリエーションのみ表示されます。
- **在庫振替は数量指定なしの全量移動**: 2026-07-01実機確認では、在庫振替ダイアログに数量欄はなく、`ロケーション` / `振替先のバリエーション` / `理由` の3項目でした。複数バリエーション商品 `486125-09-XL` から `486125-09-L` へ振り替えると、選択ロケーションの元SKU販売可能/手持ち6が全量0になり、振替先が6になりました。部分数量の振替として案内しないでください。
- **一括操作は提供されていない**: 一覧でチェックボックスを使って全件選択できますが、一括操作ボタンは表示されません。
- **鉛筆編集は調整伝票を作らない**: 販売可能数の直接編集は調整履歴に残りますが、調整伝票（`/admin/inventory_adjustment_orders`）には新しい伝票が作成されません。別経路の在庫変更です。
- **在庫依頼の「確保済み」はステータス名**: 在庫区分の「取置中」（旧「確保済み」）と、在庫依頼ステータスの「確保済み」は別物です。在庫依頼を引当てて「確保済み」になっても、在庫の数字はまだ動きません（動くのは移動伝票作成時）。
- **移動・入荷の数量入力は初期値0**: 移動伝票の明細数量、入荷実績登録画面の今回受領数は、どちらも初期値が `0` でした。商品を選んだだけでは数量1にならないため、保存・登録前に数量を入力してください（2026-06-30 `#IM-1033` / `#II-1035`）。
- **出荷完了後も副次ステータスが残る場合がある**: `#IO-1041` では出荷実績登録後に上部ステータスが `成功 完了 / 出荷完了`、出荷済みが1になりましたが、同じ詳細画面内に `情報 未完了 / 引当待ち` も残って表示されました。出荷完了判定は上部ステータスと出荷済み数を優先し、副次表示だけで未出荷とは判断しないでください。
