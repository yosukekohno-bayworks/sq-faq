# 19. 店舗業務・リテールポータル

> このページは標準24分類の第19分類です。店舗スタッフが使うリテールポータルと、管理画面側で行うその接続設定・店舗受取設定を分けて理解するのが目標です。2026-07-16に実機を再確認し、在庫の「確保済み」追加、連携詳細のタブ化、客注注文の発送済み・売上実績反映、店舗受取の登録済みデータまで確認しました。

> **2026-07-24実機再検証**: ドキュメント全体を実機と突き合わせました。主な差分は次の4点です。(1) 連携詳細画面が「基本設定 / ユーザー / **販売員**」の**3タブ**になった（販売員タブ新設）。(2) **5つの許可設定すべてのOFF時挙動をフリップ検証で実証**（下記の許可設定の表を参照）。(3) ポータル左メニュー「在庫依頼」配下に「**確保済み**」サブメニューが新設され、「移動伝票を一括作成する」ボタンを持つ（取り寄せ販売ステップ5の画面はポータル側にある）。(4) 客注のキャンセル操作のダイアログ・キャンセル後挙動を確認。また、在庫詳細からの「下書き注文を作成」で**明細が自動セットされない**事象を観測しました（仕様変更か偶発かは未切り分け）。詳細は `faq/_analysis/retail19-live-recheck-20260724.md` を参照。

## このエリアで学べること

- リテールポータルが店舗スタッフ向けのどんな画面か、管理画面側で何を設定するのかを説明できる
- 販売員コードによる利用開始（販売員ゲート）と「画面をロックする」の仕組みを説明できる
- 店舗スタッフが行う客注（下書き注文）の作成から注文化までの流れと、在庫・出荷指示への影響を説明できる
- リテールポータル連携の設定項目（店舗ロケーション・在庫ロケーション・カタログ・販売閾値ルール・5つの許可設定）が、ポータル側のどの挙動に対応するかを説明できる
- 店舗受取（ローカルピックアップ）の設定画面と設定内容を説明できる

## 機能概要

このエリアは「チャネル」グループの店舗系機能で、大きく2つの世界から成り立ちます。

**2つの世界の役割分担:**

| 側 | URL | 誰が使う | やること |
|:--|:--|:--|:--|
| **管理画面（設定側）** | `www.sqstackstaging.com/admin` | 管理者・運営者 | リテールポータル連携の作成・店舗受取ルールの設定。ロケーション・テナント・カタログ・許可操作を指定し、利用ユーザーを追加する |
| **リテールポータル（店舗スタッフ側）** | `retail.sqstackstaging.com/admin` | 店舗スタッフ（販売員） | EC在庫の確認、客注（下書き注文）の作成・完了、店舗宛入荷の処理、店舗注文・売上の確認 |

`管理画面(連携作成・ユーザー追加) → リテールポータル(店舗スタッフ操作)` という流れで、まず管理画面で「どの店舗で・どの在庫を引き当て・何を許すか」を決め、その範囲内で店舗スタッフが現場を操作します。

| 機能 | 画面URL | 役割 |
|:--|:--|:--|
| リテールポータル連携 | `/admin/retail_portal_integrations` | 店舗スタッフが使うリテールポータルをSQと接続する（1件=1店舗ロケーション単位） |
| 店舗受取 | `/admin/local_pickup_retail_location_rules` | リテールポータル配下のサブメニュー。店舗受取可能なロケーションを設定する |
| リテールポータル本体 | `retail.sqstackstaging.com/admin` | 店舗スタッフ側の画面（stagingの実URL） |

## リテールポータル（店舗スタッフ側）の画面

### アクセスと販売員ゲート

- 認証は本体管理画面と同じアカウント（Clerk）。**リテールポータル連携の詳細画面で「ユーザーを追加する」から追加されたユーザー**がアクセスできます（2026-07-03に追加済み管理者ユーザーで実証）。
- ヘッダーには ユーザー名＋**連携の店舗ロケーション名**（例: ユニクロ - 銀座店）が表示されます。
- ログイン直後は「リテールポータルのご利用には販売員の選択が必要です。」と表示され、**販売員コードを入力して販売員を選択するまで全画面が使えません**（連携設定「リテールポータルで販売員の選択を必須にする」ON時）。
  - 「販売員を検索する」ダイアログで販売員コードを入力→「検索する」→候補を確認して「選択する」
  - 検索は**完全一致**です（部分一致はヒットしません）
  - コードは 設定 > 販売員（`/admin/settings/retail_staff_members`）で管理します（詳細は **04.基本マスタ**）
- **ページを再読み込みすると販売員選択が再要求されます**（画面内のメニュー遷移では保持）。
- ヘッダーの鍵アイコン >「**画面をロックする**」で、いつでも販売員選択ゲートに戻せます。店頭での販売員交代を想定した動きです。選択した販売員は客注に記録されます（後述）。

商品管理・発注管理・販売設定・会計・分析・CRM・販売チャネル・設定はメニューにありません。

### 在庫管理（EC在庫の確認）

ホームのカード説明は「EC在庫の有無を確認することができます。」です。

- 一覧の列: 商品 / 品番(SKU) / 状態バッジ（`積送中` 等） / 販売可能 / 引当済み / 確保済み / 取置中 / 手持ち
- ヘッダーのロケーション切替ボタンから「ロケーションを選択する」ダイアログ（すべて/店舗/倉庫タブ）で**他店舗・倉庫の在庫も参照できます**（取り寄せ・客注時の在庫確認用途）
- 行クリックで在庫詳細へ。**全ロケーション×10区分**（販売可能/引当済み/**確保済み**/取置中/破損/検品/予備/手持ち/積送中/入荷予定）を一望できます。積送中は `+5` のような符号付き表示
- 数量の編集はできません（鉛筆編集なし・読み取り専用）
- 在庫詳細の右上に「**下書き注文を作成**」ボタンがあり、これが客注の起点になります

![リテールポータルの在庫詳細。全ロケーション×10区分と「下書き注文を作成」](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/retail-portal-04-inventory-detail.png)

### 客注（下書き注文）の作成 → 注文化【2026-07-03実測】

店舗スタッフが顧客の注文を店頭で受け、指定の在庫ロケーション（倉庫）から発送する「客注」のフローです。`#202607031000` で実測しました。

**フロー全体:**

```
在庫詳細で「下書き注文を作成」（または客注管理の「注文を作成する」）
→ 明細・顧客・配送先住所を設定して「保存する」（下書き注文 = 客注）
→ 「下書き注文を完了する」（巻き戻し不可）
→ 注文が作成される（決済状況: 支払い済み / 出荷状況: 未出荷）
→ 在庫ロケーションで引当済み+1、倉庫宛の出荷指示（出荷待ち・配送）が自動作成
→ 以降は通常の出荷業務（倉庫が出荷 → 顧客へ配送）
```

**作成フォームの項目:**

| セクション | 内容 |
|:--|:--|
| 注文明細 | SKU・単価×数量。鉛筆アイコン→「商品を編集する」で**単価**（許可設定ON時）と**数量**を編集。数量には上限あり（後述） |
| 注文明細の追加 | ⚠️ 2026-07-24の再検証では、在庫詳細の「下書き注文を作成」から遷移しても**注文明細は空**で、「追加する」からバリエーション選択ダイアログで手動追加が必要でした（07-03記録では自動セット）。仕様変更か経路による偶発かは未切り分け |
| 送料明細 | 「追加する」で送料行を追加（許可設定ON時） |
| メモ | 自由記入 |
| 顧客 | 「顧客を選択する」（**メールアドレス完全一致検索**）または「バーコードから顧客を選択する」 |
| 配送先住所 | 顧客選択後に「選択する/作成する」。**必須**（未設定で保存すると「配送先住所を選択してください」エラー）。住所作成の郵便番号は `104-0061` 形式（**ハイフン必須**）。請求先住所は自動で「配送先住所と同じ」 |
| 販売員 | ゲートで選択中の販売員が自動記録される（編集不可） |
| タグ | 自由入力。保存時に **`リテールポータル (客注)`** タグが自動付与される |

**数量上限 = 在庫ロケーションの販売可能数 − 販売閾値**。実測では倉庫の販売可能97、連携の販売閾値ルールのデフォルト閾値5に対し、上限は92でした。**販売閾値ルールはリテールポータルの客注数量を制御します**（実効を実機で確認済み。詳細は **11.価格・販売制御**）。なお2026-07-24の再検証では、商品編集ダイアログの数量欄に07-03記録の `/92` のような分母（上限）表示が見当たりませんでした（対象SKUが異なるため単純比較不可。表示条件は要確認）。

**下書き注文詳細の操作:**

| ボタン | 動作 |
|:--|:--|
| 下書き注文を完了する | 確認ダイアログ「この下書き注文を完了して注文を作成します。**この処理は巻き戻すことができません。**」→ 注文を作成 |
| その他の操作 > バーコードを表示する | 「バーコード(NW7)」ダイアログ。SKUにバーコード未登録の場合は商品名のみで画像は空欄（POSレジ読み取り用途と推定） |
| その他の操作 > 下書き注文をキャンセルする | 客注の取り消し。確認ダイアログはタイトル「注文をキャンセルする」、本文「注文をキャンセルします。この処理は巻き戻すことができません。」、ボタン「注文を保持する」/「キャンセルする」（2026-07-24実測）。実行後は詳細に黄色の「キャンセル済み」バッジが付き、下書き一覧には**削除されず取り消し線付きで残る** |

**完了（注文化）で起きること（実測）:**

- 注文ヘッダーに「客注(店舗ポータル) #202607031000 から注文」とソースが明記される。番号は下書きと同一
- 決済状況は**即「支払い済み」**になる（決済履歴は空。店頭決済を前提にSQ上はトランザクションを持たない挙動）
- **連携設定の在庫ロケーション（倉庫）で 販売可能−1 / 引当済み+1**。店舗側在庫は変化しない
- **倉庫宛の出荷指示が自動作成**される（作成元=注文、出荷方法=配送、出荷待ち・引当待ち）
- タグ `#ON-xx` が自動追加される
- ポイント付与対象・会員ランク算出対象に含まれる（トグルは「はい」）
- 注文化直後は売上実績が計上されず、出荷後に計上される

**2026-07-16後続確認:** 注文 `#202607031000` は現在「発送済み」です。リテールポータルの売上管理には、計上日 `2026-07-13 23:19`、作成元 `客注(店舗ポータル)`、店舗 `ユニクロ - 銀座店`、SKU `487973-64-36`、売上 `4,500円`、税 `0円` の実績が表示されました。客注の出荷完了後に売上実績へ反映されることを確認済みです。

![リテールポータルの売上管理に反映された客注売上](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/retail-portal-11-sales-posted.png)

**本体管理画面側での見え方:**

- 注文管理 > 下書き に行が表示される（列に**販売員**があり、誰が受けた客注か追跡できる）
- 注文詳細には「**外部システムから連携されている注文です** — 編集（キャンセル・返品・返金など）は制限されています。連携元システムの管理画面より操作を行ってください。」という警告が出る。**客注の操作はリテールポータル側で行う**運用になります
- 本体側の 注文管理 > 下書き の「注文を作成する」ボタンは従来どおり無効（客注の作成入口はリテールポータル側のみ）

### 取り寄せ販売（EC注文起点）でのリテールポータルの位置づけ

リテールポータルは、EC注文起点の取り寄せ販売では「店舗側の在庫確保・出荷作業」の画面として位置づけられています（全体像は **02-by-task/取り寄せ販売（EC注文起点）の流れ** を参照）。

| # | 担当 | 画面 | 操作 |
|:--|:--|:--|:--|
| 1-3 | 顧客/自動 | EC・SQ | 注文作成→SQ連携→倉庫在庫不足で在庫リクエスト発生 |
| 4 | 店舗スタッフ | **リテールポータル（在庫依頼）** | 在庫リクエストを確認し在庫を「確保」 |
| 5 | 店舗スタッフ | **リテールポータル（在庫依頼 > 確保済み）** | 「確保済み」一覧（h1表記は「承認済み」）の「移動伝票を一括作成する」で移動伝票を一括作成（2026-07-24に画面を確認。0件のためボタンはdisabled。管理画面側に同等機能があるかは未確認） |
| 6 | 店舗スタッフ | **リテールポータル（出荷管理）** | 店舗から出荷実績を登録 |
| 7 | 倉庫スタッフ | SQ（入荷管理） | 倉庫で入荷実績を登録 |
| 8 | 自動 | SQ | EC注文の出荷指示が出荷可能に |

> 2026-07-24時点でも店舗宛の在庫リクエストが存在せず、ポータルの在庫依頼一覧・「確保済み」一覧はどちらも空でした（在庫依頼に作成ボタンなし）。ステップ4の「確保」操作の実画面は在庫リクエスト発生時に再検証します。逆方向（店舗が受けた客注を倉庫から発送する）は上記の客注フローとして実測済みです。

## 画面・項目の説明（管理画面側の設定）

### 前提となるマスタ（作成前に用意するもの）

| マスタ | 何のことか | 作成場所 |
|:--|:--|:--|
| **テナント** | 事業部（運用ルール分離単位。詳細は **03.組織・通知**） | 設定 > テナント |
| **カタログ** | 商品を束ねる汎用的な商品グループ。リテールポータルの在庫管理（EC在庫）に表示される商品の範囲になる | カタログ機能（詳細は **06.カタログ**） |
| **販売閾値ルール** | 在庫がいくつ残っていれば販売OKかのしきい値。リテールポータルでは客注の数量上限（販売可能−閾値）として実効する | 設定 > 販売設定 > 販売閾値（詳細は **11.価格・販売制御**、作業手順は **02-by-task/販売閾値を設定する**） |
| **販売員** | 店舗スタッフのマスタ。ポータル利用開始時のコード入力と客注への操作者記録に使う | 設定 > 販売員（詳細は **04.基本マスタ**） |

## 主な操作手順

### リテールポータル連携を作成する（管理者）

1. 管理画面で `/admin/retail_portal_integrations` を開く
2. 「追加する」ボタンをクリックし、作成フォームに進む
3. 「店舗ロケーション」の「選択」ボタンを押し、ダイアログで店舗種別のロケーションを選ぶ
4. 「在庫ロケーション」の「選択」ボタンを押し、ダイアログで倉庫種別のロケーションを選ぶ
5. 「テナント」「カタログ」をコンボボックスから選ぶ（必須）
6. 必要に応じて「販売閾値ルール」を選ぶ（任意。客注の数量上限に効く）
7. 店舗スタッフに許可する操作のチェックボックスをオンにする（5種類、デフォルトすべてオフ）
8. 「保存する」をクリックする
9. 詳細画面の「ユーザー」タブで「追加する」を押し、ポータルを使うユーザーを追加する

### リテールポータルを使い始める（店舗スタッフ）

1. `retail.sqstackstaging.com/admin` を開き、SQアカウントでログインする
2. 「販売員を選択する」をクリックする
3. 自分の販売員コードを入力し「検索する」をクリックする（完全一致）
4. 候補に自分の名前が出たら「選択する」をクリックする
5. 交代するときはヘッダーの鍵アイコン >「画面をロックする」で販売員選択に戻す

### 客注（下書き注文）を作成して注文にする（店舗スタッフ）

1. 「在庫管理」で商品を探し、行をクリックして在庫詳細を開く（他店舗・倉庫の在庫もここで確認できる）
2. 「下書き注文を作成」をクリックする
3. 注文明細の鉛筆アイコンから数量（必要なら単価）を設定する（明細が空で開いた場合は、先に「追加する」からバリエーションを選んで追加する）
4. 「顧客を選択する」で顧客のメールアドレスを入力して検索し、選択する
5. 「配送先住所を選択する」または「配送先住所を作成する」で配送先を設定する（必須。郵便番号はハイフン付き）
6. 必要に応じて送料明細・メモ・タグを設定し、「保存する」をクリックする
7. 内容を確認し「下書き注文を完了する」→ 確認ダイアログで「完了する」をクリックする（**巻き戻し不可**）
8. 注文が作成され、在庫ロケーション（倉庫）宛の出荷指示が自動発行される。以降の出荷は倉庫側の出荷業務で処理する

### 店舗受取ルールを作成する（管理者）

1. 設定 > ロケーションで、対象ロケーションが `場所種別=店舗` になっていることを確認する。
2. `店舗受け取りを有効にする` のON/OFFを確認する。2026-06-28再確認では、OFFの店舗も店舗受取ルール作成フォームの候補に表示されるため、管理画面の候補表示だけではこのチェックの実効差を判断しない。
3. 管理画面で `/admin/local_pickup_retail_location_rules` を開く。
4. 「追加する」ボタンをクリックし、作成フォームに進む。
5. 「ロケーション」の「選択」ボタンを押し、ダイアログで店舗種別のロケーションを選ぶ。
6. 「保存する」をクリックする。

### 店舗受取対象SKUを追加する（管理者）

1. 管理画面で `/admin/local_pickup_product_variants` を開く。
2. `バリエーションを追加する` をクリックする。
3. 「バリエーションを選択する」ダイアログで対象SKUを選ぶ。
4. `選択する` をクリックする。

> この商品側のSKU指定は [07.店舗受取商品](07-店舗受取商品.md) で扱います。

## 注意点・制約

- **客注の作成入口はリテールポータル側のみ**: 本体管理画面の 注文管理 > 下書き の「注文を作成する」は無効のままです（2026-07-03再確認）。客注はポータルで作成します。
- **客注から作られた注文は本体から編集できない**: 本体の注文詳細に「外部システムから連携されている注文です」と表示され、キャンセル・返品・返金などが制限されます。客注の操作（キャンセル等）はリテールポータル側で行います。
- **下書き注文の完了は巻き戻し不可**: 完了確認ダイアログにも明記されています。完了前に明細・顧客・配送先を確認してください。
- **引当は店舗在庫ではなく「在庫ロケーション」**: 客注を完了すると、連携設定の在庫ロケーション（倉庫）で引当済みが増え、倉庫宛の出荷指示が自動発行されます。店舗の在庫は動きません。
- **数量上限は販売可能−販売閾値**: 連携に販売閾値ルールを割り当てると、その閾値分は客注で注文できません。
- **販売員選択はリロードで再要求**: ポータルはページを再読み込みするたびに販売員コードの再入力が必要です。販売員コード検索は完全一致です。
- **顧客検索はメールアドレス完全一致**: 下書き注文の顧客選択は部分一致では見つかりません。
- **郵便番号はハイフン必須**: 配送先住所の作成では `104-0061` 形式で入力します。
- **事前準備が必須**: テナント・カタログ・販売員はSQ内で事前に作成しておく必要があります。販売閾値ルールを使う場合も事前作成が必要です。
- **ロケーション種別で選択肢が変わる**: 店舗ロケーションには「店舗」種別、在庫ロケーションには「倉庫」種別のみ表示されます。
- **チェックボックスはデフォルト全オフ**: 5つの許可設定はすべてデフォルトでオフです。店舗スタッフに許可したい操作のみオンにしてください。
- **Shopify連携が別途必要な場合がある**: リテールポータル経由の注文をECチャネルに連携するには、別途Shopify等のチャネル接続が必要です。
