# 23. API・Webhook・開発者連携

> このページは標準24分類の第23分類です。外部のシステム（自社アプリ・BIツール・連携サービスなど）がSQのデータを読み書きするための「入口」を理解するのが目標です。設定画面の「アプリ」を起点に、Admin API・Storefront API・Webhook・リクエストログを取り扱います。

## このエリアで学べること

- SQが外部システムに提供するAPI（Admin API / Storefront API）の違いと用途を説明できる
- 設定>アプリで「アプリ」を作ると何が発行され、何に使えるのか把握できる
- アクセストークン・シークレット・Storefrontトークン・Webhookという各要素の役割が分かる
- Webhookで受け取れるイベントの種類と、設定すべき項目が分かる
- 「Playgroundを開く」「リクエストログを見る」といった検証・監査手段があることを知っている

## 機能概要

### API/Webhook・開発者連携とは

SQの管理画面で入力したデータ（商品・注文・在庫・顧客など）は、外部のプログラムからも読み書きできます。このための入口として、SQは以下を提供しています。

- **Admin API**: 管理機能（商品・注文・在庫などの読み書き）を操作するためのAPI（GraphQL）。管理画面でできることと同等の操作を社内システム・BIツール・連携サービスからプログラム経由で行う向けです。
- **Storefront API**: 顧客向けECフロント（商品一覧の取得・カート操作など）を独自に構築する向けのAPI。
- **Webhook**: SQ側でイベント（注文の作成・在庫の更新など）が起きたときに、外部のURLへ通知を送る仕組み。

> **GraphQLとは**: URLへJSON形式で問い合わせを送り、必要な項目だけを取得・書き込みできるAPIの方式です（RESTのように複数のエンドポイントを分けず、1つのエンドポイントにクエリを投げます）。
> **Admin API と Storefront API の使い分け**: Admin API は管理者側（商品・注文・在庫の読み書き、業務システム連携）、Storefront API は顧客側（商品表示・カートなど公開ストアの構築）と用途が分かれています。実行可能な範囲・権限モデルは異なるため、用途に応じて使い分けます。

これらを使うには、まず設定画面で **アプリ** という単位を作成します。アプリはAPIを呼ぶための「権限（スコープ）」「アクセストークン」「シークレット」をまとめて管理する入れ物です。

### 対象画面

| 画面 | URL | 役割 |
|:--|:--|:--|
| アプリ 一覧 | `/admin/settings/apps` | APIアクセスを管理するアプリの一覧表示・管理 |
| アプリ 作成 | `/admin/settings/apps/create` | 新しいアプリ（APIキー群）の作成 |
| アプリ 詳細 | `/admin/settings/apps/{id}` | アクセストークン・Webhook・リクエストログ等の確認・設定 |
| リクエストログ | `/admin/settings/apps/{id}/admin_api` | 対象アプリのAdmin APIリクエストログ表示 |

> 設定画面（`/admin/settings`）の「外部連携グループ」配下に「アプリ」があります。画面の説明文（UIの原文）は「APIアクセスを管理する / アプリを作成して、SQのアドミンAPIにアクセスしましょう。」です。

### 何ができるか

- アプリを作成すると、**Admin API用のアクセストークンとシークレットが自動発行**される
- アプリごとに付与する権限（スコープ）を **76種の権限**から個別に選択できる
- 同じアプリ画面から **Storefront APIのトークン発行**と **Webhookの作成**ができる
- 「Playgroundを開く」で GraphQL Playground 上でAPIを試せる
- 「リクエストログを見る」で、そのアプリのAdmin APIログ画面へ遷移できる（2026-06-28再確認でも本文は `TODO` 表示）

### アプリ作成フォーム

![アプリ作成フォーム](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/14-apps-create-full.png)

フォームのパス: `/admin/settings/apps/create`

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| アプリ名 * | 必須 | テキストボックス（プレースホルダー: 「モバイルアプリ」） |
| 権限（チェックボックス群） | — | 76権限から付与するスコープを個別に選択。各権限には説明文が付いている（例: 「商品情報を閲覧することができます。」）。スコープの絞り方・漏洩時リスクは注意点参照 |

> **権限の用語**: **リソース** = 商品・注文・在庫などAPIで操作できる対象（全38種）。**スコープ** = 各リソースごとに閲覧（`:read`）/ 編集（`:write`）を個別に許可する権限。代表的なリソースは 商品 / 注文 / 在庫 / 顧客 / ロケーション / テナント / ブランド / 発注伝票 など。権限グループ（アカウント権限）と同じ38リソース体系で、API側は read/write の2軸で合計76権限になります。
> **スコープ挙動は実機未確認**: チェックを入れたリソースだけAPI経由で読み書き可能になることが想定されます（例: 「在庫:read」のみなら参照のみ）。API経由での実際の読み書き制御は検証できていません（本プロジェクトではAPI直接操作は確認対象外）。
> **空保存エラー（2026-06-27実機確認）**: アプリ名・権限未選択のまま保存すると、`アプリ名を入力してください` と `権限が選択されていません` が表示されます。補足文は「アプリが持つ権限は、必ず1つ以上選択されている必要があります。」です。

## 主な操作手順

### アプリを作成してAdmin APIのアクセストークンを取得する

1. 管理画面で `/admin/settings/apps` を開く（設定 > 外部連携グループ > アプリ）
2. 「**アプリを作成**」リンクをクリックし、作成フォーム（`/admin/settings/apps/create`）を開く
3. 「アプリ名」に任意の名前を入力する（例: 社内BI連携用）
4. 「権限」のチェックボックス群から、このアプリに必要な権限（スコープ）をオンにする
5. 保存する
6. アプリ詳細画面に遷移する。「Admin API」セクションの **アクセストークン** と **シークレット** を控える

アプリは削除・失効導線が管理画面上で確認できていないため、検証目的で作成する場合も、名前・権限スコープ・管理者を事前に確認してから保存してください。

> **トークン取得後のリクエスト方法**: トークンを使った具体的な呼び出し方法（エンドポイントURL・認証ヘッダの形式・クエリ例）は実機未確認です。2026-06-28再確認では、Playground導線は本番ドメインのGraphQL Playgroundをクエリなしで開くところまで確認しましたが、stagingアプリのトークンが自動で渡ることは確認できていません。本ページでは管理画面の表示・操作範囲を扱い、API直接呼び出しの実装詳細は開発元確認または許可されたPlayground実行検証後に補完します。

### Admin APIを Playground で試す

1. アプリ詳細画面を開く
2. 「**Playgroundを開く**」ボタンをクリックする
3. 開いた GraphQL Playground 上でクエリを入力し、Admin APIを試す

### Storefront APIのトークンを発行する

1. アプリ詳細画面を開く
2. 「Storefront API」セクションの「**トークンを発行する**」ボタンをクリックする
3. 発行されたStorefront APIトークンを控える

2026-06-28再確認でも「トークンを発行する」ボタンの表示のみ確認し、秘密値を増やさないため発行操作は未実行です。発行後の再表示・削除・失効手順は開発元確認または発行許可後の検証が必要です。

### Webhookを設定する

1. アプリ詳細画面を開く
2. 「**Webhookを作成する**」ボタンをクリックし、Webhook作成ダイアログを開く
3. 「イベント」でトリガーを選ぶ（「注文の作成」 / 「注文の更新」 / 「在庫の更新」）
4. 「エンドポイント」に通知先のURLを入力する
5. 作成を完了する

2026-06-28追加確認では、Webhookを実作成でき、作成後はアプリ詳細画面のWebhookセクションにイベント名・エンドポイント・形式 `JSON` が表示されます。同日の再確認でも、Webhook単位の編集/削除/停止導線、行チェックボックス、送信成功/失敗ステータスは画面上に表示されませんでした。

### APIのリクエスト履歴を確認する

1. アプリ詳細画面を開く
2. 「**リクエストログを見る**」をクリックする
3. このアプリによるAPIリクエストの履歴を確認する

## 注意点・制約

- アプリの **アクセストークンとシークレットは作成時に自動発行** されます。2026-06-28再確認では、詳細画面・一覧カードの可視操作に再発行・削除・失効の操作導線はありませんでした。管理画面外の削除/失効手順は開発元確認が必要です。
- アプリに付与できる権限は、権限グループと同じ **38リソース × 閲覧（:read）/ 編集（:write）の76種** です。付与しすぎると漏洩時の影響が大きくなるため、**必要最小限のスコープ**を付与してください（例: 在庫の参照だけなら「在庫:read」のみ）。
- Webhookのイベントは「注文の作成」「注文の更新」「在庫の更新」の3種のみです（2026-06-28時点の管理画面UIで再確認）。
- Webhookの送信時にシークレットを使ったリクエスト検証ができることが UI 上示唆されていますが、具体的な検証アルゴリズム・署名ヘッダ名は実機未確認です（候補としてHMAC-SHA256等の方式が想定されますが未確定）。**署名検証方式は現在確認中です。受信側サーバの実装に着手する前に、必ずSQサポートまたは公式ドキュメント（docs.sqstack.com）で最新の仕様をご確認ください。**<!-- 開発元確認: Webhook署名の検証方式・ヘッダ名・アルゴリズム -->
- Shopify等の外部連携（コネクターアプリ経由）は別経路であり、本エリアで扱う「アプリ（Admin API / Storefront API / Webhook）」とは別物です。**名前が同じ「アプリ」で紛らわしいため注意**: このページの「アプリ」はAPIキー（トークン・シークレット）を発行する開発者向けのもの。Shopify/スマレジ等の連携は設定画面の別メニュー（コネクターアプリ経由）でOAuth等により接続するもので、APIキーとは経路・仕組みが異なります。連携の前提は [外部連携のよくある質問](../03-faq/外部連携のよくある質問.md) を参照してください。
- API直接操作（GraphQLの実行結果・レスポンススキーマなど）は読み取り・書き込みを含むため、本プロジェクトでは管理画面の表示・操作のみを「実機確認」としています。
