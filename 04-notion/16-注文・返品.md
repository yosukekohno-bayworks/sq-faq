# 16. 注文・返品

> このページは標準24分類の第16分類です。注文を起点に、在庫・顧客・ポイント・売上の各行へどう連動するかを理解するのが目標です。

> **2026-07-02 追加検証**: stagingで内部GraphQLにより検証用注文を投入し、注文一覧・注文詳細・顧客詳細・返品一覧・返品受取・返品完了・交換返品・売上実績への正負行生成・売上CSV実ファイル・DECREMENT注文の在庫引当/出荷指示生成/出荷完了後の手持ち減少を実機確認しました。Shopify注文は商品/SKUが正式に連携済みでないとSQ既存SKUの在庫検証になりません。2026-07-02時点ではShopify連携の連携済みバリエーションが0件のため、Shopify起点の在庫引当は未検証です。未連携商品のShopify注文 `#1008` / `#1009` はSQへ流入しましたが、出荷済み・返品処理中はSQ側へ反映されず、商品行は `新規` のままでした。

## このエリアで学べること

- 注文管理の構成（注文一覧・下書き・返品）とそれぞれの役割を説明できる
- 注文データ0件の空状態と、データあり状態で確認できた一覧列・詳細表示を区別できる
- 下書き注文の「注文を作成する」ボタンが押せない既知の問題を説明できる
- 注文起点で在庫・出荷指示・顧客・返品・売上実績へどう反映されるか、確認済み範囲と未確認範囲を区別できる
- 返品の `返送待ち` → `返品受取済み` → `返品処理済み` の流れと、注文詳細・返品一覧・売上実績の変化を説明できる

## 機能概要

このエリアはサイドバー「注文管理」配下の3画面から成り立ちます。注文は販売チャネルから自動流入する設計で、SQ管理画面から注文を直接新規作成する導線は確認できていません。検証では内部GraphQLで注文を投入し、データあり状態の画面挙動を確認しました。

| 機能 | 画面URL | 役割 |
|:--|:--|:--|
| 注文管理 | `/admin/orders` | チャネル等から流入した注文の一覧。データあり状態では注文・顧客・チャネル・決済/出荷/返品状況を表示 |
| 下書き | `/admin/draft_orders` | 管理画面から手動起票する注文（ドラフト）の管理。ただし現行では作成ボタンがdisabled相当 |
| 返品 | `/admin/order_returns` | 返品処理の一覧確認。返品自体は注文に紐づき、詳細操作は注文詳細内で行われる |

### できること（2026-06-28実機確認範囲）

- **注文管理**: 空状態では「すべて」タブと空状態のみ。内部GraphQLで注文投入後は、注文一覧に列 `注文 / 制限 / 注文日時 / 顧客 / チャネル / 店舗 / アイテム / 合計 / 決済状況 / 出荷状況 / 返品状況 / タグ` が表示され、注文番号検索・絞り込み・保存ビューが表示された
- **注文詳細**: 検証注文では `支払い済み`、`発送済み`、`返品・交換を作成`、`返金`、`返金履歴`、`出荷指示`、`在庫依頼`、ポイント/会員ランク算出対象、タグが表示された
- **下書き**: `/admin/draft_orders` は表示されるが、「注文を作成する」は `href` なしのdisabled相当で、会社・担当者・購入顧客データがあっても解放されなかった
- **返品**: 空状態では検索・絞り込みUIなし。返品データ投入後は列 `返品 / 返品状況 / 交換出荷状況 / 作成日時` と、絞り込み条件 `キャンセル` / `交換出荷` が表示され、返品処理済みまで一覧に反映された
- **売上実績**: 注文投入で `+1 / ￥1,990` の売上実績明細が作られ、返品完了後に `-1 / -￥1,990` の明細が追加された

### 注文起点の連動（確認済み範囲）

| 連動先 | 実機確認結果 | 確認状態 |
|:--|:--|:--|
| 在庫（販売可能/引当済み/手持ち） | BYPASS注文ではSQ在庫は動かない。DECREMENT注文では注文作成時に販売可能が減り、引当済みが増える。注文由来の出荷指示完了後、数秒後に引当済みと手持ちが減る | ✅ 内部GraphQL注文で確定。Shopify注文は正式連携SKUが0件のため未検証 |
| 顧客管理 | 顧客詳細の「直近の注文」に注文が表示され、初回購入日・最終購入日・購入場所が `FAQ_INTERNAL_GRAPHQL` で更新された | ✅ 検証注文で確定 |
| ポイント/会員ランク | 2026-07-01の `FAQ-ORDER-DECREMENT-SHIP-202607010132` では、注文詳細に `ポイント: 付与対象に含む / ポイント付与 0ポイント`、`会員ランク: 算出対象に含む` と表示。顧客ポイント履歴にも注文由来行は出ない | ⚠️ 内部注文で対象表示と0ポイントは確認済み。正式連携SKU注文での実ポイント付与・返品時減算・ランク再算出は未確認 |

| 出荷管理 | DECREMENT + `requiresFulfillment=true` の注文で出荷指示 `#IO-1034` が `ORDER` / `WAITING` で生成された | ✅ 内部GraphQL注文で確定。Shopify等の外部チャネル流入時は正式連携SKUで再検証待ち |

#### DECREMENT注文で確認した在庫・出荷指示

`FAQ-REMAINING-DECREMENT-20260628135953` を `inventoryBehavior=DECREMENT`、`requiresFulfillment=true` で作成したところ、注文は `PAID` / `UNFULFILLED` になり、出荷指示 `#IO-1034` が `ORDER` / `WAITING` で生成されました。対象SKU `TEST_E2E_20260622_GU_1905_NAVY_M` / 倉庫 `TEST_E2E_20260622_GU倉庫_ON_1905` は、注文前後で `available 1→0`、`committed 0→1`、`onHand 1→1`、`sellableQuantity 1→0` に変化しました。

2026-07-01の `FAQ-ORDER-DECREMENT-SHIP-202607010132` / `#IO-1046` では、注文作成後に在庫が18/0/0/18→17/1/0/18になり、注文由来の出荷指示を完了すると注文は `FULFILLED`、出荷指示は `成功 完了 / 出荷完了` になりました。出荷完了直後は一時的に17/1/0/18のままでしたが、約8秒後に17/0/0/17へ変わりました。したがって、DECREMENT注文は「注文作成で販売可能→引当済み」「出荷完了後に引当済みと手持ちが減る」と説明します。

内部GraphQLの `orderCreate` では `requiresFulfillment=true` のみで出荷指示が作られ、明示的な `inventoryOutboundOrders` 入力は不要でした。ただし、Shopify/スマレジ等の外部チャネルから流入した注文でも同じタイミング・同じ条件で動くかは接続環境で確認が必要です。

Shopify注文で同じ確認をするには、注文商品のShopifyバリエーションがSQ Shopify連携のリスティングで正式に連携済みである必要があります。2026-07-02再確認では、`stack-ps-yosuke.myshopify.com` の連携済み商品数は0、連携済みバリエーション一覧も空でした。未連携商品やSKU文字列だけを合わせた手作成商品で注文しても、SQ側では既存SKUに紐づかず `新規` 行として取り込まれるため、在庫引当の検証対象にしません。

## 主な操作手順

### 下書き注文を作成しようとして失敗する（既知の問題）

1. サイドバーの「注文管理」>「下書き」を開き、`/admin/draft_orders` に遷移する
2. 右上の「注文を作成する」ボタンをクリックする → 遷移しない
3. 直接 `/admin/draft_orders/create` にアクセスする → 「予期せぬエラーが発生しました」画面になる

会社・担当者・内部GraphQLで作成した購入顧客が存在しても、ボタンは解放されませんでした。購入顧客データの存在だけでは有効化条件を満たしません。

### 返品を確認する

1. `注文管理 > 返品` で `/admin/order_returns` を開く
2. 返品データがない場合は空状態のみ表示される
3. 返品データがある場合は、返品管理コード・返品状況・交換出荷状況・作成日時が表示される
4. 返品行の単体URLへ直接アクセスしても詳細画面は開かず、「このページは存在しないようです」になる
5. 返品の受領・完了状態は、注文詳細内の返品ブロックで確認する

### 検証注文で確認した返品ステータスの流れ

| 操作/状態 | GraphQL状態 | 注文詳細/一覧の表示 | 操作可否 |
|:--|:--|:--|:--|
| 返品作成直後 | `WAITING` | `情報 返送待ち`、`1個中の0個を受け取り済み` | `商品を受け取る`、`返品をキャンセルする` が表示 |
| 商品受取後 | `RECEIVED` | `注意 返品受取済み`、`1個中の1個を受け取り済み` | `返品を完了する` が表示。受取ボタンは消える |
| 返品完了後 | `REFUNDED` | `成功 返品処理済み`、`1個中の1個を受け取り済み` | 受取/完了/キャンセルボタンは消える |

返品完了後、注文詳細の合計は `￥0` になり、支払い情報に `支払い済み ￥1,990`、`返金済み -￥1,990`、`純支払い額 ￥0` が表示されました。

### 交換返品で確認した流れ

`FAQ-REMAINING-EXCHANGE-20260628140117` に交換明細付き返品 `FAQ-REMAINING-EXCHANGE-20260628140117-R1` を作成して確認しました。

| 操作/状態 | GraphQL状態 | 交換明細 | 交換出荷指示 |
|:--|:--|:--|:--|
| 交換返品作成直後 | `WAITING` | `hasExchangeLineItems=true`、`exchangeLineItemCount=1` | まだ生成されない（`null`） |
| 商品受取後 | `RECEIVED` | 交換明細は維持 | まだ生成されない（`null`） |
| 返品完了後 | `REFUNDED` | 交換明細は維持 | `#IO-1040` が `ORDER_EXCHANGE` / `WAITING` で生成 |

## 注意点・制約

### 注文はチャネルから流入する（手動作成導線なし）

注文は販売チャネル（Shopify等）から自動的に流入する設計です。2026-06-28再確認でも、注文管理一覧に手動作成ボタンはなく、`/admin/orders/create` に直接アクセスすると「予期せぬエラーが発生しました」画面になります。

今回の検証注文は、画面表示と連動先を確認するために内部GraphQLで投入したものです。Shopify/OmnibusCore/スマレジ/リテールポータル等から実際に流入した注文の同期挙動とは分けて扱ってください。

### BYPASS注文とDECREMENT注文の違い

`inventoryBehavior=BYPASS` の注文では、注文詳細に「この注文による在庫変動は無視されます」と表示され、販売可能・引当済み・手持ちは動きませんでした。`inventoryBehavior=DECREMENT` かつ `requiresFulfillment=true` の注文では、販売可能が減り、引当済みが増え、出荷指示が生成されました。

### 返品は注文詳細内で扱う

返品一覧や `/admin/order_returns/create` から返品を新規作成する導線はありません。返品データが存在する場合でも、単体詳細URL `/admin/order_returns/{id}` は存在しない画面になります。返品の状態確認・受領・完了は注文詳細内の返品ブロックで扱います。

### 返品・キャンセル・返金の運用分担（公式）

返品・キャンセル・返金は、原則として注文の作成元（Shopify等のチャネル側管理画面）で操作し、その結果がSQへ反映される設計です。外部ツールの設計によってはSQ側で操作する場合もあると公式ガイドに明記されています（出典: 公式 `/docs/guide/orders`）。2026-06-29のShopify実機確認では、キャンセルはデータ連携OFF中にSQへ反映されず、返金はデータ連携OFF中でもSQで `返金済み` になりました。2026-07-02に全チェックONで `#1008` を返金したところ、SQ注文の決済状況と注文情報内の決済履歴には返金が反映されましたが、返金履歴タブは空でした。注文在庫引当・返品・出荷は正式連携SKUでの再検証が必要です。

### 顧客データもチャネル流入が基本

顧客管理一覧に手動新規作成ボタンはありません。内部GraphQLで検証用購入顧客を作成すると注文履歴・購買サマリーへの反映は確認できましたが、通常運用では販売チャネルからの流入が前提です。会社担当者は購入顧客とは別データで、顧客管理一覧やディスカウント対象顧客候補には出ません。

### 未完了バッジの表示ルール

注文管理は左サイドバーのメニュー項目に「未完了件数」を示すバッジが表示されます。

| 仕様 | 内容 |
|:--|:--|
| 表示タイミング | 未完了の出荷指示を持つ注文が1件以上ある場合に表示 |
| 0件のとき | バッジは表示されない |
| 100件超のとき | 「100+」と表示（正確な件数は表示されない） |

2026-06-28追加検証では、内部GraphQLの `OrderCountFilter` が `inventoryOutboundOrderWorkingStatuses` / `retailLocationID` / `isCancelled` のみを受け付け、`paymentStatus` 条件を持たないことを確認しました。DECREMENT注文で `WAITING` の出荷指示 `#IO-1034` ができると `WAITING/NON_TERMINAL` 件数が `0→1` になり、交換返品完了で `ORDER_EXCHANGE` の `#IO-1040` ができた後、サイドバーにも `未完了の注文 2件` / `未完了の出荷指示 2件` が表示されました。したがって、注文管理の未完了件数は決済状態ではなく、未完了の出荷指示ステータスに紐づく件数として案内します。

### 出荷実績登録後の取り消し不可（不可逆）

注文に紐づく出荷実績の登録は確定操作で、登録後に取り消し・変更はできません（出荷管理の仕様）。誤って登録した場合は在庫管理 > 調整伝票から在庫数を修正してください。
