# 16. 注文・返品

> このページは標準24分類の第16分類です。注文を起点に、在庫・顧客・ポイント・売上の各行へどう連動するかを理解するのが目標です。

> **2026-07-02 追加検証 / 2026-07-03 Shopifyリスティング再確認**: stagingで検証用の注文データを用意し、注文一覧・注文詳細・顧客詳細・返品一覧・返品受取・返品完了・交換返品・売上実績への正負行生成・売上CSV実ファイル・DECREMENT注文の在庫引当/出荷指示生成/出荷完了後の手持ち減少を実機確認しました。管理画面UIから注文を直接新規作成する導線が確認できなかったため、Stack社へ連動仕様を確認する目的で用意した検証データです。Shopify注文は商品/SKUが正式に連携済みでないとSQ既存SKUの在庫検証になりません。2026-07-03再確認でもShopifyリスティングは連携済み商品数0、未連携商品数5で、商品連携用トークン未設定のため初回自動連携も有効化できない状態でした。したがって、Shopify起点の在庫引当は未検証です。未連携商品のShopify注文 `#1008` / `#1009` はSQへ流入しましたが、出荷済み・返品処理中はSQ側へ反映されず、商品行は `新規` のままでした。

## このエリアで学べること

- 注文管理の構成（注文一覧・下書き・返品）とそれぞれの役割を説明できる
- 注文データ0件の空状態と、データあり状態で確認できた一覧列・詳細表示を区別できる
- 下書き注文（客注）の作成入口がリテールポータル側にあり、本体管理画面の「注文を作成する」ボタンは無効のままであることを説明できる
- 注文起点で在庫・出荷指示・顧客・返品・売上実績へどう反映されるか、確認済み範囲と未確認範囲を区別できる
- 返品の `返送待ち` → `返品受取済み` → `返品処理済み` の流れと、注文詳細・返品一覧・売上実績の変化を説明できる

## 機能概要

このエリアはサイドバー「注文管理」配下の3画面から成り立ちます。注文は販売チャネルから自動流入する設計で、**本体管理画面**から注文を直接新規作成する導線はありません。手動起票の唯一の入口は**リテールポータル側の客注（下書き注文）**です（2026-07-03実機確認。詳細は **19.店舗業務・リテールポータル**）。検証では、staging環境に検証用注文データを用意し、データあり状態の画面挙動を確認しました。

| 機能 | 画面URL | 役割 |
|:--|:--|:--|
| 注文管理 | `/admin/orders` | チャネル等から流入した注文の一覧。データあり状態では注文・顧客・チャネル・決済/出荷/返品状況を表示 |
| 下書き | `/admin/draft_orders` | リテールポータルで作成された客注（下書き注文）の一覧。本体側の作成ボタンはdisabled相当のまま |
| 返品 | `/admin/order_returns` | 返品処理の一覧確認。返品自体は注文に紐づき、詳細操作は注文詳細内で行われる |

### できること（2026-06-28実機確認範囲）

- **注文管理**: 空状態では「すべて」タブと空状態のみ。検証用注文投入後は、注文一覧に列 `注文 / 制限 / 注文日時 / 顧客 / チャネル / 店舗 / アイテム / 合計 / 決済状況 / 出荷状況 / 返品状況 / タグ` が表示され、注文番号検索・絞り込み・保存ビューが表示された
- **注文詳細**: 検証注文では `支払い済み`、`発送済み`、`返品・交換を作成`、`返金`、`返金履歴`、`出荷指示`、`在庫依頼`、ポイント/会員ランク算出対象、タグが表示された
- **下書き**: `/admin/draft_orders` は表示されるが、「注文を作成する」は `href` なしのdisabled相当で、会社・担当者・購入顧客データがあっても解放されなかった。2026-07-03に、リテールポータルで作成した客注が本体のこの一覧に表示されることを確認（列: 注文/日時/顧客/店舗/アイテム/販売員/注文/合計）
- **返品**: 空状態では検索・絞り込みUIなし。返品データ投入後は列 `返品 / 返品状況 / 交換出荷状況 / 作成日時` と、絞り込み条件 `キャンセル` / `交換出荷` が表示され、返品処理済みまで一覧に反映された
- **売上実績**: 注文投入で `+1 / ￥1,990` の売上実績明細が作られ、返品完了後に `-1 / -￥1,990` の明細が追加された

### 注文起点の連動（確認済み範囲）

検証では用途別に2種類の注文を使いました。

- **在庫を動かさない検証注文**: 顧客履歴・返品・売上実績への反映を見るための注文です。SQ在庫は変わりません（内部値: `inventoryBehavior=BYPASS`）。
- **在庫を動かす検証注文**: 実際の販売に近い形で、注文作成時に在庫を確保し、出荷作業まで作られるかを見るための注文です（内部値: `inventoryBehavior=DECREMENT`、`requiresFulfillment=true`）。

| 連動先 | 実機確認結果 | 確認状態 |
|:--|:--|:--|
| 在庫（販売可能/引当済み/手持ち） | 在庫を動かさない注文ではSQ在庫は変わらない。在庫を動かす注文では、注文作成時に販売可能が減り、引当済みが増える。出荷完了後に引当済みと手持ちが減る | ✅ staging検証注文で確認済み。Shopify注文は正式連携SKUが0件のため未検証 |
| 顧客管理 | 顧客詳細の「直近の注文」に注文が表示され、初回購入日・最終購入日・購入場所が検証用チャネル名 `FAQ_INTERNAL_GRAPHQL` で更新された | ✅ 検証注文で確定 |
| ポイント/会員ランク | 2026-07-01の `FAQ-ORDER-DECREMENT-SHIP-202607010132` では、注文詳細に `ポイント: 付与対象に含む / ポイント付与 0ポイント`、`会員ランク: 算出対象に含む` と表示。顧客ポイント履歴にも注文由来行は出ない | ⚠️ staging検証注文で対象表示と0ポイントは確認済み。正式連携SKU注文での実ポイント付与・返品時減算・ランク再算出は未確認 |
| 売上実績（会計） | 注文投入で正の売上行、返品完了で負の売上行が作成された | ✅ 検証注文で確定 |
| 出荷管理 | 在庫を動かす注文で、かつ出荷対象の商品だった場合、出荷管理に注文由来の出荷指示 `#IO-1034` が自動作成され、「出荷待ち」として表示された | ✅ staging検証注文で確認済み。Shopify等の外部チャネル流入時は正式連携SKUで再検証待ち |

#### 在庫を動かす注文で確認したこと

ここで確認したいのは、**注文が入ったときに、SQ在庫がどのタイミングで減るか**です。結論は次の通りです。

1. 注文が作られた時点では、商品はまだ倉庫にあります。
2. ただし、その商品は注文用に確保されるため、販売可能数は減り、引当済みが増えます。
3. 出荷が完了すると、確保していた在庫が実際に倉庫から出た扱いになり、引当済みと手持ちが減ります。

2026-06-28の検証注文 `FAQ-REMAINING-DECREMENT-20260628135953` では、注文作成直後に次の変化を確認しました。

| 項目 | 注文前 | 注文作成後 | 意味 |
|:--|:--|:--|:--|
| 販売可能 | 1 | 0 | もう他の注文には売れない |
| 引当済み | 0 | 1 | この注文のために1点確保された |
| 手持ち | 1 | 1 | まだ出荷していないので、物理在庫は倉庫に残っている |
| 売可 | 1 | 0 | 販売可能数と同じく、売れる数は0になった |

同時に、出荷管理には注文由来の出荷指示 `#IO-1034` が自動作成され、画面上は「出荷待ち」として表示されました。これは、SQが「この注文の商品を出荷してください」という作業を自動で作った、という意味です。内部値では出荷指示の種類が `ORDER`、作業状態が `WAITING` ですが、資料上は「注文由来 / 出荷待ち」と説明すれば十分です。

2026-07-01の検証注文 `FAQ-ORDER-DECREMENT-SHIP-202607010132` / 出荷指示 `#IO-1046` では、出荷完了後の変化も確認しました。

| タイミング | 販売可能 | 引当済み | 取置中 | 手持ち | 意味 |
|:--|:--|:--|:--|:--|:--|
| 注文前 | 18 | 0 | 0 | 18 | 18点売れる状態 |
| 注文作成後 | 17 | 1 | 0 | 18 | 1点が注文用に確保されたが、まだ倉庫には残っている |
| 出荷完了後 | 17 | 0 | 0 | 17 | 確保していた1点が倉庫から出た |

出荷完了直後は一時的に `販売可能17 / 引当済み1 / 取置中0 / 手持ち18` のままでしたが、約8秒後に `販売可能17 / 引当済み0 / 取置中0 / 手持ち17` に変わりました。そのため、出荷完了後の在庫確認は数秒待ってから見る必要があります。

今回の検証では、注文データ側に「出荷が必要な商品」と分かる情報が入っていたため、出荷指示データを別途作らなくても `#IO-1034` / `#IO-1046` が自動生成されました。ただし、これはstagingに用意した検証注文での確認です。Shopify/スマレジ等から入った実注文でも同じように動くかは、正式に連携済みのSKUで別途確認が必要です。

Shopify注文で同じ確認をするには、注文商品のShopifyバリエーションがSQ Shopify連携のリスティングで正式に連携済みである必要があります。2026-07-03再確認では、`stack-ps-yosuke.myshopify.com` のリスティングは連携済み商品数0、未連携商品数5でした。未連携商品やSKU文字列だけを合わせた手作成商品で注文しても、SQ側では既存SKUに紐づかず `新規` 行として取り込まれるため、在庫引当の検証対象にしません。

Shopifyで同じ検証を行う場合は、次の順で確認します。

1. SQのShopify連携リスティングで、対象商品のShopify商品ID/バリエーションIDがSQの商品/SKUに紐づいていることを確認する。
2. その連携済みバリエーションを使ってShopify側でテスト注文を作成する。
3. SQ注文管理に注文が流入し、商品行が `新規` ではなく既存SKUとして表示されることを確認する。
4. 注文作成直後に、対象SKUの販売可能・引当済み・手持ちがどう変わるかを確認する。
5. Shopify側またはSQ側の出荷操作後に、SQの出荷状況・出荷指示・在庫数がどう変わるかを確認する。
6. Shopify側で返金/キャンセル/返品を作成し、SQ注文詳細・返品一覧・売上実績へ反映されるかを確認する。

この検証は、単にShopifyで同じSKU文字列の商品を手作成するだけでは成立しません。SQ側で「正式連携SKU」として認識されていることが前提です。

## 主な操作手順

### 下書き注文（客注）を作成する

下書き注文は**リテールポータル側で作成します**（手順の詳細は **19.店舗業務・リテールポータル**）。

1. リテールポータル（`retail.sqstackstaging.com/admin`）にログインし、販売員を選択する
2. 在庫詳細の「下書き注文を作成」または客注管理の「注文を作成する」から作成フォームを開く
3. 明細・顧客・配送先住所を設定して「保存する」
4. 「下書き注文を完了する」で注文化する（巻き戻し不可）。完了すると連携設定の在庫ロケーションで引当が発生し、出荷指示が自動作成される

**本体管理画面からは作成できません（既知の挙動）**: 本体の「注文を作成する」ボタンは `href` なしのdisabled相当のままで、直接 `/admin/draft_orders/create` にアクセスすると「予期せぬエラーが発生しました」になります（2026-07-03再確認）。客注の作成をリテールポータルに限定する設計とみられますが、本体ボタンが常時無効なのが仕様か不具合かは未確認です。

### 返品・交換を作成する

返品と交換返品は別画面ではなく、注文詳細の `返品・交換を作成` から同じ画面で作成します。

1. 注文詳細で `返品・交換を作成` を押す
2. `/admin/orders/{order_id}/return` の `返品を作成する` 画面が開く
3. 返品する商品の数量を入力する
4. 通常返品にする場合は、交換先を選ばずに保存する
5. 交換返品にする場合は、数量入力後に出る `交換先のバリエーションを選択する` から交換品のSKUを選ぶ
6. 交換品を出す `出荷倉庫` を選び、保存する

2026-07-03実機確認では、`FAQ-COMPANY-ORDER-202607011221` の注文詳細に `返品・交換を作成` が表示され、数量 `1` を入れると `交換先のバリエーションを選択する` が表示されました。交換先SKUの選択ダイアログは `交換アイテムを追加する` という見出しで、列は `バリエーション / 商品コード / SKU` です。交換先SKUを選ぶと、フォームに `交換先の商品バリエーション` と `出荷倉庫` が表示されます。画面内には `交換アイテムは、返品された各アイテムごとに同じ数量で作成されます。` と表示されるため、返品数量と交換品数量は同数になります。

![交換返品の作成フォーム。返品数量を入力し、交換先のバリエーションと出荷倉庫を選択した状態](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/16-order-return-exchange-form.png)

### 返品を確認する

1. `注文管理 > 返品` で `/admin/order_returns` を開く
2. 返品データがない場合は空状態のみ表示される
3. 返品データがある場合は、返品管理コード・返品状況・交換出荷状況・作成日時が表示される
4. 返品行の単体URLへ直接アクセスしても詳細画面は開かず、「このページは存在しないようです」になる
5. 返品の受領・完了状態は、注文詳細内の返品ブロックで確認する

### 検証注文で確認した返品ステータスの流れ

| 操作/状態 | 状態値 | 注文詳細/一覧の表示 | 操作可否 |
|:--|:--|:--|:--|
| 返品作成直後 | `WAITING` | `情報 返送待ち`、`1個中の0個を受け取り済み` | `商品を受け取る`、`返品をキャンセルする` が表示 |
| 商品受取後 | `RECEIVED` | `注意 返品受取済み`、`1個中の1個を受け取り済み` | `返品を完了する` が表示。受取ボタンは消える |
| 返品完了後 | `REFUNDED` | `成功 返品処理済み`、`1個中の1個を受け取り済み` | 受取/完了/キャンセルボタンは消える |

返品完了後、注文詳細の合計は `￥0` になり、支払い情報に `支払い済み ￥1,990`、`返金済み -￥1,990`、`純支払い額 ￥0` が表示されました。

### 通常返品と交換返品の違い

通常返品は「返ってきた商品を受け取り、返品を完了する」処理です。交換返品はそれに加えて「代わりの商品を出荷する」処理が入ります。

| 種別 | 作成時の違い | 完了までの流れ | 出荷指示 |
|:--|:--|:--|:--|
| 通常返品 | 返品数量だけを指定する | `返送待ち` → `返品受取済み` → `返品処理済み` | 交換品の出荷指示は作られない |
| 交換返品 | 返品数量に加えて、交換先SKUと出荷倉庫を指定する | 返品側は通常返品と同じく受領・完了する | 返品完了後に、交換品を出すための出荷指示が作られる |

2026-07-03実機確認では、通常返品 `FAQ-YAMATO-CSV-20260701134406-R1` は注文詳細の返品ブロックに `情報 返送待ち`、`1個中の0個を受け取り済み`、`返品をキャンセルする`、`商品を受け取る` が表示され、交換品や関連出荷指示は表示されませんでした。返品一覧では列 `交換出荷状況` はありますが、この通常返品では空欄でした。

交換返品は `FAQ-REMAINING-EXCHANGE-20260628140117` で確認しました。作成直後と商品受取後は、まだ交換品の出荷指示は作られません。返品を完了したタイミングで、交換品を出すための出荷指示 `#IO-1040` が生成されました。`#IO-1040` は出荷管理に表示され、出荷明細は `TEST_E2E_20260622_GU_1905_NAVY_M` が1個、作成元は `FAQ-REMAINING-EXCHANGE-20260628140117` です。2026-07-03時点ではこの出荷指示はキャンセル済みですが、交換返品から交換品用の出荷指示が作られること自体は実機で確認済みです。

## 注意点・制約

### 注文はチャネルから流入する（手動作成導線なし）

注文は販売チャネル（Shopify等）から自動的に流入する設計です。2026-06-28再確認でも、注文管理一覧に手動作成ボタンはなく、`/admin/orders/create` に直接アクセスすると「予期せぬエラーが発生しました」画面になります。

今回の検証注文は、画面表示とSQ内部の連動先を確認するためにstaging環境へ用意したデータです。通常ユーザー向けの注文作成導線で作成した注文でも、Shopify等から実際に流入した注文でもありません。Shopify/OmnibusCore/スマレジ/リテールポータル等から実際に流入した注文の同期挙動とは分けて扱ってください。

### 在庫を動かさない注文と在庫を動かす注文の違い

在庫を動かさない検証注文（内部値: `inventoryBehavior=BYPASS`）では、注文詳細に「この注文による在庫変動は無視されます」と表示され、販売可能・引当済み・手持ちは動きませんでした。

在庫を動かす検証注文（内部値: `inventoryBehavior=DECREMENT`）では、注文作成時に販売可能が減り、引当済みが増えました。さらに出荷対象の商品だったため、出荷管理に注文由来の出荷指示が自動作成されました。

### 返品は注文詳細内で扱う

返品一覧や `/admin/order_returns/create` から返品を新規作成する導線はありません。返品データが存在する場合でも、単体詳細URL `/admin/order_returns/{id}` は存在しない画面になります。返品の状態確認・受領・完了は注文詳細内の返品ブロックで扱います。

### 返品・キャンセル・返金の運用分担（公式）

返品・キャンセル・返金は、原則として注文の作成元（Shopify等のチャネル側管理画面）で操作し、その結果がSQへ反映される設計です。外部ツールの設計によってはSQ側で操作する場合もあると公式ガイドに明記されています（出典: 公式 `/docs/guide/orders`）。2026-06-29のShopify実機確認では、キャンセルはデータ連携OFF中にSQへ反映されず、返金はデータ連携OFF中でもSQで `返金済み` になりました。2026-07-02に全チェックONで `#1008` を返金したところ、SQ注文の決済状況と注文情報内の決済履歴には返金が反映されましたが、返金履歴タブは空でした。注文在庫引当・返品・出荷は正式連携SKUでの再検証が必要です。

### 顧客データもチャネル流入が基本

顧客管理一覧に手動新規作成ボタンはありません。検証用に用意した購入顧客では注文履歴・購買サマリーへの反映を確認できましたが、通常運用では販売チャネルからの流入が前提です。会社担当者は購入顧客とは別データで、顧客管理一覧やディスカウント対象顧客候補には出ません。

### 未完了バッジの表示ルール

注文管理は左サイドバーのメニュー項目に「未完了件数」を示すバッジが表示されます。

| 仕様 | 内容 |
|:--|:--|
| 表示タイミング | 未完了の出荷指示を持つ注文が1件以上ある場合に表示 |
| 0件のとき | バッジは表示されない |
| 100件超のとき | 「100+」と表示（正確な件数は表示されない） |

2026-06-28追加検証では、注文件数の内部的な絞り込み条件が `inventoryOutboundOrderWorkingStatuses` / `retailLocationID` / `isCancelled` で、`paymentStatus` 条件を持たないことを確認しました。DECREMENT注文で `WAITING` の出荷指示 `#IO-1034` ができると `WAITING/NON_TERMINAL` 件数が `0→1` になり、交換返品完了で `ORDER_EXCHANGE` の `#IO-1040` ができた後、サイドバーにも `未完了の注文 2件` / `未完了の出荷指示 2件` が表示されました。したがって、注文管理の未完了件数は決済状態ではなく、未完了の出荷指示ステータスに紐づく件数として案内します。

### 出荷実績登録後の取り消し不可（不可逆）

注文に紐づく出荷実績の登録は確定操作で、登録後に取り消し・変更はできません（出荷管理の仕様）。誤って登録した場合は在庫管理 > 調整伝票から在庫数を修正してください。
