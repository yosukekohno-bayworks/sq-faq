## 0. エグゼクティブサマリ

- **SQ = EC運営の基幹システム（バックエンドERP）**。商品・在庫・注文・顧客・CRM・販売設定を一体で持ち、Shopify／POS／WMS と連携する。フロント（購入画面）ではなく、運営者が裏側を操作する管理システム。
- **システムは広大かつ運用が複雑**。特に「取り寄せ在庫販売」（在庫依頼→確保→移動伝票→入荷→出荷）が最複雑で、ここが提案の主戦場。
- **現行ヘルプ（docs.sqstack.com）は"目次だけ"**。13ページ中12ページが概要1文＋リンクのみで、手順・スクショゼロ。中身があるのは Shopify連携1ページのみ。
- **ギャップは2層**：①ページ自体が無い領域（取り寄せ販売・会計・発注など）、②ページはあるが空の領域（商品・在庫・注文・顧客など）。
- **結論**：FAQ制作は「穴埋め」ではなく **ほぼ全機能を実機ベースでゼロから手順化**する規模。工数はこの前提で算定する。
- **未実装（"TODO"表示）**: 分析、卸売(B2B) の2機能 → 今回の整備対象外にできる。

## 3. 機能別 詳細

### 3-1. 商品管理（/admin/products）
- **できること**: 商品の作成・編集、メディア管理、バリエーション（カラー×サイズ）管理、検索エンジンリスティング、カタログによるグループ化、店舗受取設定。
- **一覧の列**: 商品 / ステータス / 商品コード / 在庫(バリエーション数) / カタログ / 商品タイプ / 製造元。タブ＝すべて/公開中/下書き/アーカイブ済み。
- **詳細画面**: 基本情報（商品名*・説明文※5000字）／メディア／バリエーション（カラー: BEIGE/GRAY/NAVY/BLACK、サイズ: XS〜XL）／検索エンジンリスティング／ステータス（公開中・下書き）／商品分類（商品タイプ・製造元・ブランド）／タグ。
- **サブ機能**: カタログ（商品グループ化・現在空）、店舗受取（店舗受取可能バリエーション・現在空）。
- **サンプルデータ**: UNIQLO系商品6件（全ステータス下書き）。
- **FAQ論点**: ステータス「情報/下書き」の違い、バリエーション追加手順、カタログの使い方とチャネル紐付け。

### 3-2. 在庫管理（/admin/inventory_items）
- **できること**: SKU×ロケーション単位の在庫数確認、移動・調整・取置の伝票管理。
- **在庫数量4区分**: 販売不可 / 確定済み / 販売可能 / 手持ち（販売可能 = 手持ち − 確定済み − 販売不可。標準的な在庫恒等式だが全SKU=0のため未検算・要確認）。← 必須のFAQトピック。<!-- 訂正2026-06-07: 04-inventory.md L117・CONFIRMATION-LIST.md L14と整合。実機検算未了のため断定を緩和 -->
- **伝票3種**:
  - 移動伝票: ロケーション間移動（タブ: すべて/出荷作業/一部受領済み/受領済み/キャンセル）
  - 調整伝票: 棚卸し差異・廃棄等の手動調整（タブ: 未実施/実施済み/キャンセル）
  - 取置伝票: 在庫確保（タブ: 未処理/処理済み）
- **FAQ論点**: 在庫4区分の定義、3伝票の使い分け、ロケーション切り替え（「物流倉庫」ボタン）。

### 3-3. 注文管理（/admin/orders）
- **できること**: 販売チャネル（Shopify等）から流入した注文の管理、追跡、顧客通知。下書き注文、返品。
- **重要仕様**: 注文は**チャネル経由で流入が主**。管理画面の「下書き」から手動作成も可能だが、**顧客データ（purchasing_customers）が無いと「注文を作成する」が無効状態**（`<a>` タグ＋`Polaris-Button--disabled` CSS クラス、`pointer-events:none`。HTML disabled 属性ではなくCSSで制御）。会社データ・担当者データがある状態でも変化しない（2026-06-07実証済み）。<!-- 訂正: 旧「ボタンがdisabled」→ 正確にはCSSクラスで制御された`<a>`タグの無効化 -->
- **現状**: 注文データ0件のため、注文詳細画面・下書き作成フォームは未確認。
- **FAQ論点**: 注文はどこから来るか、下書き注文の作り方と前提、返品フロー。

### 3-4. 顧客管理（/admin/purchasing_customers）
- **重要仕様**: **管理画面から顧客を直接作成できない**（作成ボタン無し・/createはエラー）。Shopify等のチャネル経由で自動生成される設計と推測（未検証）。会社の「担当者」は purchasing_customers とは別エンティティ（担当者を作成しても顧客管理に出現しない、2026-06-07実証済み）。<!-- 訂正: 旧「Shopify等のチャネル経由で自動生成される設計」→ 推測マーカー追加。断定は不正確 -->
- インポートは「ポイント一括付与」のみ。サブメニュー「会社」＝B2B法人顧客。
- **現状**: 顧客0件のため詳細画面未確認。
- **FAQ論点**: 「顧客はどうやって登録するのか」← ユーザーが必ず疑問に持つ最重要Q。

### 3-5. 発注管理（/admin/inventory_purchase_orders）
- **できること**: 取引先（仕入先）への発注伝票作成。
- **作成フォーム**: 取引先*・テナント*（ユニクロ）・通貨（5通貨・既定 日本円）／商品追加。
- **重要**: 発注伝票が**入荷管理の起点とされる設計**（入荷指示は発注から生成される設計と推測）。ただし2026-06-07時点でGraphQLバグにより発注伝票の保存が不可（`inventoryItemID` unknown field）のため、発注経由の入荷生成は未実証。実証済みの入荷指示生成元は移動伝票のみ。通貨が発注ごとに選べる＝国際仕入れ対応。<!-- 訂正(2026-06-07): 断定から推測+バグ状況の明記に修正 -->
- **FAQ論点**: 発注→入荷の連携フロー、取引先マスタの登録。

## 5. 重要仕様・落とし穴（FAQの源泉）

ユーザーが必ずつまずく＝優先度の高いQ&A候補。

1. **顧客は管理画面から作れない** — チャネル連携経由で自動生成と推測（未検証・Shopify接続環境が必要）。会社の「担当者」は purchasing_customers とは別エンティティで顧客管理には現れない（2026-06-07実証済み）。「顧客の登録方法は？」<!-- 訂正: 旧「チャネル連携経由で自動生成」断定 → 推測マーカー追加 -->
2. **下書き注文は顧客が無いと作れない** — 「注文を作成する」は `<a>` タグ＋`Polaris-Button--disabled` CSS クラスで無効化（`pointer-events:none`）。会社データ・担当者データがある状態でも変化しない（2026-06-07実証済み）。前提がUIから読めない。<!-- 訂正: 旧「ボタンが disabled」→ 正確には`<a>`タグにCSSクラス disabled が付与された状態。HTML disabled 属性ではない -->
3. **入荷管理に作成ボタンが無い** — 移動伝票から自動生成（実証済み）。発注伝票からの生成は未実証（GraphQLバグ継続中）。「入荷はどこから作る？」<!-- 訂正(2026-06-07): 「発注伝票から生成」→「移動伝票から生成（実証済み）」 -->
4. **在庫4区分の意味** — 販売不可/確定済み/販売可能/手持ち。
5. **取り寄せ販売の全体像** — 在庫依頼→確保済み→移動伝票→入荷→出荷。最複雑・ヘルプ空白。
6. **ナビ名と画面名の不一致** — 会計=売上実績、ポイント=注文ポイント。
7. **CSVテンプレは外部Googleスプレッドシート** — 場所が分かりにくい。
8. **連携の前提作業** — スマレジはアプリインストール、RecustomerはAPIキー取得が事前に必要。
9. **権限グループの初期値は特権管理者1種のみ** — カスタムグループの追加作成が可能で、実測では「TEST_FAQ_店舗スタッフ（4権限）」を作成済み（2026-06-06検証済み）。店舗スタッフ用のカスタムグループ作成が運用上必要。<!-- 訂正: 旧「1種のみ」→ 正確には「初期値1種、カスタム追加可能」 -->

## 6. FAQ章立ての推奨方針（議事録準拠）

議事録の合意「機能別ではなく**担当者ロール別の目的志向**」「**正確性・精度を優先**（平易化は後工程）」に沿う。

### ロール想定
- **運営者/管理者（本部）**: 商品・在庫・発注・販売設定・CRM・連携・マスタ・権限の設定。
- **店舗スタッフ**: リテールポータル中心、在庫依頼・店舗受取・取り寄せ対応。

### 3分類への落とし込み（既存テンプレ活用）
- **01-by-feature（機能別）**: 各機能の「できること・画面・項目」。本報告書 第3章がほぼ素材。
- **02-by-task（作業別）**: ロール×目的の手順。例「取り寄せ販売を1件処理する」「ディスカウントを作る」「商品を公開する」「発注して入荷する」。
- **03-faq（Q&A）**: 第5章の落とし穴がそのまま種。

### 優先順位（ヘルプ空白×重要度）
1. 取り寄せ販売（オペレーション3画面の通し手順）
2. 発注→入荷→在庫反映
3. 商品・在庫の基本（4区分・伝票）
4. CRM（ディスカウント/ポイント/会員ランク）
5. 連携（スマレジ/OmnibusCore/Recustomer/ロジザード）
6. CSV運用（物流＝ヤマトB2クラウド）
