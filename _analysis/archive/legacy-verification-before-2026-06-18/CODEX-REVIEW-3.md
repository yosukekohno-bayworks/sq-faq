## 結論

既存の実機調査記録は、管理画面のUIラベル・URL・フォーム項目・バリデーション・テストデータ操作の証跡としてかなり強い。一方、公開ヘルプ側の情報が2026-06-05時点の分析より増えており、`01-helpcenter-coverage-gap.md` / `02-helpcenter-pages.md` / `SUMMARY.md` の「既存ヘルプは13ページ中12ページが薄い」「Shopifyだけが使える」という前提は現在の公開ヘルプと齟齬がある。

FAQ執筆前に、公開ヘルプの再クロールと、未確認リストの再分類が必要。

## Critical

### C3-01 公開ヘルプの品質評価が古い

**対象**
- `faq/_analysis/01-helpcenter-coverage-gap.md`
- `faq/_analysis/02-helpcenter-pages.md`
- `faq/_analysis/SUMMARY.md`

**問題**
- 既存分析では「現行ヘルプ13ページ」「12ページthin」「Shopifyだけmedium」としている。
- 2026-06-08時点で `https://docs.sqstack.com/docs/llms.txt` はトップページとして12件を列挙しており、`import-export` は同indexに存在しない。
- 一方で、トップページの子ページは公開されている。例:
  - `products/create-product`: 商品作成・バリエーション作成手順あり
  - `point/handle-returns-and-cancellations`: 返品/キャンセル時のポイント挙動あり
- つまり「索引ページだけ見てthin判定」になっており、子ページの存在が評価に入っていない。

**影響**
- FAQ工数見積もりで「既存ヘルプ本文はほぼ流用不可」と過大評価する可能性。
- 既存ヘルプにすでにある仕様（返品時ポイントなど）を「未確認」として扱い続ける可能性。

**推奨対応**
- `docs/llms.txt` とページ内リンクを起点に再クロールする。
- `02-helpcenter-pages.md` を「トップページ」と「子ページ」に分けて再作成する。
- `SUMMARY.md` の第4章と冒頭サマリを更新する。

### C3-02 `/guide/...` は404という記述が現在と合わない

**対象**
- `faq/_analysis/02-helpcenter-pages.md:4`
- `faq/_analysis/SUMMARY.md:190`

**問題**
- 既存分析は「正規URL形式は `/docs/guide/<slug>`、`/guide/<slug>` は404」としている。
- 2026-06-08時点で以下は閲覧可能だった。
  - `https://docs.sqstack.com/guide/shopify-integration`
  - `https://docs.sqstack.com/guide/retail-portal-integration`
- さらに `/docs/guide/shopify-integration` と `/guide/shopify-integration` は内容が完全一致ではない。前者は接続詳細5項目、後者は接続詳細10項目を示している。

**影響**
- URL正規化の判断を誤る。
- Shopify連携の「接続詳細ステップ数」を5で断定すると、検索流入で見える旧/別URLの10項目版と矛盾する。

**推奨対応**
- 正規URLは `docs/llms.txt` 掲載の `/docs/guide/...` としつつ、公開中の `/guide/...` との差分を別途記録する。
- Shopify連携FAQでは「公開ヘルプ内に5項目版と10項目版が混在」として、Stack社にどちらを正とするか確認する。

### C3-03 返品/キャンセル時ポイントは「公開ヘルプ上は仕様あり」

**対象**
- `faq/_analysis/CONFIRMATION-LIST.md`
- `faq/_analysis/12-crm.md`
- `faq/_analysis/SUMMARY.md`

**問題**
- 既存確認リストでは「返品/キャンセル時のポイント挙動」は実データ依存の未確認項目。
- 公開ヘルプ `point/handle-returns-and-cancellations` には、利用ポイントの返却条件と、獲得ポイントのマイナスポイント相殺が明記されている。

**影響**
- 「未確認」として完全保留するより、「公開ヘルプ上の仕様はあるが、staging実機では未検証」と扱う方が正確。

**推奨対応**
- `CONFIRMATION-LIST.md` の該当項目を「公開ヘルプ記載あり / 実機データでの再現未確認」に変更する。
- FAQ本文では公開ヘルプを根拠に書く場合、実機未検証の注記を残す。

**問題**

### M3-02 公開ヘルプが示す仕様が、未確認リストに未反映

**対象**
- `faq/_analysis/CONFIRMATION-LIST.md`
- `faq/_analysis/14-settings.md`
- `faq/_analysis/12-crm.md`

**問題**
- 公開ヘルプ `data-management` では、決済方法について「会計監査時に必要な決済種別データを会計システムに送信」と説明している。
- 同じく販売員について、店舗ポータル利用時に販売員コード入力を求め、個人情報閲覧や客注注文作成者を記録できると説明している。
- 既存実機記録では、決済方法チェックボックスの効果や販売員/店舗ポータル連携が未確認として残っている。

**影響**
- 仕様の出典候補があるのに、未確認リスト上では「手がかりなし」に見える。

**推奨対応**
- 確認リストに「公開ヘルプ記載あり、実機未検証」として分類を追加する。
- FAQに書く場合は、管理画面上の実機確認範囲と公開ヘルプ由来の仕様を明示的に分ける。

### M3-03 公開ヘルプの子ページをFAQ素材として再評価すべき

**対象**
- `faq/_analysis/01-helpcenter-coverage-gap.md`
- `faq/_analysis/02-helpcenter-pages.md`

## FAQ執筆時の安全ルール（更新案）

1. 実機Playwright記録で確認済みのUI・URL・フォーム項目は確定情報として使う。
2. 公開ヘルプにしかない仕様は「公開ヘルプ上の仕様」として扱い、stagingで未再現なら未検証注記を付ける。
3. 外部接続後の挙動（Shopify / OmnibusCore / スマレジ / Recustomer / WMS）は、フォーム項目が見えていても実動作を断定しない。
4. 既存ヘルプの評価は、トップページだけでなく子ページを確認してから工数見積もりへ反映する。
5. `/docs/guide/...` と `/guide/...` の差分があるページは、Stack社に正規版を確認する。
