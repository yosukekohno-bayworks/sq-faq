## 移動伝票
- **分類**: 伝票
- **何か**: ロケーション間（例: 物流倉庫 → 店舗）の在庫移動を管理する伝票。手動の単体作成と、取り寄せ販売フロー経由の自動作成の2つの作られ方がある。
- **作成場所**:
  - 単体作成: 在庫管理 > 移動伝票 >「移動伝票を作成する」（`/admin/inventory_movement_orders/create`）
  - 自動作成: 確保済み一覧（`/admin/inventory_allocation_request_confirmations`）の「移動伝票を作成する」ボタン → ダイアログで「移動元」を選択 →「実行する」（トースト「移動伝票を一括作成しました」）
- **主な項目**:
  - 配送元（必須。「選択」ボタンでロケーション選択モーダル。**配送元を選択するまで商品欄がグレーアウト**）
  - 配送先（必須。「選択」ボタンでロケーション選択モーダル）
  - 商品を追加（任意。テキスト検索または「参照」ボタン。配送元選択までdisabled）
- **作用先**:
  - 出荷管理: 移動伝票作成と同時に**出荷指示が自動生成**される（実測例: #IM-1000 → 出荷指示 #IO-1000）
  - 入荷管理: 移動伝票作成と同時に**入荷指示が自動生成**される（実測例: #IM-1000 → 入荷指示 #II-1000）。入荷指示一覧の「種別」列に「移動」、詳細では「在庫移動」と表示される
  - サイドバーのバッジ: 移動伝票作成で「入荷管理 未完了の入荷指示 1件」「出荷管理 未完了の出荷指示 1件」が表示される（実測）
  - 在庫区分: キャンセル実行で「確保していた在庫が『販売可能』区分に戻る」
  - 在庫依頼: 取り寄せ販売フロー経由の移動伝票をキャンセルすると、**元の在庫依頼は「確保済み」のまま残る**（行き止まり。再作成もクローズもできない）
- **ステータス**:
  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 出荷作業（一覧タブ） | 出荷待ちの伝票 | 作成直後の初期状態（実測バッジ:「未完了 出荷待ち」「情報 未完了」） | — |
  | 一部受領済み（一覧タブ） | 一部の商品が入荷済みの伝票 | ⚠️未検証（遷移条件・タブ振り分けルールは未確認） | — |
  | 受領済み（一覧タブ） | すべての商品が入荷済みの伝票 | ⚠️未検証（遷移条件未確認） | — |
  | 完了 入荷完了 | 入荷実績登録後の状態 | 入荷管理で「入荷実績を登録する」実行に連動して**自動遷移**（実測） | 取り寄せ販売フロー経由では「完了」「入荷完了」の2バッジが並ぶ |
  | キャンセル | キャンセルされた伝票 | 詳細の「その他の操作」>「キャンセル」 | 紐づく入荷予定・出荷指示もすべてキャンセル。確保在庫は「販売可能」へ戻る |
  - 移動伝票の詳細画面には**ステータスを直接進めるボタンがない**。出荷指示（出荷管理）と入荷指示（入荷管理）の実績登録に連動して自動的に変わる
  - キャンセル確認ダイアログ原文: 見出し「移動伝票をキャンセルしますか？」/ 本文「移動伝票をキャンセルすると、この伝票に紐づく入荷予定および出荷指示もすべてキャンセルされます。」
- **制約・注意**:
  - 取り寄せ販売フロー経由の伝票は編集不可。注意文原文:「この移動伝票は外部システムから連携されているため、編集（商品の追加や削除など）は制限されています。連携元システムの管理画面より操作を行ってください。」
  - ⚠️未検証: 単体作成した移動伝票に同じ編集制限がかかるか
  - ⚠️未検証: 配送元=配送先の同一ロケーション指定はUIではエラーにならない。保存時のサーバーサイドバリデーションは未確認
  - ⚠️未検証: 単体作成の保存後の遷移先・トースト文言・伝票番号の採番形式（実測の自動作成例は #IM-1000）
  - キャンセル後の出荷指示は出荷管理の一覧タブに表示されなくなる（「条件指定でエクスポート」でのみ追跡可能）
- **出典**: 01-by-feature/移動伝票.md、02-by-task/移動伝票を単体で作成する.md、_analysis/11-operations.md

## 調整伝票
- **分類**: 伝票
- **何か**: 棚卸し差異・廃棄・紛失・見本などによる在庫数の手動増減を記録する伝票。マイナス在庫（手持ちが0未満）の修正にも使う。
- **作成場所**: 在庫管理 > 調整伝票（`/admin/inventory_adjustment_orders/create`）
- **主な項目**:
  - ロケーション（必須。エラー文言:「ロケーションを選択してください」）
  - **理由**（必須。選択肢: 廃棄 / 見本 / 紛失 / 棚卸差異 / その他 の5種類。「破損」「検品」「予備」は選択肢にない。エラー文言:「理由を選択してください」）
  - 商品（商品を検索する）（必須。エラー文言:「商品を1つ以上追加してください」）
  - 増減数（必須。プラスで増加、マイナスで減少。デフォルト0）
  - 詳細画面の表示項目: 作成者 / 作成日 / 実行者（実施済みのみ） / 実行日（実施済みのみ）
- **作用先**:
  - 在庫数: 「実行」で**明細の差分値が在庫数に即時反映**される。ダイアログ原文:「在庫調整を実行することで明細に登録されている差分値が在庫数に反映されます。この処理は巻き戻すことができません。」
- **ステータス**:
  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 未実施 | まだ在庫に反映されていない（バッジ:「○ 未実施」白丸） | 作成・保存 → 未実施。ボタン: キャンセル / 実行 | 在庫未反映。ロケーション・商品・増減数・理由を変更可 |
  | 実施済み | 在庫への反映が完了（バッジ:「● 実施済み」黒丸、実測ではグレーバッジ） | 「実行」→ 確認ダイアログ →「実行する」 | 在庫数に即時反映。**取り消し不可**。全フィールド読み取り専用 |
  | キャンセル | キャンセルされた伝票 | 「キャンセル」ボタン | — |
- **制約・注意**:
  - 未保存の変更がある状態（「⚠ 保存されていない変更があります」バナー表示中）は「実行」ボタンがグレーアウトして押せない。先に「保存する」が必要
  - 実施済みの伝票にも「キャンセル」ボタンは表示される。⚠️未検証: 実施済み伝票にキャンセルを実行したあとのステータスと在庫の動き
  - 在庫詳細画面（`/admin/inventory_items/[id]`）の「販売可能」列の鉛筆アイコン（「販売可能在庫数を編集する」ダイアログ）からも在庫数を調整できる（別経路）
  - 実測データ: 伝票番号 #IA-1000（増減数 +5。「+5」は青いリンク表示。⚠️未検証: 遷移先）
- **出典**: 01-by-feature/調整伝票.md、_analysis/11-operations.md

## 取置伝票
- **分類**: 伝票
- **何か**: 店舗での取置など、特定の用途のために在庫を確保する伝票。**作成・保存と同時に対象在庫が「確保済み」区分に移動する**。
- **作成場所**: 在庫管理 > 取置伝票 >「取置伝票を作成する」（`/admin/inventory_reservation_orders/create`）
- **主な項目**:
  - ロケーション（必須。「選択」ボタンでロケーション選択モーダル）
  - メモ（任意。自由記述。プレースホルダ「例: 店舗での取置用在庫」。調整伝票の「理由」に相当する選択肢欄はなく自由記述のみ）
  - 商品（商品を検索する）（テキスト検索または「参照」ボタン。⚠️未検証: ロケーション・商品の必須/任意のバリデーションと空保存時のエラー文言）
- **作用先**:
  - 在庫区分: 保存時に対象商品の在庫が「**確保済み**」区分に移動。「処理済みとしてマークする」実行時に「確保済み」の在庫が**解放**される
- **ステータス**:
  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 未処理 | まだ処理が完了していない伝票 | 作成・保存 → 未処理 | 在庫が「確保済み」になる（解放まで確保が継続） |
  | 処理済み | 処理が完了した伝票（バッジ:「完了 処理済み」） | 「処理済みとしてマークする」→ 確認ダイアログ →「実行する」 | 確保済みの在庫が解放される |
  - 確認ダイアログ原文: タイトル「処理済みとしてマークする」/ 本文「取置伝票を処理済みにすることで、確保済みの在庫が解放されます。」/ ボタン「キャンセル」「実行する」
  - 一覧タブ: すべて / 未処理 / 処理済み
- **制約・注意**:
  - 未処理状態の詳細画面に「処理済みとしてマークする」以外の独立したキャンセルボタンは画面上で確認できていない。⚠️未検証: 取置伝票のキャンセル操作（公式機能一覧には記載があるが画面未確認）
  - ⚠️未検証: 保存後の遷移先・トースト文言・伝票番号の採番形式
- **出典**: 01-by-feature/取置伝票.md、02-by-task/取置伝票を作成する.md、_analysis/11-operations.md

## 発注伝票
- **分類**: 伝票
- **何か**: 取引先（仕入先）への発注を管理する伝票。仕入れる商品・数量・単価・税率を記載し金額を管理する。
- **作成場所**: 発注管理 >「発注伝票を作成する」（`/admin/inventory_purchase_orders/create`）
- **主な項目**:
  - **取引先***（必須。コンボボックス。「設定 > 取引先」に登録された取引先が選択肢に表示される。エラー文言:「取引先を選択してください」）
  - テナント*（必須。コンボボックス。実測選択肢: ユニクロ。エラー文言:「テナントを選択してください」）
  - 通貨（任意。米ドル / ユーロ / 日本円 / タイ バーツ / シンガポール ドル。デフォルト: 日本円。発注伝票ごとに設定でき国際仕入れに対応）
  - 商品を追加する（必須・1件以上。エラー文言:「商品を1つ以上追加してください」）
  - 明細行: SKU（読み取り専用・自動入力） / 単価（デフォルト値はSKUごとに異なる。実測: 486125-31-XL=¥3600、486102-70-S=¥1800。原価マスタから自動連動と推測⚠️） / 数量（デフォルト0） / 税率（デフォルト10%） / 金額（読み取り専用。**単価×数量の税抜き金額**。税率は金額計算に含まれない。リアルタイム更新） / ×（行削除）
  - 商品追加後・未保存状態ではヘッダーに「⚠ 保存されていない発注伝票」警告と「取り消す」「保存」ボタンが固定表示される
  - 「参照」で開くバリエーション選択ダイアログ: 検索「SKUコードで検索する」/ 絞り込みを追加は「商品コード」のみ / 列: バリエーション・商品コード・SKU / フッター「キャンセル」「選択する」
- **作用先**:
  - 入荷管理: ⚠️未検証 — 発注伝票保存後に入荷指示が自動生成されるかは未実証（「発注伝票から入荷指示が自動生成される設計」は**推測**。実証された入荷指示の生成元は移動伝票のみ）
- **ステータス**: 一覧タブは「すべて」のみ（ステータス別タブは⚠️未検証。発注データ0件のため一覧テーブル列構成も⚠️未検証）
- **制約・注意**:
  - **不具合（2026-06-07時点・2026-06-14再確認でも継続）**: 「作成する」を押すとGraphQLエラー（2026-06-07の調査では `InventoryPurchaseOrderCreate` の `lineItems` の `inventoryItemID` が `unknown field`、HTTP 500。2026-06-14の実機ではPOST /api/graphql が ERR_ABORTED＋「エラーが発生しました。しばらくしてから再度お試しください」）で**保存が一切できない**。アプリ側不具合とみられStack社への確認・修正依頼が必要（※本プロジェクトから報告した記録はない）。詳細画面・一覧列・入荷指示自動生成は検証不可
  - 取引先マスタに登録したデータは発注伝票の「取引先」選択肢に即反映される
- **出典**: 01-by-feature/発注管理.md、_analysis/07-purchase-orders.md、_analysis/11-operations.md

## 取引先
- **分類**: マスタ
- **何か**: 発注伝票で使用する取引先（仕入先）のマスタデータ。
- **作成場所**: 設定 > 取引先（`/admin/settings/suppliers`）>「取引先を作成」（フォーム: `/admin/settings/suppliers/create`。※`/admin/inventory_suppliers` は404）
- **主な項目**:
  - 取引先名*（必須。プレースホルダ: NIKE Japan。空保存時エラー:「取引先名を入力してください」）
  - 取引先コード（任意。プレースホルダ: SUP-001）
  - 保存ボタン:「保存する」→ 一覧へ遷移、トースト「取引先を作成しました」
- **作用先**:
  - 発注伝票作成フォーム: 登録した取引先が「取引先」コンボボックスの選択肢に**すぐ反映**される（実測: TEST_FAQ_Supplier / TEST_FAQ_Supplier2）
- **ステータス**: 一覧タブ「すべて」/「アーカイブ」（アーカイブ済み取引先を表示）
- **制約・注意**:
  - 一覧テーブル列: 名前 / コード
  - 一覧右上の「インポート」ボタンはクリックしても画面遷移・ダイアログが表示されない（2026-06-07時点で**未実装**）
- **出典**: 01-by-feature/発注管理.md、_analysis/07-purchase-orders.md、_analysis/11-operations.md

## 在庫依頼
- **分類**: 伝票
- **何か**: 在庫の確保・移動依頼。取り寄せ販売の起点となる機能（例:「店舗Aが倉庫Bから在庫をX個確保して欲しい」）。
- **作成場所**: 在庫依頼（`/admin/inventory_allocation_requests`）>「在庫依頼を作成する」（`/admin/inventory_allocation_requests/create`）。保存トースト:「在庫依頼を作成しました」
- **主な項目**:
  - 商品バリエーション*（必須。テキスト検索 +「選択」ボタン）
  - 希望数*（必須。数値入力、初期値0）
  - 移動先ロケーション*（必須）
  - リクエスト先ロケーション（任意。⚠️未検証: 意味が不明、在庫の移動元か？）
  - 詳細画面の表示: 引当先ロケーション / 確保済み数量（例: 0 / 2 点） / 残り確保数 / 対応不可ロケーション / チャネル（例: web-admin） / 依頼先
- **作用先**:
  - 引当て:「在庫を引当てる」ボタン → ダイアログ（ロケーション=リクエスト先として設定したロケーションのみ、数量=最大値は希望数）→「引当てる」で「確保済み」へ
  - 在庫区分: ⚠️未検証（推測）— 引当てで「販売可能」区分から「確保済み」区分へ移動
  - 確保済み一覧: 引当て後、確保済み一覧（`/admin/inventory_allocation_request_confirmations`）に行が表示され、移動伝票作成の起点になる
- **ステータス**:
  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 確認待ち | 作成直後（バッジ:「情報 確認待ち」） | 保存 → 確認待ち |  — |
  | 確保済み | 引当て完了（バッジ:「成功 確保済み」。依頼先行には「成功 確保済」バッジ） | 「在庫を引当てる」→「引当てる」 | 確保済み数量が更新（例: 2/2点、残り0点）。「在庫を引当てる」ボタンが消え「その他の操作」のみ残る |
  | 終了 | 一覧タブの1つ | ⚠️未検証（遷移条件の記載なし） | — |
- **制約・注意**:
  - **行き止まりの罠**: 取り寄せ販売フロー経由の移動伝票をキャンセルすると、元の在庫依頼は「確保済み」のまま残り、移動伝票の再作成も在庫依頼のクローズもできない。同じ取り寄せをやり直すには在庫依頼を新規作成する
- **出典**: _analysis/11-operations.md、01-by-feature/移動伝票.md

## 確保済み（在庫引当リクエスト確認）
- **分類**: システム生成データ
- **何か**: 引当て済みの在庫依頼が並ぶ一覧画面（`/admin/inventory_allocation_request_confirmations`）。ここから移動伝票を一括作成する。
- **作成場所**: 在庫依頼の引当て完了で自動的に行が追加される（手動作成不可）
- **主な項目**: テーブル列: 商品 / SKU / 確保数 / 移動元 / 移動先 / 確保日時
- **作用先**:
  - 移動伝票:「移動伝票を作成する」ボタン → ダイアログ（説明文原文:「確保済みの在庫引当リクエストから在庫移動伝票を作成します。対象のロケーションを選択してください。」、入力:「移動元」、移動元未選択時「実行する」はdisabled）→「実行する」で移動伝票を自動作成。トースト「移動伝票を一括作成しました」。実行後は該当行が消え、「移動伝票を作成する」ボタンはdisabledになる
- **制約・注意**: 移動伝票キャンセル後はこの一覧から再作成できない（行き止まり）
- **出典**: _analysis/11-operations.md、01-by-feature/移動伝票.md

## 入荷指示
- **分類**: システム生成データ
- **何か**: 倉庫・ロケーションへの在庫入荷を管理する指示データ。**移動伝票の作成と同時に自動生成される**（実証済み #II-1000）。入荷管理画面から新規作成するボタンはない。
- **作成場所**: 移動伝票の作成に伴い自動生成（入荷管理 `/admin/inventory_inbound_orders` で確認）。発注伝票経由の生成は設計上想定されるが⚠️未検証（GraphQLバグにより未実証）

## 在庫区分（バリエーション詳細の在庫区分カラム）
- **分類**: ステータス群
- **何か**: バリエーション詳細の在庫一覧に表示される在庫の区分カラム。伝票の各操作がこの区分間で在庫を動かす。
- **作成場所**: システム定義（バリエーション詳細画面の在庫一覧で表示。タブ: すべて / 店舗 / 倉庫）
- **主な項目**: カラム構成: ロケーション / 販売可能 / 確定済み / 確保済み / 破損 / 検品 / 予備 / 手持ち
- **作用先**（対象ファイルに記載がある区分間の動き）:
  - 取置伝票の保存 → 在庫が「確保済み」区分へ移動。「処理済みとしてマークする」→ 解放
  - 在庫依頼の引当て → 「販売可能」区分から「確保済み」区分へ移動（⚠️推測・未検証）

  - 調整伝票の実行 → 増減数（差分値）が在庫数に反映
  - 「販売可能」列の鉛筆アイコン → 「販売可能在庫数を編集する」ダイアログで在庫数を直接調整可
- **制約・注意**: 出荷実績/入荷実績の登録による在庫数の増減反映は未実証（社内フロー実証の範囲外）⚠️未検証
