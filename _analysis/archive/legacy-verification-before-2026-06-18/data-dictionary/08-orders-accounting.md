## 注文

- **分類**: 伝票（システム生成データ — 販売チャネルから自動流入）
- **何か**: 販売チャネル（Shopify等）からSQへ流入した受注データ。SQ管理画面で一覧表示・確認・追跡する。
- **作成場所**: ユーザーがSQ内で直接作成する画面はない。販売チャネル（Shopify等）から自動的に流入する設計。Shopify経由の注文は、受注内容の変更などを**Shopify側の管理画面で行い、変更内容がSQへ自動反映**される運用（開発元の説明）。SQ側から操作するのは、ECモール（楽天など）経由の注文の返品処理などが中心。
- **主な項目**:
  - 注文番号（「注文番号で検索する」テキストボックスの検索キー）
  - **タグ（注文タグ）**: 一覧絞り込み「タグ付けされている」「タグ付けされていません」の判定対象。ヤマトB2クラウドCSVエクスポートの「注文タグ（含む）」「注文タグ（除外）」の絞り込みにも作用（例: priority / test）
  - **テナント**: 一覧絞り込み「テナント」の対象
  - **チャネル**: 一覧絞り込み「チャネル」の対象（Shopify等）
  - テーブル列構成: ⚠️未検証（注文データ0件のため未確認）
- **作用先**:
  - 注文管理一覧（/admin/orders）: 一覧表示・絞り込み（テナント / タグ付けされている / タグ付けされていません / チャネル の4種）の対象
  - サイドバーの未完了バッジ: 未完了の注文が1件以上ある場合、注文管理メニューにバッジ表示（0件で非表示、100件超は「100+」表示）
  - 売上実績: CSVエクスポートに「売上実績（注文軸）」があることから、注文データを元に集計される仕組みと推測 — ⚠️未検証（注文データ0件のため未実証）
  - 返品: 返品は注文に紐づく。起票入口は注文詳細画面（/admin/orders/:id）内にあると強い推測 — ⚠️未検証
- **ステータス**: 注文詳細画面の構成・ステータス遷移・出荷との連動は ⚠️未検証（注文データがないため未確認。チャネル接続後に追補予定）
- **制約・注意**:
  - チャネルが1件も接続されていない場合、注文一覧は空（「アイテムが見つかりませんでした」）
  - タブは「すべて」のみ固定（データ量でタブが増える設計ではない。実機確認 2026-06-07）
  - `orders:read` のみ（writeなし）でログインした場合の注文画面差異（返品/キャンセルボタンの表示/非表示等）は ⚠️未検証
- **出典**: 注文管理.md, 注文と出荷のよくある質問.md, 05-orders.md, 09-accounting.md

## 下書き注文（下書き）

- **分類**: 伝票（ユーザー手動起票）
- **何か**: 管理画面から手動で起票する注文。下書き一覧（/admin/draft_orders）で管理する。
- **作成場所**: 注文管理 > 下書き（/admin/draft_orders）の「注文を作成する」ボタン。ただし現在は**無効状態**（クリックしても何も起きない）。
- **主な項目**: 作成フォームの構成（商品追加/顧客指定/金額/配送/支払の項目）は ⚠️未検証（ボタン無効のため到達不可）
- **作用先**:
  - 下書き一覧（/admin/draft_orders）: 一覧表示・「注文番号で検索する」での絞り込み対象（絞り込みは注文番号検索のみ。「絞り込みを追加」「名前を付けて保存」ボタンはなし）
  - 権限: `draft_orders:read`（下書き注文の閲覧権限）/ `draft_orders:write`（下書き注文の編集権限）の対象
- **ステータス**: なし（記録上の記載なし）
- **制約・注意**:
  - 「注文を作成する」は `<a>` タグ＋`Polaris-Button--disabled` CSSクラスの擬似disabled状態（`href` 属性なし、`pointer-events:none`、HTMLの `disabled` 属性は不使用。実機DOM確認 2026-06-07）
  - 会社データ・担当者データが存在しても有効化されない（実機確認 2026-06-07で確定）。「購入顧客（purchasing_customers）」データが必要との推測 — ⚠️未検証（Shopify未接続環境では実証不可）
  - `/admin/draft_orders/create` への直接アクセスは「予期せぬエラーが発生しました」（アプリ内500系エラー。ルートは存在するがデータ条件不足の可能性が高い）
  - 空状態文言: ローディング「注文を読み込んでいます...」→「注文が見つかりませんでした」「絞り込みや検索ワードを変更してみてください」
- **出典**: 注文管理.md, 注文と出荷のよくある質問.md, 05-orders.md

## 返品

- **分類**: 伝票
- **何か**: 注文に紐づく返品処理のデータ。返品一覧（/admin/order_returns）で一覧確認と絞り込みを行う。
- **作成場所**: 返品一覧画面に「返品を作成する」ボタンは存在しない。`/admin/order_returns/create` は「このページは存在しないようです」（404系・ルート未定義）。よって「返品はcreate専用ページや一覧画面からは起票しない」は確定（実機確認 2026-06-07）。起票入口は注文詳細画面内と強い推測 — ⚠️未検証（注文データなしのため実測不可）
- **主な項目**:
  - **キャンセル**（属性/種別）: 「絞り込みを追加」で「キャンセル扱いの返品」として絞り込み可能
  - **交換出荷**（属性/種別）: 「絞り込みを追加」で「交換出荷を伴う返品」として絞り込み可能
- **作用先**:
  - 返品一覧（/admin/order_returns）: 一覧表示・絞り込み（キャンセル / 交換出荷 の2種のみ）の対象
  - Recustomer連携: 返品・交換フローを外部SaaS（Recustomer）に委譲する設計。接続時に返品フロー全体がRecustomerに委譲される可能性 — ⚠️未検証
- **ステータス**: ⚠️未検証（返品フロー本体・ステータス遷移は到達不可のため未確認）
- **制約・注意**:
  - 返品一覧はタブが存在しない（「すべて」タブも表示されない）。注文番号の検索テキストボックスもない
  - 絞り込み選択肢2種（キャンセル / 交換出荷）はデータ0件の空状態での実測値。返品データに応じて選択肢が増えるかは断定不可 — ⚠️未検証
  - Recustomer未接続時のSQ内返品フローの有無 — ⚠️未検証
  - 注文詳細画面内の「返品する」「返金する」「キャンセルする」ボタンの存在と挙動、返品商品選択・数量・理由・返金方法の各UI — ⚠️未検証
- **出典**: 注文管理.md, 05-orders.md

## 購入顧客（purchasing_customers）

- **分類**: システム生成データ（マスタ）
- **何か**: 販売チャネル（Shopify等）との連携によって生成されると推測される顧客データ。下書き注文作成の解放条件と推測されている。
- **作成場所**: SQ内での手動作成入口は未確認。`/admin/purchasing_customers/create` への直接アクセスは「予期せぬエラーが発生しました」（500系）。チャネル経由で自動生成される設計と推測 — ⚠️未検証
- **主な項目**: ⚠️未検証（データ0件）
- **作用先**:
  - 下書き注文の「注文を作成する」ボタン: 購入顧客データが必要と推測（会社データ・担当者データでは解放されないことは実機確認で確定。「会社担当者」は purchasing_customers とは独立した別エンティティ） — ⚠️未検証（有効化条件の実証はShopify接続環境が必要）
- **制約・注意**: 顧客管理（/admin/purchasing_customers）は会社・担当者を作成しても空のまま（担当者 ≠ 購入顧客）
- **出典**: 注文管理.md, 注文と出荷のよくある質問.md, 05-orders.md

## 売上実績

- **分類**: システム生成データ（推測）
- **何か**: 会計機能の唯一のデータ。売上実績一覧（/admin/sale_change_line_items）で確認する。サイドナビ「会計」とサブメニュー「売上実績」は同じ画面に遷移し、現在の「会計」機能は「売上実績」のみで構成される。
- **作成場所**: 手動作成は現時点では不可。「売上実績を作成する」ボタンは**無効（disabled）状態**（`<a>` タグ＋`Polaris-Button--disabled`、`href` 属性なし。`/admin/sale_change_line_items/create` は404）。注文・CSVエクスポートから自動生成される設計と推定 — ⚠️未検証（売上データ0件のため集計挙動は未確認）
- **主な項目**: テーブル列構成は ⚠️未検証（データ0件）
- **作用先**:
  - CSVエクスポート（/admin/csv_export）: 「実績」グループの「売上実績（注文軸）」カテゴリのエクスポート対象。売上実績データの主な出力手段はCSVエクスポート
  - 権限: `sales:read/write`（売上データの閲覧/編集権限）の対象
- **ステータス**: なし（タブなし。記録上の記載なし）
- **制約・注意**:
  - 「売上実績を作成する」ボタンが注文データの有無によって有効になるか — ⚠️未検証
  - 会計画面URL（`/admin/sale_change_line_items`）とCSVエクスポートURL（`csv_export_operation_sale_changes`）は語幹 `sale_change` が共通だが、同一データソースかは未実証 — ⚠️未検証
  - 請求管理・入金管理・損益管理等の機能は存在しない。請求/入金/返金反映/外部会計ソフト連携/税区分/締め処理は記録上の一次根拠なし — ⚠️未検証
- **出典**: 会計（売上実績）.md, 売上実績をCSVエクスポートする.md, 09-accounting.md

## 売上実績（注文軸）CSVエクスポート

- **分類**: システム生成データ（エクスポート実行・実行履歴）
- **何か**: CSVエクスポート画面から売上実績を期間指定でCSV出力する操作とその実行履歴。
- **作成場所**: CSVエクスポート（/admin/csv_export）>「実績」グループ >「売上実績（注文軸）」>「エクスポート」ボタン → 実行フォーム（/admin/csv_export/csv_export_operation_sale_changes/create）で「エクスポートを開始する」
- **主な項目**:
  - **テナント**: 必須（*）。ドロップダウンからエクスポート対象のテナントを選ぶ
  - **開始日時**: 必須（*）。絞り込みを開始する日時
  - **終了日時**: 必須（*）。絞り込みを終了する日時
- **作用先**:
  - メール: エクスポートは非同期処理。完了後、登録メールアドレスへダウンロードリンクが届く（メール文言・リンク有効期限は ⚠️未検証）
  - 「売上実績（注文軸）」一覧画面: 実行履歴を確認できる
- **制約・注意**:
  - 3項目のいずれかが未入力だと「エクスポートを開始する」を押せない・何も起きない
  - 実行後の画面表示変化・完了トーストの有無は ⚠️未検証（実行系操作のため実証未実施）
  - 同様の手順でエクスポート可能な他カテゴリ: ポイント変動履歴（ポイント）/ 在庫（マスターデータ）/ ロケーション（マスターデータ）/ ディスカウントの利用履歴（マスターデータ）/ 商品バリエーション（マスターデータ）/ セール価格（価格）。ポイント変動履歴のフォーム項目は売上実績と同じ構成（テナント・開始日時・終了日時すべて必須）
- **出典**: 売上実績をCSVエクスポートする.md, 会計（売上実績）.md

## ヤマトB2クラウド出荷指示CSV・納品書PDF（条件指定でエクスポート）

- **分類**: システム生成データ（エクスポート）
- **何か**: 出荷管理からヤマトB2クラウド取り込み用CSVと同梱する納品書PDFを同時に生成するエクスポート。ヤマトB2クラウドでの発送処理の入力になる。
- **作成場所**: 出荷管理（/admin/inventory_outbound_orders）右上の「条件指定でエクスポート」ボタン > ドロップダウンで「ヤマトB2クラウド」を選択 → フォーム「条件指定でヤマトB2クラウドのCSVをエクスポートする」で「実行する」
- **主な項目**:
  - **開始日時**: 必須（*）
  - **終了日時**: 必須（*）。デフォルトで現在日時が入力済み
  - 配送先 (国): 任意。マルチ選択可
  - 決済方法: 任意。すべての決済 / 代引きのみ / 代引き以外の決済
  - 出荷作業ステータス: 任意。指定しない / 出荷待ち / 保留中 / 依頼済み / 作業中 / 欠品・要対応 / キャンセル済み / 出荷完了
  - 注文タグ（含む）: 任意。入力したタグを含む出荷指示のみが対象（例: priority）
  - 注文タグ（除外）: 任意。入力したタグを含まない出荷指示のみが対象（例: test）
  - **CSVの出力後に出荷指示のステータスを出荷作業中に変更する**: 任意のチェックボックス（デフォルト: 未チェック）。**ONにするとエクスポートと同時に対象の出荷指示ステータスが変わる**
- **作用先**:
  - 出荷指示のステータス: 上記チェックボックスONでエクスポートと同時に「出荷作業中」へ変更される
  - メール: 非同期処理完了後、「ヤマトB2クラウド取り込み用CSV」と「同梱する納品書PDF」の2つのダウンロードリンクが届く（同じフォームから同時生成。別々の操作は不要）
  - ヤマトB2クラウド（外部システム）: CSVを取り込んで発送処理を行う（外部側の手順はSQ外のため記録なし — ⚠️未検証）
- **制約・注意**:
  - 「条件指定でエクスポート」ドロップダウンの選択肢は「ヤマトB2クラウド」のみ。DHLのエクスポートは存在しない（DHLは出荷実績インポートのみ対応）
  - 「実行する」クリック後の画面表示変化・完了トーストの有無は ⚠️未検証
- **出典**: ヤマトB2クラウドで出荷業務を行う.md, 注文と出荷のよくある質問.md

## ヤマトB2クラウド（出荷実績）CSVインポート / DHL（出荷実績）CSVインポート

- **分類**: 連携設定（インポート実行・実行履歴）
- **何か**: ヤマトB2クラウド（またはDHL）からダウンロードした出荷実績CSVをSQへ取り込む操作。出荷実績の登録手段。
- **作成場所**: 出荷管理（/admin/inventory_outbound_orders）右上の「インポート」ボタン > ドロップダウンで「ヤマトB2クラウド（出荷実績）」または「DHL（出荷実績）」を選択 →「新規インポート」→「ファイルを選択する」でCSVを選び「保存する」。CSVインポート（/admin/csv_import）の「出荷 > ヤマトB2クラウド」からも到達可。DHLのインポート画面URL: `/admin/csv_import/csv_import_operation_fulfillment_by_dhls`
- **主な項目**（ヤマトB2クラウドCSVフォーマット。固定カラム順・ヘッダー名一致方式。「テンプレート」リンクからGoogleスプレッドシートの定義書を確認可）:
  | 列 | 項目名 | 必須/任意 |
  |:--|:--|:--|
  | A | 伝票番号 | 必須 |
  | B | お届け先コード | 任意 |
  | C | お届け先名 | 任意 |
  | D | 荷物状況 | 任意 |
  | E | 日付 | 任意 |
  | F | 時刻 | 任意 |
  | G | 出荷日 | 必須 |
  | H | サイズ品目 | 任意 |
  | I | 運賃 | 任意 |
  | J | お客様管理番号 | 必須 |
  - 必須列はA（伝票番号）・G（出荷日）・J（お客様管理番号）の3列のみ。任意列はインポート処理で読み捨てられる
- **作用先**:
  - 出荷実績: インポートにより出荷実績がSQへ反映される
  - インポート一覧画面: 実行履歴が残る。各履歴行をクリックすると詳細（検証ステータス・実行ステータス・成功件数・失敗件数）を確認できる。検証失敗がある場合は失敗件数のリンクから内容を確認できる
- **ステータス**（実行履歴の項目）:
  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 検証ステータス | CSV検証の結果 | ⚠️未検証（遷移値未記録） | 検証失敗があると出荷実績が反映されない |
  | 実行ステータス | インポート実行の結果 | ⚠️未検証（遷移値未記録） | 同上 |
- **制約・注意**:
  - CSVファイル未選択では「保存する」でアップロードできない
  - 「保存する」クリック後の処理結果・成功/エラー時の表示は ⚠️未検証（実行系操作のため実証未実施）
  - DHL用CSVフォーマットの仕様・テンプレートリンクの有無は ⚠️未検証（2026-06-10時点。DHLのフォームはCSVアップロードのみ確認）
- **出典**: ヤマトB2クラウドで出荷業務を行う.md, 注文と出荷のよくある質問.md

## 出荷実績

- **分類**: 伝票（システム反映データ）
- **何か**: 出荷の完了を記録するデータ。登録すると在庫数に作用する。
- **作成場所**: ヤマトB2クラウド／DHLの出荷実績CSVインポート（上記エンティティ参照）
- **主な項目**: ⚠️未検証（対象ファイルにCSV列以外の項目記録なし）
- **作用先**:
  - 在庫数（手持ち）: **出荷実績を登録すると在庫数（手持ち）が減算される**
- **制約・注意**:
  - **出荷実績の取り消し（削除・変更）はできない**。誤って登録した場合は、在庫管理 > 調整伝票から在庫数の調整を行う
  - 誤登録に対する公式推奨の訂正手順（調整伝票による手持ち数の加算が対処策か）は ⚠️未検証
  - インポート後に出荷実績が反映されない場合は、インポート一覧画面で検証ステータス・実行ステータスを確認する
- **出典**: 注文と出荷のよくある質問.md, ヤマトB2クラウドで出荷業務を行う.md

## 出荷作業ステータス（出荷指示のステータス）

- **分類**: ステータス群
- **何か**: 出荷指示の作業状態を表すステータス。ヤマトB2クラウドCSVエクスポートの絞り込み条件として全値が確認されている。
- **作成場所**: —（出荷指示に付随。遷移操作の全容は対象ファイルに記録なし）
- **ステータス**:
  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 出荷待ち | ⚠️未検証（意味の説明は記録なし） | ⚠️未検証 | ⚠️未検証 |
  | 保留中 | ⚠️未検証 | ⚠️未検証 | ⚠️未検証 |
  | 依頼済み | ⚠️未検証 | ⚠️未検証 | ⚠️未検証 |
  | 作業中 | ⚠️未検証 | エクスポートフォームのチェックボックスONで「出荷作業中」へ変更される（チェックボックスのラベル表記は「出荷作業中」、絞り込み選択肢の表記は「作業中」） | ⚠️未検証 |
  | 欠品・要対応 | ⚠️未検証 | ⚠️未検証 | ⚠️未検証 |
  | キャンセル済み | ⚠️未検証 | ⚠️未検証 | ⚠️未検証 |
  | 出荷完了 | ⚠️未検証 | ⚠️未検証 | ⚠️未検証 |
- **作用先**:
  - ヤマトB2クラウドCSVエクスポート: 「出荷作業ステータス」絞り込み条件（指定しない / 上記7値）としてエクスポート対象を限定する
  - サイドバーの未完了バッジ: 「未完了の出荷指示」件数として出荷管理メニューにバッジ表示（実機表示例: 「出荷管理 未完了の出荷指示 1件 1」。0件で非表示、100件超は「100+」）。どのステータスが「未完了」に該当するかは ⚠️未検証
- **制約・注意**: 各ステータスの意味・遷移条件は対象ファイルに記録なし — ⚠️未検証
- **出典**: ヤマトB2クラウドで出荷業務を行う.md, 注文と出荷のよくある質問.md, 注文管理.md

## 保存フィルター（名前を付けて保存 / ビュー）

- **分類**: ルール（ユーザー作成の保存条件）
- **何か**: 注文管理一覧の現在のフィルター条件をビューとして保存する機能・データ。
- **作成場所**: 注文管理一覧（/admin/orders）> 「検索と絞り込みの結果」で展開 > フィルター条件を設定 >「名前を付けて保存」ボタン
- **主な項目**: フィルター条件（テナント / タグ付けされている / タグ付けされていません / チャネル / 注文番号検索）
- **作用先**:
  - 注文管理一覧: よく使う絞り込み条件として再利用できる
- **制約・注意**:
  - 条件が空の場合は保存されない（フィルター条件がない状態ではクリックしても何も起きない。モーダル等は非表示）
  - 下書き一覧・返品一覧には「名前を付けて保存」ボタンは存在しない（注文管理一覧のみ）
- **出典**: 注文管理.md, 05-orders.md

## Recustomer連携

- **分類**: 連携設定
- **何か**: 返品・交換フローを外部SaaS（Recustomer）に委譲するための接続設定。
- **作成場所**: 設定 > 外部連携 > Recustomer（/admin/recustomer_integrations、説明文「Recustomerとの連携を管理することができます」）>「アカウントを接続」リンクボタン → 接続フォーム（/admin/recustomer_integrations/create、ページタイトル「アカウントを接続する」）で「保存する」
- **主な項目**:
  - **ストアID**: 必須（*）。空送信時のバリデーション文言「ストアIDを入力してください」
  - **シークレット**: 必須（*）。空送信時のバリデーション文言「シークレットを入力してください」
  - 項目は2つのみ（他に非表示項目や追加フィールドなし。実機確認 2026-06-07）。認証情報はRecustomerから取得して設定する構成
- **作用先**:
  - 返品フロー: 接続時に返品・交換フロー全体がRecustomerに委譲される可能性 — ⚠️未検証
- **制約・注意**:
  - 未接続時は一覧が空（「アイテムが見つかりませんでした」）
  - Recustomer接続後の返品フローとSQ内返品フローの切り替え有無 — ⚠️未検証
  - Recustomer未接続時のSQ内返品フロー本体の有無 — ⚠️未検証
- **出典**: 05-orders.md, 注文管理.md

## 権限グループ（注文・売上関連の権限）

- **分類**: ルール（アクセス制御）
- **何か**: 設定 > 権限グループで管理する権限の集合。注文・下書き注文・売上等の閲覧/編集可否を制御する。全70種の権限識別子があり、カスタムグループを追加可能。
- **作成場所**: 設定 > 権限グループ（実測例: 「特権管理者」=70権限、「TEST_FAQ_店舗スタッフ」=4権限）
- **主な項目**（注文・会計領域の関連権限。実測抜粋）:
  | 権限識別子 | 日本語ラベル |
  |:--|:--|
  | orders:read | 注文の閲覧権限 |
  | orders:write | 注文の編集権限 |
  | draft_orders:read | 下書き注文の閲覧権限 |
  | draft_orders:write | 下書き注文の編集権限 |
  | sales:read/write | 売上データの閲覧/編集権限 |
  | inventory:read/write | 在庫の閲覧/編集権限 |
  | inventory_allocation_requests:read/write | 在庫リクエストの閲覧/編集権限 |
  | companies:read/write | 取引先企業の閲覧/編集権限 |
  | customers:read/write | 顧客の閲覧/編集権限 |
- **作用先**:
  - 注文管理・下書き・売上実績の各画面: 閲覧/編集可否を制御（`orders:read` のみでログインした場合の画面差異 — 返品/キャンセルボタンの表示/非表示等 — は ⚠️未検証）
- **制約・注意**: UIラベルは「会社」だが権限識別子は `companies`（表示は「取引先企業の閲覧権限」）
- **出典**: 05-orders.md
