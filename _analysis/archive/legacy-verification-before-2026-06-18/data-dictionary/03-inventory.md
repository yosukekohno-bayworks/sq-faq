## 在庫アイテム（SKU×ロケーション別在庫）

- **分類**: システム生成データ
- **何か**: SKU（商品バリエーション）×ロケーション単位で持つ在庫数のレコード。一覧は4区分、SKU詳細は7区分で表示される。
- **作成場所**: 専用の作成画面の記載なし。前提として「調整対象の商品バリエーション（SKU）が在庫管理に登録済みであること」とされる。確認は 在庫管理（`/admin/inventory_items`）、詳細は `/admin/inventory_items/[id]`。
- **主な項目**:
  - 一覧（4区分）: 商品 / 品番（SKU） / 販売不可 / 確定済み / 販売可能（鉛筆アイコン付き） / 手持ち
  - 詳細（7区分）: ロケーション / 販売可能（鉛筆アイコン付き） / 確定済み / 確保済み / 破損 / 検品 / 予備 / 手持ち
  - 検索バー: プレースホルダ「SKUコードで検索する」。「絞り込みを追加」の条件は「**商品コード**」の1種類のみ
  - 一覧の「販売不可」= 詳細の「破損」「検品」「予備」に対応
- **作用先**:
  - ロケーション切替: 一覧ヘッダーのロケーション名ボタン（例:「物流倉庫 ▼」）→「ロケーションを選択する」モーダル（タブ: すべて / 店舗 / 倉庫、列: 名前 / 場所コード、ボタン: キャンセル / 選択する）。h1は常に「在庫管理 :」のまま変化しない
  - 在庫詳細ヘッダーボタン:「調整履歴」→ `/admin/inventory_items/[id]/history`、「商品管理」→ バリエーション詳細、「その他のアクション ▼」→「在庫振替を行う」のみ
  - CSV: 「インポート」→ `/admin/csv_import/csv_import_operation_inventory_logical_available_quantities`、「エクスポート」→ `/admin/csv_export/csv_export_operation_inventory_logical_quantities`
- **制約・注意**:
  - 一覧でチェックボックスの全選択は可能だが、一括操作ボタンは表示されない（「30を選択済み」バーにアクションなし）
  - 行クリックで在庫詳細画面へ遷移
- **出典**: 在庫管理.md、04-inventory.md

## 在庫数量 7区分（販売可能・確定済み・確保済み・破損・検品・予備・手持ち）

- **分類**: ステータス群（在庫の区分）
- **何か**: SKU詳細でロケーション別に表示される在庫数の内訳。手持ちは他6区分の合計。
- **作成場所**: 画面で直接作成するものではなく、各操作（伝票・編集・実績登録）により増減する。
- **主な項目**: 計算式 `手持ち = 販売可能 + 確定済み + 確保済み + 破損 + 検品 + 予備`
- **ステータス**:

  | 表記 | 意味 | 遷移（何で増減するか） | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 販売可能 | 販売できる在庫数 | 移動伝票作成で減 / 移動伝票キャンセルで元に戻る /「販売可能在庫数を編集する」で増（手持ちも同時に増）/ CSVインポート「販売可能在庫」で更新 | 鉛筆アイコンから直接編集可能 |
  | 確定済み | 注文確定済み（注文確定・出荷待ち）の在庫数 | 増減の操作経路の記載なし | — |
  | 確保済み | 取置伝票などで確保された在庫数（FAQでは「在庫依頼で引当て済みの在庫数」） | 移動伝票作成で増（販売可能から同数移動）/ 移動伝票キャンセルで販売可能に戻る。**在庫依頼の「在庫を引当てる」だけでは動かない** | 手持ちは変わらない |
  | 破損 | 破損品（品質問題により販売不可）の在庫数 | ⚠️未検証: UIから能動的に移す専用フローは確認されず。調整伝票の理由・在庫振替・取置伝票・移動伝票のいずれにも区分指定欄なし。システム内部処理（入荷検品フロー・WMS〈ロジザードZERO〉等外部連携）で自動設定の可能性（要ベンダー確認） | 一覧では「販売不可」に合算 |
  | 検品 | 検品中で販売不可の在庫数 | 同上 ⚠️未検証 | 一覧では「販売不可」に合算 |
  | 予備 | 予備として確保された在庫数 | 同上 ⚠️未検証 | 一覧では「販売不可」に合算 |
  | 手持ち | 実在庫合計（上記6区分の合計） | 出荷実績登録で移動元が減 / 入荷実績登録で移動先が増 /「販売可能在庫数を編集する」で販売可能と同時に増 | **マイナスになり得る**（下記） |

- **操作と在庫変化の対応表（実証済み）**:

  | 操作 | 在庫の変化 |
  |:--|:--|
  | 在庫依頼を作成する（確認待ち） | 変化なし |
  | 在庫を引当てる（確保済みへ） | 変化なし（「確保済み」列は動かない） |
  | 移動伝票を作成する | 移動元の販売可能が減り、確保済みが増える（手持ちは変わらない） |
  | 移動伝票をキャンセルする | 販売可能・確保済みが元の数に戻る |
  | 出荷実績を登録する | 移動元の手持ちが減る |
  | 入荷実績を登録する | 移動先の手持ちが増える |

- **制約・注意**:
  - **マイナス在庫**: 手持ち0のロケーションから出荷実績を登録してもSQはブロックしない。実在庫を超えて出荷登録すると在庫数がマイナスになる。修正は調整伝票（FAQでは「在庫管理 > 調整伝票」、在庫管理.mdでは調整伝票・移動伝票での補充、在庫依頼での補充も言及）
  - 一覧4区分の「販売可能」の計算ロジックは「手持ち - 確定済み - 販売不可 と推測」⚠️未検証（全SKU=0のため実機で未検算）
- **出典**: 在庫管理.md、在庫と取り寄せ販売のよくある質問.md、取り寄せ販売の処理手順.md、04-inventory.md

## 調整履歴（在庫の変更履歴）

- **分類**: システム生成データ
- **何か**: SKU×ロケーション単位の在庫変更履歴。「販売可能在庫数を編集する」での変更などが記録される。
- **作成場所**: 自動記録。閲覧は 在庫詳細 >「調整履歴」ボタン → `/admin/inventory_items/[id]/history`。ロケーションを選択して絞り込む。
- **主な項目**（列）: 日時 / アクティビティ（変更の理由・操作名。鉛筆ダイアログで入力した「理由」がここに表示される）/ 販売可能 / 確定済み / 確保済み / 破損 / 検品 / 予備 / 手持ち（各区分は変更後数量＋差分表示。例: 5 (+5)）
- **作用先**:
  - 在庫変更の監査: 調整伝票を経由しない「販売可能在庫数を編集する」の変更もここで追跡できる
- **制約・注意**: データがない場合は「**在庫の変更履歴はありません**」と表示
- **出典**: 在庫管理.md、在庫を調整する.md、VIRTUAL-LOCATION-DEEPDIVE-2026-06-11.md、04-inventory.md

## 調整伝票

- **分類**: 伝票
- **何か**: 棚卸差異・廃棄・紛失・見本などの理由を記録しながら在庫数を手動で増減する伝票。マイナス在庫の修正にも使う。
- **作成場所**: 在庫管理 > 調整伝票（`/admin/inventory_adjustment_orders`）>「調整伝票を作成する」→ `/admin/inventory_adjustment_orders/create`
- **主な項目**:
  - ロケーション（必須。「選択」ボタン → ロケーション選択モーダル）
  - **理由**（必須。ドロップダウン。選択肢は **廃棄 / 見本 / 紛失 / 棚卸差異 / その他** の5種類のみ。破損・検品・予備は存在しない）
  - 商品（必須。「参照」ボタンで商品検索）
  - 増減数（商品追加後に表示。プラスで増加、マイナスで減少。デフォルト0）
  - ※04-inventory.mdの初期記録には「理由 任意」「商品なしでも保存可」とあるが、その後のV7検証で3項目とも必須エラーが出ることを実測確認
- **作用先**:
  - 在庫数:「実行」で明細の差分値が在庫数に反映される
  - マイナス在庫の修正手段（FAQで案内）
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 未実施 | 保存済み・在庫未反映（「○ 未実施」白丸） | 「実行」→ 確認ダイアログ →「実行する」で実施済みへ /「キャンセル」でキャンセルへ | 在庫変化なし。ロケーション・理由・商品・増減数を編集可。変更後「変更を破棄」で元に戻せる |
  | 実施済み | 在庫反映済み（「● 実施済み」黒丸） | 終端（巻き戻し不可） | 在庫数に反映。全フィールド読み取り専用。右サイドバーに実行者・実行日が追加表示 |
  | キャンセル | キャンセルされた伝票（一覧タブに存在） | — | — |

  一覧タブ: すべて / 未実施 / 実施済み / キャンセル
- **制約・注意**:
  - 実行確認ダイアログ「在庫調整を実行する」本文:「在庫調整を実行することで明細に登録されている差分値が在庫数に反映されます。この処理は巻き戻すことができません。」
  - 未保存変更がある状態（「保存されていない変更があります」バナー表示中）は「実行」ボタンがグレーアウトして押せない。先に「保存する」が必要
  - バリデーションエラー文言（原文）: 「ロケーションを選択してください」「理由を選択してください」「商品を1つ以上追加してください」。「保存する」ボタンはグレーアウトせず、押した後にインラインエラーを後出し表示
  - 伝票番号は #IA-XXXX 形式（例: #IA-1000）。詳細に作成者・作成日を表示
- **出典**: 在庫を調整する.md、在庫管理.md、在庫と取り寄せ販売のよくある質問.md、04-inventory.md

## 販売可能在庫数を編集する（鉛筆アイコンによる直接編集）

- **分類**: ルール（在庫の直接変更操作。調整伝票とは別経路）
- **何か**: 在庫詳細の「販売可能」列の鉛筆アイコン（aria-label:「販売可能数を編集」）から、特定ロケーションの販売可能数を直接変更するダイアログ。論理ロケーションへのEC販売用在庫枠の設定にもこの方法を使う。
- **作成場所**: 在庫管理 > SKU行クリック > 在庫詳細（`/admin/inventory_items/[id]`）>「販売可能」列の鉛筆アイコン →「販売可能在庫数を編集する」ダイアログ
- **主な項目**:
  - **理由**（必須。テキスト自由入力。空のまま「保存する」を押すと「理由を入力してください」と表示され保存できない）
  - 数量（数値入力。初期値0、単位「個」）
  - ボタン: キャンセル / 保存する
- **作用先**:
  - 在庫数（実証済み）: 入力した数量だけ「**販売可能」と「手持ち」が同時に増える**（例: 数量5 → 販売可能 +5・手持ち +5。他5区分は0のまま）
  - 調整履歴: 変更が記録され、入力した「理由」がアクティビティ名として表示される
  - 調整伝票: **新しい伝票は作成されない**（調整伝票とは別経路の在庫変更）
  - 取り寄せ販売（EC注文起点）: 論理ロケーションにECで売ってよい数量を設定する事前設定手段
- **制約・注意**: 「数量」のバリデーションは⚠️未検証（理由エラーが先に表示されるため）
- **出典**: 在庫管理.md、在庫を調整する.md、取り寄せ販売（EC注文起点）の流れ.md、VIRTUAL-LOCATION-DEEPDIVE-2026-06-11.md、04-inventory.md

## 在庫振替

- **分類**: ルール（同一商品内バリエーション間の在庫移動操作）
- **何か**: 同一商品内の別バリエーション（例: 同じ商品の別サイズ）へ在庫を振り替えるダイアログ操作。
- **作成場所**: 在庫詳細 >「その他のアクション ▼」>「在庫振替を行う」
- **主な項目**:
  - ロケーション（必須。テキスト入力 +「選択」ボタン → ロケーション選択モーダル。振替元のロケーション）
  - 振替先のバリエーション（必須。ドロップダウン「選択してください」。**同一商品内の他バリエーションのみ表示**。振替元自身は除外、別商品は表示されない）
  - 理由（任意。テキストエリア。プレースホルダ「在庫振替の理由を入力してください」）
  - ボタン: キャンセル / 保存する
- **作用先**:
  - 在庫: 同一商品内の別バリエーションへ在庫を振り替える
- **制約・注意**:
  - 別商品への振替、振替元と同一バリエーションへの振替は不可
  - ⚠️未検証: 振替後の在庫数変化（振替元の「販売可能」が減り、振替先のどの区分に加算されるか）は保存操作を伴うため未実施（要確認・開発元）
  - 区分（破損/検品/予備）を指定するフィールドは存在しない
- **出典**: 在庫管理.md、在庫を調整する.md、04-inventory.md

## 移動伝票

- **分類**: 伝票
- **何か**: 在庫をあるロケーションから別のロケーションへ移動する伝票。手動作成のほか、在庫依頼の確保済み一覧から一括自動生成される。
- **作成場所**:
  - 手動: 在庫管理 > 移動伝票（`/admin/inventory_movement_orders`）>「移動伝票を作成する」→ `/admin/inventory_movement_orders/create`
  - 自動: 在庫依頼 >「確保済み」一覧 >「移動伝票を作成する」→ ダイアログで「移動元」を選択 →「実行する」（移動元未選択時はグレーアウト）。「移動伝票を一括作成しました」と表示
- **主な項目**（手動作成フォーム）:
  - 配送元（必須。「選択」ボタン → ロケーション選択モーダル。プレースホルダ「配送元を選択してください」）
  - 配送先（必須。同上）
  - 商品（「参照」ボタン。**配送元未選択時はフィールドがdisabled**）
- **作用先**:
  - 在庫: 作成した時点で移動元ロケーションの「販売可能」が減り「確保済み」が同数増える（手持ちは変わらない）
  - 出荷指示・入荷指示: （確保済み一覧からの作成で）移動伝票と同時に**出荷指示と入荷指示が自動生成**される。出荷管理・入荷管理に未完了件数バッジが表示される
  - 在庫依頼: 確保済み一覧から作成すると在庫依頼の「確保済み数量」が増加する（引当て操作だけでは増えない）
  - EC注文起点フロー: 移動伝票は**ロケーションごとに分割して生成**される（複数店舗から集まる場合は別伝票）
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | （一覧タブ）すべて / 出荷作業 / 一部受領済み / 受領済み / キャンセル | 一覧の絞り込みタブ | ⚠️未検証: 出荷作業→一部受領済み→受領済み の遷移詳細はデータなしで未確認 | — |
  | 完了 / 入荷完了 | 入荷実績登録後、詳細のステータスバッジに2つ並んで表示 | 出荷・入荷の実績登録に連動して自動的に完了（伝票自体にステータスを進めるボタンはない） | — |
  | キャンセル | 「その他の操作」>「キャンセル」で実行 | 終端（再作成不可） | 下記キャンセル時の連動参照 |

- **キャンセル時の連動（実証済み）**:
  - 確認ダイアログ文言（原文）:「移動伝票をキャンセルすると、この伝票に紐づく入荷予定および出荷指示もすべてキャンセルされます。」
  - 紐づく出荷指示・入荷指示がすべてキャンセルされる。出荷指示は出荷管理のどのタブにも表示されなくなる。入荷指示は入荷管理の「キャンセル」タブに「入荷キャンセル」として残る
  - 確保していた在庫は「販売可能」に戻る
  - **元の在庫依頼は「確保済み」のまま残り、移動伝票の再作成も在庫依頼のクローズもできない行き止まりになる**。やり直す場合は在庫依頼を新規作成
- **制約・注意**:
  - 確保済み一覧から自動生成された移動伝票の詳細には次のメッセージが表示され編集制限される。見出し「外部システムから連携されている移動伝票です」、本文「この移動伝票は外部システムから連携されているため、編集（商品の追加や削除など）は制限されています。連携元システムの管理画面より操作を行ってください。」
  - 配送元=配送先の同一ロケーション選択はUIレベルでブロックされない（⚠️未検証: 保存時のサーバーサイドバリデーション）
- **出典**: 在庫管理.md、取り寄せ販売の処理手順.md、取り寄せ販売（EC注文起点）の流れ.md、在庫と取り寄せ販売のよくある質問.md、在庫依頼.md、04-inventory.md

## 取置伝票

- **分類**: 伝票
- **何か**: 特定用途への在庫確保（取置）を管理する伝票。「確保済み」区分に在庫が移動すると推測（⚠️未検証）。
- **作成場所**: 在庫管理 > 取置伝票（`/admin/inventory_reservation_orders`）>「取置伝票を作成する」→ `/admin/inventory_reservation_orders/create`
- **主な項目**:
  - ロケーション（必須。「選択」ボタン → ロケーション選択モーダル）
  - メモ（任意。テキストエリア。プレースホルダ「例: 店舗での取置用在庫」。調整伝票と異なり「理由」フィールドはない）
  - 商品（テキスト検索 +「参照」ボタン）
- **作用先**:
  - 在庫7区分: 在庫詳細の「確保済み」列の定義に「取置伝票などで確保された在庫数」とある
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 未処理 | 処理前の伝票 | ⚠️未検証: 未処理→処理済みの操作手順はデータなしで未確認 | ⚠️未検証 |
  | 処理済み | 処理完了した伝票 | — | — |

  一覧タブ: すべて / 未処理 / 処理済み
- **制約・注意**: 区分（破損/検品/予備）の指定欄はない
- **出典**: 04-inventory.md、在庫管理.md

## 在庫依頼

- **分類**: 伝票（リクエスト）
- **何か**: 取り寄せ販売の起点。倉庫や他店舗にある在庫を、指定したロケーションへ確保・移動するよう依頼するデータ。
- **作成場所**: 在庫依頼（`/admin/inventory_allocation_requests`）>「在庫依頼を作成する」→ `/admin/inventory_allocation_requests/create`。保存すると「在庫依頼を作成しました」と表示され詳細画面へ遷移
- **主な項目**:
  - 商品バリエーション*（必須。テキスト検索 +「選択」ボタン）
  - 希望数*（必須。数値入力、初期値0）
  - 移動先ロケーション*（必須。在庫の届け先＝引当先。詳細画面では「**引当先ロケーション**」と表記が変わる）
  - **リクエスト先ロケーション**（任意。在庫の引当元。詳細画面では「**依頼先**」と表記が変わる。**指定すると引当てダイアログの「ロケーション」選択肢がこのロケーションのみに絞られる**）
  - 一覧の列: 商品 / SKU / ステータス / チャネル（依頼の発生元。例: `web-admin`）/ 希望数 / 確保済み / 作成日時
  - 詳細の表示: 引当先ロケーション / 確保済み数量（例: 0 / 2 点）/ 残り確保数 / チャネル / 依頼先
- **作用先**:
  - 引当てダイアログ（「在庫を引当てる」）: ロケーション（リクエスト先として指定したロケーションのみ表示）、数量（上限は希望数まで）。「引当てる」で確定するとステータスが「確保済み」になり「在庫を引当てる」ボタンが消える
  - 確保済み一覧（サブメニュー「確保済み」/ `/admin/inventory_allocation_request_confirmations`）: 引当て完了済み在庫の一覧（列: 商品 / SKU / 確保数 / 移動元 / 移動先 / 確保日時）。ここから「移動伝票を作成する」で移動伝票・出荷指示・入荷指示が自動生成され、一覧の該当行が消える
  - サイドバー: 未完了の依頼件数がバッジ表示（0件は非表示、100件超は「100+」）
  - 在庫数: 依頼作成・引当てでは変化なし。移動伝票作成時に初めて変化（販売可能→確保済み）
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 確認待ち | 作成済みで未引当の初期ステータス | 「在庫を引当てる」→ 確保済み /「在庫依頼をクローズする」→ 終了 | 在庫変化なし |
  | 確保済み | 「在庫を引当てる」でロケーションと数量を確定した状態 | 確保済み一覧から「移動伝票を作成する」→ 移動伝票・出荷指示・入荷指示が自動生成（後続の出荷／入荷フローへ） | この時点では在庫変化なし。クローズ不可（「在庫依頼をクローズする」がグレーアウト） |
  | 終了 | クローズ操作の結果 | 終端。⚠️未検証: 終了ステータスの依頼を再開・再利用する方法の有無 | — |

  一覧タブ: 確認待ち / 確保済み / 終了
- **制約・注意**:
  - クローズは**確認待ちのときのみ**実行可能（詳細画面「その他の操作」>「在庫依頼をクローズする」）
  - 確保済みになった後は、移動伝票をキャンセルしても依頼は「確保済み」のまま残り、クローズもできない。やり直しは在庫依頼の新規作成のみ
- **出典**: 在庫依頼.md、取り寄せ販売の処理手順.md、在庫と取り寄せ販売のよくある質問.md

## 出荷指示

- **分類**: システム生成データ
- **何か**: 移動伝票作成時に自動生成される出荷の指示。出荷管理画面で実績登録して完了させる。EC注文起点の場合は注文連携時に生成され引き当て状態を持つ。
- **作成場所**: 自動生成（確保済み一覧の「移動伝票を作成する」実行時、またはEC注文のSQ連携時）。手動作成の記載なし
- **主な項目**: 商品・数量・配送元・配送先ロケーション。出荷実績登録ダイアログの入力: 配送キャリア / 追跡コード（**いずれも任意。空のまま登録できる**）
- **作用先**:
  - 在庫: 「出荷実績を登録する」→「登録する」で移動元ロケーションの**手持ちが減る**。「出荷を完了しました」と表示
  - 左メニュー「出荷管理」: 未完了件数バッジが表示される
  - 移動伝票: 出荷・入荷の実績登録に連動して移動伝票が完了する
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 未完了 出荷待ち | 移動伝票作成直後（「出荷待ち」タブに表示） | 「出荷実績を登録する」→ 出荷完了 | — |
  | 出荷完了 | 実績登録済み | 終端（**巻き戻し不可**） | 移動元の手持ちが減る |
  | 出荷待ち・引き当て待ち | EC注文がSQに連携された直後の状態（⚠️未検証: 正確な画面上の表示文言） | 入荷実績登録後の「次の引き当て処理」実行で「引き当て済み」に変わる（⚠️未検証: 引き当て処理の実行タイミング・手動トリガーの有無） | 引き当て済みで出荷可能になる |
  | （キャンセル） | 移動伝票キャンセルに連動 | — | **出荷管理のどのタブにも表示されなくなる** |

- **制約・注意**:
  - 出荷実績の登録は巻き戻せない
  - マイナス在庫: 手持ち0のロケーションから出荷登録してもブロックされない
  - 出荷指示の分割: 複数のEC注文分の出荷指示を独立した出荷指示に分割可能（⚠️未検証: 分割操作の具体的な手順とボタン名）
  - 論理ロケーションへの出荷指示は在庫リクエストに変換される（⚠️未検証:「出荷業務が不可能」と判定される条件が場所種別・在庫依頼フラグ・別フラグのどれか）
- **出典**: 取り寄せ販売の処理手順.md、取り寄せ販売（EC注文起点）の流れ.md、在庫と取り寄せ販売のよくある質問.md

## 入荷指示

- **分類**: システム生成データ
- **何か**: 移動伝票作成時に出荷指示と同時に自動生成される入荷の指示。入荷管理画面で実績登録して完了させる。
- **作成場所**: 自動生成（確保済み一覧の「移動伝票を作成する」実行時）
- **主な項目**: 種別（移動伝票由来の場合「**移動**」と表示）/ 商品 / 数量 / ロケーション
- **作用先**:
  - 在庫: 「入荷実績を登録する」→「登録する」で移動先ロケーションの**手持ちが増える**。「入荷を完了しました」と表示
  - 移動伝票: 入荷完了で移動伝票のステータスバッジに「完了」「入荷完了」の2つが並んで表示される
  - EC注文起点: 入荷実績登録で倉庫在庫が増え、次の引き当て処理でEC注文の出荷指示が「引き当て済み」になる
  - 左メニュー「入荷管理」: 未完了件数バッジが表示される
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 入荷待ち | 移動伝票作成直後（「入荷待ち」タブに表示） | 「入荷実績を登録する」→ 入荷完了 | — |
  | 入荷完了 | 実績登録済み | 終端（**巻き戻し不可**） | 移動先の手持ちが増える。移動伝票も完了になる |
  | 入荷キャンセル | 移動伝票キャンセルに連動 | — | 入荷管理の「キャンセル」タブに残る |

- **制約・注意**: 入荷実績登録ダイアログに「この処理は巻き戻すことができません。」と明記。出荷（ステップ4）と入荷（ステップ5）は並行処理可能（順序は問わない）
- **出典**: 取り寄せ販売の処理手順.md、取り寄せ販売（EC注文起点）の流れ.md、在庫と取り寄せ販売のよくある質問.md

## 在庫リクエスト（EC注文起点・自動発生）

- **分類**: システム生成データ
- **何か**: EC注文（Shopify）がSQに連携された際、出荷に必要な在庫（倉庫の実在庫）が不足しているSKUに対して自動で発生し、対象の店舗ロケーションに通知されるリクエスト。
- **作成場所**: 自動発生（手動作成の記載なし）。EC注文連携 → 出荷指示が「出荷待ち・引き当て待ち」→ 実在庫のないSKUに在庫リクエストが発生
- **主な項目**: ⚠️未検証（リテールポータルの画面ラベル原文は実機環境の提供待ち）
- **作用先**:
  - リテールポータル（`retail.sqstackstaging.com` など店舗向け専用画面）: 店舗スタッフがログインして在庫リクエストを確認し、在庫の「確保」登録を行う（⚠️未検証: 一覧画面ラベル・確保操作のボタン名）
  - 確保登録でSQ側の在庫リクエストが「確保済み」になる
  - その後「移動伝票を作成する」（リテールポータルからかSQ管理画面の「確保済み」一覧からかは⚠️未検証）→ ロケーションごとに分割生成
- **ステータス**: 確保済み（確保登録後）。それ以外の表記は⚠️未検証
- **制約・注意**:
  - ロケーショングループ内で「在庫依頼を受け付ける」が有効な店舗にだけ在庫リクエストが送られる
  - 論理ロケーションは「出荷業務が不可能」であり、そこへの出荷指示が在庫リクエストに変換される（録画40:01より。⚠️未検証: 判定条件）
- **出典**: 取り寄せ販売（EC注文起点）の流れ.md、VIRTUAL-LOCATION-DEEPDIVE-2026-06-11.md、在庫と取り寄せ販売のよくある質問.md

## ロケーション

- **分類**: マスタ
- **何か**: 倉庫・店舗を表すマスタ。在庫はSKU×ロケーション単位で管理される。
- **作成場所**: 設定 > ロケーション >「ロケーションを作成」（取り寄せ販売（EC注文起点）の流れ.mdの表記）。作成フォームは3セクション11項目
- **主な項目**:
  - 基本情報: 名前* / 表示名 / コード*（例: W0001、R0001、TESTEC01）/ **場所種別***（必須ドロップダウン。選択肢は「倉庫」「店舗」の2種類。場所コードの先頭文字とは独立した属性）
  - 所在地・連絡先: ロケーションの住所を登録する（チェック）/ マップ / 電話番号 / メールアドレス
  - 販売設定: **店舗受け取りを有効にする**（チェック・デフォルトOFF）/ **在庫依頼を受け付ける**（チェック・デフォルトOFF）/ ポイント利用種別*（値引き / 金種）
  - 詳細画面: ロケーションID / 外部ID / コード / 表示名 / 場所種別 / 営業状態（営業中バッジ）/ 所属ロケーショングループ（n個のグループに属しています）/ 公開設定（公開中バッジ）/ 販売設定 / 備考欄 / タグ /「その他の操作」
  - 名前のヘルプ文（原文）:「このロケーションに短い名前を付けて、簡単に識別できるようにします。注文や商品などのエリアでこの名前が表示されます。」
- **作用先**:
  - 在庫管理: 作成後、在庫一覧のロケーション切替・SKU詳細のロケーション別テーブル・在庫依頼のロケーション選択ダイアログに**自動で表示される**
  - **場所種別**: 在庫管理のロケーション選択モーダル・在庫詳細の「すべて / 店舗 / 倉庫」タブの絞り込みに作用（店舗タブ→種別「店舗」のみ、倉庫タブ→種別「倉庫」のみ表示）
  - **在庫依頼を受け付ける**（ON時）: この店舗が在庫依頼の送付先として選択可能になる。ロケーショングループ内でこのフラグが有効な店舗にだけ在庫リクエストが送られる（録画で言及された在庫リクエスト有効フラグの実体）
  - 調整伝票・移動伝票・取置伝票・在庫依頼: 各フォームのロケーション選択肢になる
- **制約・注意**: 実測の登録例: 物流倉庫（W0001）、ユニクロ - 銀座店（R0001）
- **出典**: 取り寄せ販売（EC注文起点）の流れ.md、VIRTUAL-LOCATION-DEEPDIVE-2026-06-11.md、04-inventory.md、在庫と取り寄せ販売のよくある質問.md

## ロケーショングループ

- **分類**: マスタ
- **何か**: 複数のロケーションをまとめるグループ。Shopify連携に指定することで、在庫リクエストの送付対象を制御する。
- **作成場所**: 設定 > ロケーショングループ でグループを作成し、論理ロケーションと「在庫依頼を受け付ける」をオンにした実店舗を追加する
- **主な項目**:
  - グループ名*（プレースホルダ: 関東全店）
  - ロケーション: 含まれるロケーション n ＋「編集する」→ /locations サブページ（h1「グループに含まれるロケーション」、「ロケーションを追加」→ 選択ダイアログ）
  - **「このグループに新しいロケーションを自動的に含める」**（チェック。既存グループではdisabled＝作成後変更不可の可能性 ⚠️未検証・要確認）
  - **デフォルトのロケーション**（例: 物流倉庫。変更ボタンあり）
- **作用先**:
  - Shopify連携: Shopify連携の設定でこのロケーショングループを指定する → グループ内で「在庫依頼を受け付ける」が有効な店舗にだけ在庫リクエストが送られる
  - ロケーション詳細: 「所属ロケーショングループ（n個のグループに属しています）」として表示される
- **制約・注意**: **デフォルトロケーションの行はグループ編集画面でチェック不可＝グループから除外できない**。⚠️未検証: Shopify連携詳細内でのロケーショングループ設定画面のUIラベル・操作手順
- **出典**: VIRTUAL-LOCATION-DEEPDIVE-2026-06-11.md、取り寄せ販売（EC注文起点）の流れ.md

## 論理ロケーション（仮想店舗。例: 店舗在庫EC販売用）

- **分類**: マスタ（ロケーションの運用パターン）
- **何か**: 物理的な場所を持たない、「店舗在庫のうちECで販売してよい数だけ」を管理するための入れ物として作る仮想のロケーション。録画2026-05-20の「ECの架空の店舗」の実体。
- **作成場所**: 専用の機能はなく、設定 > ロケーション で通常のロケーション作成と同じ手順。名前（例: 店舗在庫EC販売用）、コード、場所種別「**店舗**」を入力して保存する
- **主な項目**: 通常のロケーションと同じ（上記「ロケーション」参照）。実証テストデータ: 「TEST_FAQ_店舗在庫EC販売用」（コードTESTEC01・種別=店舗、TEST_FAQ_ロケーショングループに所属、486125-31-Lの販売可能5/手持ち5）
- **作用先**:
  - 在庫管理: 作成後、在庫一覧のロケーション切替・SKU詳細のロケーション別テーブル・在庫依頼のロケーション選択ダイアログに自動で表示される
  - EC販売枠: SKU詳細の「販売可能数を編集」（理由必須・数量）でECに出してよい数量を設定する（例: カラーAに5個、カラーBに3個）
  - Shopify連携: このロケーションをShopifyロケーションと連携させると、Shopify側はここの在庫数を見てECサイトの販売可否を制御する
  - 取り寄せフロー: EC注文が入り倉庫に実在庫が不足している場合、在庫リクエストが店舗へ発行され、店舗から取り寄せるフローが動く
  - 出荷指示変換: 論理ロケーションは「出荷業務が不可能」であり、そこへの出荷指示が在庫リクエストに変換される（⚠️未検証: 判定条件が場所種別・在庫依頼フラグ・別フラグのどれか）
- **制約・注意**: 論理ロケーション自体から物理的な出荷・入荷は行われない。あくまで「ECに出してよい在庫の枠」を管理する目的
- **出典**: 在庫と取り寄せ販売のよくある質問.md、取り寄せ販売（EC注文起点）の流れ.md、VIRTUAL-LOCATION-DEEPDIVE-2026-06-11.md

## CSVインポート「販売可能在庫」

- **分類**: 連携設定（CSV一括処理）
- **何か**: 複数SKU・複数ロケーションの在庫数（販売可能在庫）をCSVでまとめて更新する機能。CSVインポート全体は21カテゴリあり、在庫グループには「販売可能在庫」「予約販売」がある。
- **作成場所**: CSVインポート >「在庫」グループ >「販売可能在庫」> 一覧画面（`/admin/csv_import/csv_import_operation_inventory_logical_available_quantities`）右上の「新規インポート」
- **主な項目**:
  - **反映方法**（ラジオボタン）:
    - 「**絶対値で反映する**」（デフォルト）: CSVに記載した数値をそのまま在庫数として上書きする
    - 「**差分値で反映する**」: 現在の在庫数にCSVの値を加算する
    - インフォボックス原文:「現在の在庫数が10のとき、CSVで20を指定すると、絶対値で反映した場合の在庫数は20になります。差分値で反映した場合の在庫数は30になります。」
  - ファイル: 「ファイルを選択する」ボタン（ドラッグ＆ドロップ可）→「保存する」で実行
  - テンプレート: 一覧画面右上「テンプレート」ボタン → Googleスプレッドシート（`https://docs.google.com/spreadsheets/d/1UJVfvgl5wb_5FYazFowbS1UMtfOhV4nNqFkAn4c1FHI`）。⚠️未検証: テンプレートの列構成（SKU列・ロケーション列・在庫数列の具体的な列名）
- **作用先**:
  - 在庫: 反映方法に応じて在庫数を上書きまたは加算
  - 在庫管理一覧の「インポート」リンクの遷移先がこの画面
- **ステータス**:

  | 表記 | 意味 | 遷移 | 在庫・データへの影響 |
  |:--|:--|:--|:--|
  | 成功 完了 | インポート成功（履歴テーブルの「実行ステータス」） | — | 在庫数に反映 |
  | 失敗 / エラー | 検証失敗等（⚠️未検証: 検証ステータスと実行ステータスの具体的な表示パターン・行単位エラー詳細の表示形式） | 行クリックで詳細ページを開きエラー内容を確認 → CSV修正後に再実行 | 反映されない |

- **制約・注意**: ファイルをアップロードして「保存する」を押した時点でインポートが開始される。それ以前なら画面を離れれば実行されない
