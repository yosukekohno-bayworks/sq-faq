## 1. SQとは何か — 全体像をつかむ

この章では、SQが何をするシステムなのか、どんな課題を解決するのか、そして画面の中に出てくるデータがどうつながっているかを説明します。ここを読めば、以降の章で登場する言葉や操作が「なぜそうなっているのか」わかるようになります。

---

### SQをひとことで言うと

SQは、ネット通販（Shopify）と実店舗のPOSレジ（スマレジ）の**商品・在庫・注文を、裏側でまとめて管理する**システムです。

お客様が商品を見て買う「売り場」ではありません。運営スタッフが在庫の数を確認し、倉庫に発注し、出荷指示を出す**バックヤード（業務管理の裏方）**です。

| 役割 | 担当するもの | 例 |
|:--|:--|:--|
| 売り場（お客様が見る） | 商品を見せる・売る | ShopifyのECサイト、店舗のレジ（スマレジ） |
| **バックヤード（SQ）** | 商品台帳・在庫表・注文の控え・仕入れ記録 | SQの管理画面 |

売り場で何かが売れると、その情報は自動でSQに流れ込みます。スタッフはSQを見て「どこに何個あるか」「何を出荷すべきか」を判断します。

---

### SQが解決する課題

ネット通販と実店舗の両方で販売していると、こんな悩みが出てきます。

- ネットでは「在庫あり」なのに、実際は店舗で売り切れていた
- 店舗にない商品を倉庫から取り寄せたいが、電話とメモで管理していて漏れる
- 注文・在庫・顧客の情報がチャネルごとにバラバラで、全体が見えない

SQはこれらを**1か所に集めて**解決します。中でも、**倉庫と店舗のあいだで在庫を取り寄せる流れを伝票でしっかり管理できる**のがいちばんの特徴です。

---

### Shopify・スマレジ（表）とSQ（裏）の役割分担

SQは単独で動くのではなく、外部サービスと連携して力を発揮します。大まかな役割の分担は次のとおりです。

| 外部サービス | SQとの関係 |
|:--|:--|
| Shopify（ECサイト） | 商品・在庫・注文データをSQと双方向でやり取りする |
| スマレジ（POSレジ） | 在庫を同期する（同期の方向は設定で選択）<!-- TODO: 要確認（スマレジ連携の実際の同期方向・頻度は接続後でないと確認できないため、ここでは仕様の説明にとどめます） --> |
| ロジザードZERO（倉庫管理） | 入出荷の実績データをやり取りする |
| Recustomer（返品・交換） | 返品・交換に関するデータをやり取りする |

注文はShopifyやスマレジから自動でSQに入ってきます。スタッフがShopifyの管理画面とSQの両方を行き来して情報を転記する必要はありません。

---

### SQに登場するデータの大きな地図

SQの中でデータがどうつながっているかを示します。

```
① マスタ設定（テナント・ロケーション・取引先）
        ↓
② 商品を登録
        ↓
③ 在庫（どのSKUが、どのロケーションに、何個あるか）
        ↓
④ 注文（Shopify・スマレジから自動で入る）→ 在庫を引き当て
        ↓
⑤ 出荷 → 倉庫（WMS）へ出荷指示
        ↓
顧客（チャネル経由で自動登録）
```

ここで出てくる言葉を4つだけ押さえておきましょう。

| 言葉 | 意味 | 例 |
|:--|:--|:--|
| **商品とバリエーション** | 「商品」は品物のまとまり。「バリエーション」は色・サイズの組み合わせ1つ1つ | 商品「オーバーサイズスウェットシャツ」の中の「BEIGE / L」が1バリエーション |
| **SKU** | バリエーション1つ1つに付く管理番号。在庫はこの単位で数える | `486125-31-L` |
| **ロケーション** | 在庫の置き場所。倉庫と店舗の2種類 | 物流倉庫 W0001 / 銀座店 R0001 |
| **マスタ** | 最初に登録しておく基本情報のこと | テナント（事業部）・ロケーション・取引先（仕入先）・決済方法 など |

SQの在庫はすべて「**どのSKUが、どのロケーションに、何個あるか**」という形で記録されます。この1文を覚えておくと、どの画面も読み解けます。

### SQでできること一覧

| 機能 | ひとことで言うと |
|:--|:--|
| 商品管理 | 商品台帳。商品とバリエーション（色・サイズ）を登録する |
| 在庫管理 | 在庫表。SKU×ロケーションごとの数を見る・直す |
| 注文管理 | 注文の控え。Shopifyなどから自動で入ってくる |
| 顧客管理 | お客様台帳。チャネル経由で自動登録される（手で作ることはできない） |
| 発注管理 | 仕入先（取引先）への発注書を作る |
| 販売設定 | 値段・予約販売・売りすぎ防止（上限/閾値）のルールを決める |
| 会計 | 売上実績を見る（画面名は「売上実績」） |
| 入荷・出荷管理 / 在庫依頼 | 取り寄せ販売の3画面。倉庫↔店舗間の在庫移動を伝票で管理する |
| CRM（ディスカウント / ポイント / 会員ランク） | クーポン・ポイント付与・会員ランクのルールを決める |
| 販売チャネル | Shopify・スマレジなど外部サービスとの接続設定 |
| 設定 | マスタ登録・権限・翻訳などの初期設定一式 |
| CSVインポート/エクスポート | データの一括取り込み・一括出力（21カテゴリ対応） |
| 分析 / 卸売 | **準備中**（画面はあるが未実装） |

### SQの目玉：取り寄せ販売

SQがもっとも力を発揮するのが「取り寄せ販売」です。たとえば、店頭に欲しいサイズがないお客様のために「倉庫から取り寄せますね」と言ったとき、裏側では次のような流れが動きます。

| ステップ | 操作する画面 | やること |
|:--|:--|:--|
| ① 依頼 | 在庫依頼 | 「倉庫からLサイズを1枚、銀座店へ」という在庫依頼を作る |
| ② 取り置き | 在庫依頼 詳細 | 「在庫を引当てる」で在庫を確保する |
| ③ 伝票発行 | 確保済み 一覧 | 「移動伝票を作成する」を実行。この瞬間に出荷指示と入荷指示が自動で生まれる |
| ④ 倉庫が発送 | 出荷管理 | 「出荷実績を登録する」 |
| ⑤ 店舗に到着 | 入荷管理 | 「入荷実績を登録する」。銀座店の在庫が増える |

詳しい手順は後の章で説明します。
