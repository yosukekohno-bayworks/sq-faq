# SQ 実機CRUD/E2Eテスト計画 2026-06-22

## 目的

前回の全ページUIスキャンでは押下しなかった、保存・削除・実行・キャンセル・ステータス遷移を伴う操作を staging 実機で確認する。確認結果をFAQ/Notion資料へ反映し、未確認・連携待ち・不可逆で残る検証データを明確にする。

## 対象環境

- URL: `https://www.sqstackstaging.com/admin`
- ログイン: 既存の `bay-works.com` ブラウザプロファイル
- 実施日: 2026-06-22

## 操作ルール

| 項目 | 方針 |
|:--|:--|
| テストデータ接頭辞 | `TEST_E2E_20260622_` |
| 既存ユーザー | 触らない。ユーザー作成・編集・削除も実施しない |
| 商品 | 追加する場合はGU製品として作成する。製造元/ブランドは `GU` を使う |
| 既存データ | ユーザー以外は必要に応じて参照・編集・キャンセル可。ただしできるだけテストデータで完結させる |
| 不可逆操作 | 実行前後の状態を記録する。戻せないデータは残存データとしてログに残す |
| 外部連携 | Shopify/OmnibusCore/スマレジ/ロジザード/Recustomer等は接続後挙動まで断定しない |
| メール/外部ファイル | 受信・実ファイル取得できない場合は未確認として残す |

## 優先テスト範囲

### A. マスタ/設定

- テナント: 作成、編集
- ロケーション: 倉庫/店舗作成、住所ON/OFF、店舗受取ON/OFF、在庫依頼ON/OFF、編集
- ロケーショングループ: 作成、ロケーション追加、デフォルトロケーション確認
- ブランド: 作成、編集、削除/アーカイブ可否
- 取引先: 作成、編集、削除/アーカイブ可否
- 決済方法: 作成、編集、`支払い待ちでも注文を出荷する` / `代引き` の保存可否
- 販売員: 作成、編集、削除/アーカイブ可否

### B. 商品/カタログ/店舗受取

- 商品: GU製品として作成、編集、アーカイブ/削除
- バリエーション: 追加、編集、原価登録、在庫追跡ON/OFF、在庫切れ販売ON/OFF
- カタログ: 作成、商品追加、編集、削除/アーカイブ可否
- 店舗受取商品: バリエーション追加、削除/解除可否

### C. 在庫/伝票/入出荷/在庫依頼

- 在庫直接編集: 販売可能数編集、履歴確認
- 調整伝票: 作成、キャンセル、実行、実行済みキャンセル不可確認
- 取置伝票: 作成、処理済み、在庫影響確認
- 移動伝票: 作成、キャンセル、出荷指示/入荷指示生成確認
- 出荷管理: 出荷実績登録、不可逆確認
- 入荷管理: 入荷実績登録、不可逆確認
- 在庫依頼: 作成、引当、確保済み一覧から移動伝票作成

### D. 発注/販売設定/CRM

- 発注伝票: 作成、発注済み化、キャンセル、入荷連動確認
- 販売価格: ルール作成、通常価格/セール価格追加、削除可否
- 予約販売: ルール作成、バリエーション追加、削除可否
- 販売閾値: ルール作成、SKU閾値追加、削除、リロード後反映
- ディスカウント: 作成、対象商品追加、削除/アーカイブ可否
- ポイント: 注文ポイント付与ルール作成、タブ追加導線確認
- 会員ランク: 算出ルール作成、初期ランク編集、削除不可項目確認

### E. CSV/PDF/未実装導線

- CSVインポート: 小さい検証CSVで保存→検証→実行を確認できるカテゴリを選ぶ
- CSVエクスポート: 実行可能カテゴリで処理中→完了→ダウンロードリンク表示を確認
- ヤマトB2条件指定エクスポート: 実行導線とステータス変更チェックの影響を確認
- PDFエクスポート: `/create` 404、実行導線有無を再確認
- TODO/エラー画面: 下書き注文作成、顧客作成、分析/卸売の状態を確認
