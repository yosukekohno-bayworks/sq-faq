# Kickoff Meeting Notes for SQ Help Center

- date: 2026-06-12
- source: `株式会社Stack 様 定例 - 2026_06_12 09_54 JST - Gemini によるメモ.docx`
- purpose: SQヘルプセンター/FAQ構築の進め方を決めるキックオフ

## 決定事項

| 項目 | 内容 | FAQ/資料への反映方針 |
|:--|:--|:--|
| 優先順位 | いきなりFAQを書かず、まず製品理解を優先する | 機能別・データ別の分類、依存関係、実測ステータスを先に整備する |
| フェーズ1 | 製品理解・情報設計 | 管理画面を一通り触り、どこまで確認済みかを可視化する |
| フェーズ2 | 活用術・FAQ構成 | 操作手順だけでなく「どんな時にどの機能を使うか」を書く |
| フェーズ3 | 構築先選定・アウトライン | Notion等への投入先、記事構成、概要を決める |
| 対象読者 | 顧客が自力で課題解決できること、社内オンボーディングにも使えること | 初学者が通読できる完全ガイドと、検索向けFAQの両方を用意する |
| 情報設計 | 誰に何をどう伝えるかを先に整理する | 機能別・作業別・FAQ別・はじめにの分類を維持する |
| 標準機能と個別開発 | 顧客問い合わせは個別開発・連携ロジックに偏る場合がある | 標準機能の説明と、顧客別の個別連携/バックエンド処理は切り分ける |
| 活用事例 | 既存ヘルプは操作方法中心で、機能の使いどころが不足 | 「いつ必要か」「何を先に設定するか」「何に影響するか」を各資料に入れる |
| ロケーション | 登録方法よりも、どんな時に登録が必要かで顧客が迷う | ロケーションの判断基準、依存関係、論理ロケーションの使い方を明記する |
| stagingの注意 | stagingには開発中・未実装の画面が混ざる | 動かない画面は不具合/未実装/外部連携不足を分けて記録する |
| Shopify連携 | staging環境は本番Shopifyと接続できない。開発環境が必要 | SQ Connect本番用/staging用の違いを外部連携資料に明記する |
| 顧客ローデータ | チャットログは難しいため、顧客別要望リストを共有予定 | FAQ化前の追加インプットとして要望リストを待つ |
