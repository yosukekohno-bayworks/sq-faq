# Stack社 会議内容フォローアップ（2026-07-02反映）

## 会議録から確定扱いにした内容

- 在庫区分の `引当済み` は、在庫依頼で確保した状態ではなく、注文済みかつ未出荷の注文に引き当てられた在庫を指す。
- `破損` / `検品` / `予備` は現状使われておらず、将来削除される可能性がある。管理画面にも3区分へ直接振り替える専用UIはない。
- 権限の38リソース/76権限は2026-06-27時点の実機値であり、リソース数は今後増える可能性がある。重要なのは個数ではなく、権限グループとユーザーテナントで画面・データ範囲を制御する点。
- 設定の `アプリ` はSQ API用のアクセストークン/スコープ管理であり、Shopify/スマレジなどの外部コネクター接続画面ではない。
- `お知らせ` は右上ベルに関係するが、現状あまり使われていない。
- `マイル商品` はSQ管理画面側ではなく、Shopifyアプリ側に管理画面がある機能。
- ロケーショングループはロケーションをまとめる汎用グループで、Shopify連携の対象ロケーション指定など複数箇所で使われる。
- ブランドはマスターであり、ブランドコードは現在カタログ/販売閾値などの自動追加条件に使われる。今後条件項目が増える可能性がある。
- 販売員はリテールポータル用。販売員コードが全リクエストに付与され、操作追跡に使われる。販売員はロケーショングループではなく個別ロケーションに紐づく。
- 商品オプションはカラー/サイズなどの軸、オプション値はブルー/Mなどの値、バリエーションはオプション値の掛け合わせで作られる。
- ギフト/ラッピングは、SKUが変わらないなら商品オプション軸にしない方が自然。有料ならラッピング用商品、無料の属性ならメタフィールドや注文属性で扱う考え方。
- `在庫を追跡する` は、注文連携時に在庫を減らすかどうかのフラグ。ONなら注文で在庫が減り、OFFなら注文されても在庫が減らない。
- 販売価格ルールは商品マスターのSKU価格を直接変えない。チャネル別に参照される販売価格ルール。
- 販売価格ルールで税込価格を連携する場合は、税込だけでなく税抜価格も登録する必要がある。セール価格だけでなく通常価格側も対で持たせるのが安全。
- 販売上限ルールの販売チャネルは、販売数を集計する注文ソース/チャネルを指す。
- 翻訳は裏側に翻訳データとして保存され、再翻訳すると前回翻訳を上書きする。個別編集は考え方としてはあるが、管理画面は未実装。
- 採寸ルールの項目上限は、2026-06-27時点の実機では5件。開発元説明では最終的に10件へ変更予定。

## 実機確認と会議説明を併記した内容

- ロケーションの `在庫依頼を受け付ける` は、開発元説明ではリクエスト先に出すためのマスター設定。ただし、過去の実機確認ではOFFのロケーションも手動在庫依頼のリクエスト先に保存できたため、資料では「想定仕様」と「現行実機挙動」を分けて記載。
- 店舗受取は今後商品内に内包される可能性がある。現行UIでは商品管理配下の店舗受取画面でSKUを直接追加し、カタログ選択はない。
- メタフィールド定義の一部対象（在庫移動伝票以下）で作成エラーが出る件は、開発元が「変」と反応し調査依頼したため、仕様ではなく不具合/調査中として扱う。
- Shopify/OmnibusCore/リテールポータルへの販売ルール適用範囲は、販売価格・予約販売はOmnibusCore連携対象、販売閾値は現状リテールポータルのみ。販売上限のOmnibusCore適用は要確認。
