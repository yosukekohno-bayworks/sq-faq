# SQ FAQレビュー: 標準機能と顧客別運用差分

- 作成日: 2026-06-16
- 対象: 既存FAQ/HTML、SQ実機検証ログ、MICHAELKORS/CHAMPION Backlogドキュメントの画面目視結果
- 方針: Backlogは画面で見える内容のみを根拠にする。Backlogドキュメント本文の編集、API取得、内部データ取得は行わない。

## 結論

Claudeが作った既存FAQは、SQ標準機能の説明としては概ね正しい。特に、外部連携なしで実機操作できる範囲のステータス、ボタン挙動、不可逆操作、CSV/CRM/在庫移動の説明は、現行の検証ログと整合している。

ただし、そのまま「MICHAELKORSやCHAMPION向けのFAQ」として使うには不足がある。理由は、顧客別に見るシステム、データの発生元、外部連携、運用判断が大きく違うため。

FAQは次の2層に分けるべき。

| 層 | 内容 | 既存FAQの状態 |
|:--|:--|:--|
| SQ標準機能 | 商品、在庫、伝票、CRM、販売設定、CSV、設定、外部連携フォーム | 概ね整備済み。実機確認済み範囲は信頼できる |
| 顧客別運用 | MICHAELKORS/CHAMPION固有の基幹、WMS、POS、外部モール、バッチ、タグ、ロケーション、例外処理 | 追加が必要。標準FAQに混ぜると誤案内になる |

## MICHAELKORSで追加すべき顧客別論点

MICHAELKORSは、SQ標準の在庫依頼/移動伝票だけでは説明が足りない。Logizard、OmnibusCore、MK基幹、Shopify、リテールポータル、倉庫/店舗/BtoB在庫の運用が絡む。

### システム構成

| システム | 役割 |
|:--|:--|
| Shopify | EC storefront/admin。注文、返金、商品表示、顧客接点 |
| SQ | 商品管理、在庫依頼、移動伝票、販売閾値、顧客統合、OmnibusCore/Logizard連携設定 |
| SQ Retail Portal | 店舗側の在庫リクエスト確認、承認、店舗出荷操作 |
| Logizard | EC倉庫/WMS。出荷依頼、入荷予定、出荷実績、追跡番号 |
| MK基幹 CRM | 顧客データ連携先 |
| MK基幹 SA | 売上実績連携先 |
| OmnibusCore | 外部モール注文、商品、在庫、売上/出荷の中継 |

### 重要データと設定

| データ | 内容 |
|:--|:--|
| 顧客ID | full-priceは `77405 + 0 + 7桁sequence`、outletは `77400 + 2 + 7桁sequence` |
| 顧客除外タグ | `sq:no-sync-crm` |
| 売上除外タグ | `sq:no-sync-sa` |
| 顧客統合タグ | `sq:account-disabled` を統合前にShopify側へ付与 |
| EC/BtoB在庫 | `77910` |
| Full-price在庫 | `77901` |
| Outlet在庫 | `77903` |
| Logizard EC倉庫 | `77940` |
| ギフト/メッセージカード | 非在庫商品。ピッキングリストや備考欄へ印字指示 |

### データ流れ

| 流れ | 頻度/条件 |
|:--|:--|
| 顧客連携 | SQ Proxy ServerからMK基幹CRMへ毎日19:00。前日19:00:00から当日18:59:59の作成/更新が対象 |
| 売上連携 | SQ Proxy ServerからMK基幹SAへ毎日19:00。出荷/返品完了が対象 |
| 倉庫在庫連携 | `77910/77901/77903` はWMS等から取り込み。安全在庫を差し引いた値がSQ/Shopify側の売れる数になる |
| 店舗在庫連携 | Full-price/outlet店舗は30分ごと。ゼロ在庫レコードが出ない問題に注意 |
| Logizard出荷依頼 | SQからLogizardへ約15分間隔の前提。追跡番号がないと出荷完了/Shopify通知が進まない |

### MICHAELKORS FAQに必ず入れるべき問い

| 問い | 回答方針 |
|:--|:--|
| EC倉庫に在庫が足りない注文はどうなるか | BtoB倉庫、店舗倉庫、店舗在庫から取り寄せる。SQの在庫依頼/移動伝票とLogizard入荷後に出荷可能になる |
| 店舗は何を見るのか | SQ本体ではなくSQ Retail Portalを見る。自店舗分の依頼/注文だけを扱う |
| 店舗が承認しない場合はどうするか | HQが毎朝確認し、必要に応じて依頼終了、Shopify注文編集、返金を行う |
| 安全在庫はどこで効くか | 外部在庫取り込み時に差し引かれる。管理画面で直接入れた在庫には即時反映されない |
| ゼロ在庫レコード問題とは何か | 店舗在庫ファイルでゼロSKUが連携されない場合、SQ側に古い在庫が残るリスクがある |
| ZOZO/外部モールは通常ECと同じか | 同じではない。OmnibusCore、梱包番号/ケース番号、Logizard、出荷実績連携の論点がある |

## CHAMPIONで追加すべき顧客別論点

CHAMPIONはAS400を中心に、SQ、Logisteed WMS、ReTELA POS、Shopify、OmnibusCoreが接続される構成。POSポイント/会員ランク、AS400連携、3rd運用、預託在庫/客注在庫補充が重要。

### システム構成

| システム | 役割 |
|:--|:--|
| AS400 | 基幹。商品、在庫、顧客、注文、EC注文などの連携元/連携先 |
| SQ | OMS/商品管理/在庫管理/顧客基盤。各システムのHub |
| WMS(Logisteed) | 出荷指図CSV、出荷実績CSV |
| 店舗POS(ReTELA) | 顧客更新、ポイント照会、ポイント更新、新規顧客登録、店舗注文 |
| Shopify | ECサイト。SQ SDK等で注文/顧客/ポイント情報と連動 |
| OmnibusCore | 3rdモール、預託在庫、客注在庫補充 |

### 重要データと操作

| 領域 | 内容 |
|:--|:--|
| CRM | 会員ランク、ポイント算出、バースデー特典、初回購入特典 |
| POS連携 | ポイント照会、ポイント更新、新規顧客登録、店舗注文CSV |
| 顧客統合 | ShopifyではなくSQから統合する。統合前に `sq:account-disabled` を付与 |
| 出荷指示 | AS400/WMS/SQの状態を見ながら作業。出荷待ち、依頼済み、作業中、欠品・要対応、出荷完了 |
| 3rd運用 | 注文キャンセル、住所変更、物流管理、出荷指示の手順が別管理 |

### CHAMPION FAQに必ず入れるべき問い

| 問い | 回答方針 |
|:--|:--|
| SQだけ見れば運用できるか | できない。AS400、WMS、POS、Shopify、OmnibusCoreのどこが正かを業務別に分ける |
| ポイント/会員ランクはSQ標準だけか | 標準機能に加え、ReTELA POSやShopify自動ディスカウント連携の運用がある |
| 顧客統合はどこで行うか | SQから行う。Shopifyの統合機能は使わない |
| 3rdモールの注文変更はどうするか | 3rd運用マニュアルに従う。通常SQ標準の注文操作として扱わない |
| 古い/不採用手順はどう扱うか | `使用不可` やゴミ箱配下の資料はFAQ本文に混ぜず、履歴または注意として隔離する |

## ステータスとデータ流れの整理方針

FAQで一番混乱しやすいのは、同じ言葉が標準SQと顧客別運用で違う意味を持つこと。

| 用語 | 標準SQ | MICHAELKORS | CHAMPION |
|:--|:--|:--|:--|
| 在庫依頼 | 管理画面で作る取り寄せ依頼 | EC注文確定時に倉庫/店舗へ割当依頼が発生 | 仕様書上の中心ではなく、AS400/WMS/POS/3rd連携が主 |
| 移動伝票 | 出荷指示/入荷指示を自動生成する伝票 | 店舗/BtoB/倉庫からLogizard EC倉庫へ集約する運用で重要 | 預託在庫/客注在庫補充ではAS400/WMS/OmnibusCore連携と絡む |
| 出荷完了 | SQの出荷実績登録後 | Logizard出荷実績と追跡番号が重要。Shopify通知にも影響 | WMS出荷実績CSV、AS400反映、Shopify側扱いに注意 |
| 顧客 | チャネル経由で流入 | MK基幹CRMへ日次連携。除外タグあり | POS/Shopify/SQの統合とポイント参照が重要 |
| 売上実績 | 会計/CSVとして扱う | MK基幹SAへ日次連携。返品も対象 | AS400/POS/Shopify/3rd側の扱いを分ける必要あり |
