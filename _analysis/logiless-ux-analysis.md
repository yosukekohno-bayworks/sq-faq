## (B) 使いやすさの要素

### 1. ロール別完全分離 × 横断ナビゲーション

- **「マーチャント（OMS）」「オペレーター（WMS）」「外部サービス連携」** の3軸でコンテンツを完全分離。ユーザーは自分のロールのドメインに集中できる
- 括弧内に略称（OMS/WMS）を付記することで、ソフトウェア区分が一目でわかる
- グローバルナビはどのページでも全ロールに行き来できるため、ロール間の参照が容易

### 2. 同一トップページレイアウトによる一貫性

- マーチャントもオペレーターも同じ構造（はじめての方 / 機能からさがす / よくある質問 / お知らせ）を持つ
- ユーザーがロールを切り替えても、どこに何があるかを再学習する必要がない

### 3. 「はじめての方」と「機能からさがす」の二分法

- 初回ユーザー向け: **設定ガイド**（初期設定手順）と**日々の運用**（ルーティンフロー）の2本立て
- リピートユーザー向け: **機能カテゴリ**から直接目的ページに飛べる
- 同じページに両方の入口を置くことで、習熟度に応じた使い分けができる

### 4. 機能カテゴリカードのアイコン＋説明文フォーマット

- 各カテゴリカードに「アイコン画像 + カテゴリ名 + 説明1文 + →リンク」の統一フォーマット
- 説明文が機能の目的を動詞ベースで表現（例: 「店舗、倉庫、メールの配信など、組織の設定を行います」）
- ユーザーが目的から機能を逆引きできる

### 5. 記事内の段階的情報開示

**設定ガイド記事（/merchant/getting-started/）** の構成:
- 冒頭にページ全体の目的を1段落で説明
- ページ内目次（「目次」セクション）でジャンプ可能
- H2/H3の階層見出しで「まずはじめに → 基本的な設定 → 便利な機能 → 高度な自動化」と難易度順に展開
- 各機能へのカードリンクで詳細記事に遷移

### 6. ヒント・注意書きの視覚的区別

- 「ヒント」バナー: アイコン付きの別枠で補足情報を提示（例: 「CSVによる一括登録も可能です」）
- 「ロジごま」（マスコット）によるコラム形式のワンポイント情報
- 「マクロの設定後は必ず動作確認を行なってください」のような警告も別枠で強調

### 7. フィードバックUI（「この記事は役に立ちましたか？」）

- 全記事末尾に「解決した」「解決しなかった」の2択ボタン
- 機能カテゴリ一覧ページにも同様に設置
- シンプルな2択で回答コストを最小化

### 8. 関連記事リンク（記事末尾）

FAQページ例（/faq/26675/）では記事末尾に「関連記事」セクションを設け、機能記事・他FAQへのリンクを列挙:
- 受注伝票のマクロ（機能記事）
- 受注確定時の処理（機能記事）
- 出荷伝票のマクロ（機能記事）
- 自動処理（カテゴリ記事）
- 受注伝票のマクロが意図したとおり適用されないのはなぜですか？（FAQ記事）

機能記事とFAQが相互に参照し合う構造。

### 9. 記事ページ末尾の「よくある質問」ブロック

機能カテゴリ記事（/merchant/order-management/sales-orders/create-sales-order/ 等）では、記事本文の下に関連FAQ 10件 + 「もっと見る」リンクを配置。ユーザーが「やり方」を読んだ直後に「よくある疑問」に自然に誘導される。

### 10. 4階層パンくず + サイドバーツリーナビゲーション

- パンくず: ホーム > マーチャント > 受注処理 > 受注伝票 > [現在記事] の4〜5段階
- サイドバー: 現在地を開いた状態のカテゴリツリー（全記事名リスト）を常時表示
- 記事を読みながら同カテゴリの隣接記事に移動できる

### 11. 「運用アイデア集」でノウハウを機能説明と分離

- 「特定商品の購入時に、ノベルティやチラシを自動で同梱する」等、ユースケース指向のタイトル
- 各アイデア記事の構成: やりたいこと → 具体例 → 運用の流れ → 利用する機能 → 設定内容 → 実際の動作例
- 機能説明（What）とユースケース（How/Why）を明確に分離することで、上級ユーザーが「活用できるか」を確認しやすい

### 12. 運用フェーズ別のアイコンナビゲーション

運用アイデア集トップのアイコンナビ（販売施策→仕入→保管→受注処理→出荷指示→出荷作業→集荷→データ活用）は業務フロー順に並んでいる。ユーザーが「自分の業務フェーズ」から検索できる。

### 13. 「もっと見る」による段階的開示

トップページの「よくある質問」「お知らせ」は各6〜5件に限定し「もっと見る」で全件ページへ。情報過多を防ぎつつ、必要なユーザーには完全なリストを提供。

### 14. 検索のユビキタス配置

- グローバルナビバー（常時表示）
- 各カテゴリトップのメイン検索ボックス
- 記事ページのサイドバー検索
- 3箇所に検索があることで、どこからでも全文検索できる

## (C) SQのFAQ多軸設計への応用提案

### SQの現軸案

```
はじめに / ロール別 / 作業手順（動作別）/ 機能説明書（機能別）/ よくある質問
```

### ロジレスの仕組みをどう取り入れるか

#### 応用1: ロール別分離を軸に据える（ロジレスの「マーチャント/オペレーター」分離）

**ロジレスの仕組み**: グローバルナビでロールを完全分離し、各ロールが独自のトップページを持つ。ロール間の参照はグローバルナビで横断。

**SQへの応用**:
- SQのユーザーロール（管理者・担当者・閲覧者など）を特定し、ロールごとにFAQの入口ページを作る
- 「あなたは何をする人ですか？」という問いかけで最初に分岐させる
- ただしSQのロール体系を要確認 <!-- TODO: 要確認 -->

#### 応用2: 「はじめての方」= 初回ステップガイド として設定ガイドを独立させる（ロジレスの「設定ガイド」）

**ロジレスの仕組み**: 設定ガイドは「まずはじめに → データモデル理解 → 基本設定 → 便利な機能 → 高度な自動化」と学習曲線順に展開。記事内目次でジャンプ可能。

**SQへの応用**:
- 「はじめに」セクションを「初回セットアップ手順書」として構成し、ステップ番号付きで展開
- 冒頭にSQのデータモデル（主要概念の関係図）を置く
- 初期設定完了後に「次は○○をやってみよう」という誘導リンクを置く

#### 応用3: 機能別カテゴリカードに「目的の説明文」を必ず付ける（ロジレスの「機能からさがす」カード）

**ロジレスの仕組み**: カードには「この機能は何のためにあるか」を1文で記載。「店舗、倉庫、メールの配信など、組織の設定を行います」のようにユーザーの目的語で書く。

**SQへの応用**:
- `faq/01-by-feature/` の各機能記事のイントロに必ず「この機能でできること」を1文で記載
- 目次代わりとなるカテゴリ一覧ページでも各機能カードに説明文を追加

#### 応用4: 「よくある質問」は機能記事の末尾にも埋め込む（ロジレスの記事末FAQブロック）

**ロジレスの仕組み**: 機能説明記事の末尾に関連FAQを10件 + もっと見るリンク。「やり方を読んだ直後に疑問が出る」ユーザー心理に対応。

**SQへの応用**:
- `faq/01-by-feature/` の各機能記事末尾に、その機能に関連するFAQを3〜5件インラインで記載
- Notionデータベースでタグ（機能名）で紐付けることで管理を自動化できる

#### 応用5: 「運用アイデア集」相当の「活用Tips集」軸を追加する（ロジレスの「運用アイデア集」）

**ロジレスの仕組み**: 機能説明（What）とは別に「○○をしたいときはこう使う」形式のユースケース集。業務フロー順（販売→仕入→受注→出荷）のナビゲーション。

**SQへの応用**:
- `faq/04-tips/`（仮）として「SQを使いこなすアイデア集」軸を追加
- タイトルは「○○したいときに使える機能」の形式（ユーザーの目的ファースト）
- SQの業務フロー（発注→受注→出荷→アフター等）順に並べる
- 各Tips記事の構成: やりたいこと → 利用する機能 → 設定手順 → 動作確認方法

#### 応用6: フィードバックUIを全記事に設置する（ロジレスの「この記事は役に立ちましたか？」）

**ロジレスの仕組み**: 全記事末尾の「解決した」「解決しなかった」2択ボタン。回答コストが極めて低い。

**SQへの応用**:
- Notionにインポート後、もしフォームリンクを埋め込める場合は記事末に「役に立ちましたか？ [Google Form リンク]」を設置
- 未解決ユーザーの声を収集してFAQを継続改善する仕組みを作る

#### 応用7: 記事の階層に「なぜこの機能を使うか」の先行説明を必ず置く

**ロジレスの仕組み**: 設定ガイドの「まずはじめに」、運用アイデア記事の「具体的な利用シーン → やりたいこと → 具体例」。機能の目的・背景を先に示してから手順に入る。

**SQへの応用**:
- `faq/02-by-task/` の各タスク記事は「この作業をする状況（いつ必要か）」を冒頭1〜2文に置く
- 手順はステップ番号付きで記載し、各ステップにUIラベル原文（「〇〇ボタン」）を明記

## 要約

### ロジレスの強み TOP5

1. **ロール別完全分離 × 統一レイアウト**: マーチャント/オペレーターでコンテンツを分離しつつ、各ロールトップが同構造なので切り替えコストが低い
2. **初回ユーザー/リピーター二分法**: 「はじめての方（設定ガイド・日々の運用）」と「機能からさがす（カテゴリ）」を同一ページに共存させ、習熟度別の導線を一画面で提供
3. **機能説明 × ユースケース集の分離**: 機能記事（What）と運用アイデア集（How/Why）を軸として分けることで、「使い方を知りたい人」と「活用アイデアを探す人」の双方を満たす
4. **記事内の4要素構成**: 目次→本文→ヒント/警告バナー→関連FAQ埋め込み→フィードバックUI の標準テンプレートが全記事で一貫している
5. **パンくず + サイドバーの二重ナビゲーション**: 現在地が常に分かり、同カテゴリの隣接記事へジャンプできる

### SQに取り入れるべき TOP5

1. **「設定ガイド」= 初回ステップ記事の整備**: SQの初期設定を「まずはじめに → 基本設定 → 応用機能」の難易度順1記事にまとめ、最初の壁を下げる
2. **機能カテゴリカードに目的説明文を付ける**: 「○○のために使う機能です」の1文を各カードに必ず添える。何のためのページかが一目で分かる
3. **機能記事末尾へのFAQ埋め込み**: `01-by-feature` の各記事末に関連FAQ 3〜5件を直接記載。ユーザーの疑問を先回りして解消する
4. **「活用Tips集」の軸追加**: `04-tips/` として「○○したいときの使い方」形式のユースケース集を作る。機能説明と使いこなし方を分けることでリピーターの価値を高める
5. **全記事へのフィードバックUI設置**: 記事末に「役に立ちましたか？」の2択リンクを設置してFAQの継続改善データを収集する
