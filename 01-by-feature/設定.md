# 設定

> 対象画面: 設定 / `/admin/settings`　|　最終確認: 2026-07-02

## この機能でできること

- 組織全体のマスタデータ（ロケーション・ブランド・決済方法など）を管理する
- 管理メンバーの追加・権限グループの設定をする
- 外部サービス（ロジザード・Recustomer など）との連携を設定する
- メタフィールド定義などのカスタムデータを管理する

---

## 設定トップページ

設定トップページには「組織ID」セクションがあり、`{uuid}_Organization` 形式の組織IDが読み取り専用inputで表示されます。右隣のコピーボタン（クリップボードアイコン）を押すと、同じIDがクリップボードに入ります（2026-06-28実機再確認）。

## 管理メンバー（/admin/settings/users）

管理画面にアクセスできるメンバーを管理します。

### 一覧画面の主な操作

| 操作 | 説明 |
|:--|:--|
| 追加する | メンバー追加フォームへ遷移 |
| 権限グループ一覧 | 権限グループ管理画面へ遷移 |
| インポート | 管理ユーザーCSVインポート（`/admin/csv_import/csv_import_operation_users`）へ遷移 |

一覧にはキーワード検索欄（「キーワードで検索する」）があります。2026-06-28の実機再確認では、キーワード検索は姓・名・メールアドレス断片で絞り込めます（例: `河野` / `陽介` / `kohno` は各1件）。権限グループ名（例: `特権管理者`）や複数語（例: `河野 kohno`）ではヒットしません。

「絞り込みを追加」から追加できる条件は「メールアドレス」のみです。これはキーワード検索とは別の専用フィルタで、断片ではなくメールアドレス形式の入力が必要です。

### メンバー追加フォーム（/admin/settings/users/create）

フォームタイトル: **「ユーザーを追加する」**

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| 姓 * | 必須 | 姓・名は同じ行に横並びで表示 |
| 名 * | 必須 | — |
| メールアドレス * | 必須 | — |
| 権限グループ | 任意 | ラジオボタンで1グループのみ選択可。作成済みのグループが一覧表示される。未選択でも保存でき、一覧・詳細では `未設定`、適用権限は `0件` になる |

- パスワード入力欄はありません。メールアドレスによる認証でログインします。
- 権限グループは1人に1グループのみ割り当て可能です（複数グループへの同時割り当て不可）。
- 空保存時は `姓を入力してください` / `名を入力してください` / `メールアドレスを入力してください` が表示されます。
- 2026-06-28追加確認では、検証用ユーザーを権限グループ未選択で保存でき、保存後に詳細画面へ遷移して `ユーザーを追加しました` が表示されました。

### メンバー詳細画面（保存後）

メンバー詳細は以下の3セクションで構成されます。

| セクション | 内容 |
|:--|:--|
| 基本情報 | 姓・名・メールアドレスの編集 |
| ユーザーテナント | アクセスを許可するテナントのチェックボックス選択 |
| 権限 | 権限グループをコンボボックスで変更。下部に「適用されている権限一覧」（件数表示付き）アコーディオンあり |

詳細画面下部には「**ユーザーを組織から除外**」セクションがあり、「**組織から除外する**」ボタンから退職・アクセス削除時の除外操作を行います。2026-06-27時点では「その他の操作」メニューではなく、独立したセクションとして表示されます。2026-06-28追加確認では、クリック後に確認ダイアログ `ユーザーを組織から除外しますか？` が表示され、確定ボタンは `除外する` でした。除外後は一覧から消え、同じメールアドレスで再追加できました。

## 権限グループ（/admin/settings/permission_groups）

管理メンバーに割り当てる権限グループを管理します。

### アクセス方法

管理メンバー一覧（`/admin/settings/users`）の「**権限グループ一覧**」ボタンから遷移します。

### 権限グループの種類

| グループ名 | 特徴 |
|:--|:--|
| 特権管理者 | グループ名のみ変更可能。権限内容（76件）は変更不可・削除不可の半固定グループ |
| カスタムグループ | 「作成する」から新規作成。76権限から任意に選択できる |

### 権限の体系

権限は、リソースごとに **閲覧（:read）/ 編集（:write）** を設定します。2026-06-27時点の実機では **38リソース × 2 = 76種**です。開発元説明では、リソース数は今後増える可能性があり、個数自体よりも「どの機能を見せ、編集させるか」を制御する仕組みが重要です。

代表的なリソース: お知らせ / アプリ / ブランド / 顧客 / ディスカウント / 在庫 / 発注伝票 / ロケーション / 注文 / 商品 / 店舗スタッフ / テナント / 翻訳 / ユーザー など

すべてのリソース: お知らせ(announcements) / アプリ(apps) / ブランド(brands) / カタログ(catalogs) / 取引先企業(companies) / 顧客マイル(customer_miles) / 顧客ポイント(customer_points) / 顧客会員ランクルール(customer_rank_rules) / 顧客会員ランク(customer_ranks) / 顧客(customers) / ディスカウント(discounts) / 下書き注文(draft_orders) / 在庫(inventory) / 在庫リクエスト(inventory_allocation_requests) / 発注返品伝票(inventory_purchase_order_returns) / 発注伝票(inventory_purchase_orders) / 店舗受取ロケーションルール(local_pickup_location_rules) / ロケーショングループ(location_groups) / ロケーション(locations) / マイル商品(mile_items) / ネクストエンジン連携(next_engine_integrations) / OmnibusCore連携(omnibus_core_integrations) / 注文(orders) / ポイントキャンペーン(point_campaign_order_rules) / ポイントルール(point_rules) / 販売価格ルール(product_price_rules) / 商品(products) / 店舗ポータル連携(retail_portal_integrations) / 店舗スタッフ(retail_staff_members) / 売上データ(sales) / Shopify連携(shopify_integrations) / スマレジ連携(smaregi_integrations) / テナント(tenants) / 翻訳ルール(translation_rules) / 翻訳(translations) / ユーザー権限(user_permissions) / ユーザーテナント(user_tenants) / ユーザー(users)

### カスタム権限グループ作成フォーム（/admin/settings/permission_groups/create）

フォームタイトル: **「権限グループを作成する」**

![「権限グループを作成する」フォーム（76権限のチェックボックス）](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/bfg-F-01-permission-group-create-form.png)

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| グループ名 * | 必須 | テキストボックス |
| 権限（チェックボックス群） | 必須 | 76権限から個別に選択。デフォルトは全オフだが、最低1つ選択しないと保存できない |

空保存時は `権限グループ名を入力してください` と `権限を選択してください` が表示されます。

作成済みの未割当カスタム権限グループは、権限グループ一覧で行を選択して「権限グループを削除」から削除できます。選択時は一覧上部に「1を選択済み」と表示されます。確認ダイアログは「権限グループを削除しますか？」で、本文は「選択された1件の権限グループを削除します。この処理は巻き戻すことができません。」です。削除後は対象行が一覧から消え、トースト「権限グループを削除しました」が表示されます。2026-06-27再確認では、特権管理者と管理メンバーに割り当て済みのカスタムグループはチェックボックスがdisabledで選択できませんでした。

## ロケーション（/admin/settings/locations）

在庫管理の拠点（店舗・倉庫）を登録・管理します。

### ロケーション作成フォーム（/admin/settings/locations/create）

フォームは3セクション11項目で構成されます。

![「ロケーションを作成」フォーム](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/14-location-create.png)

**基本情報セクション**

| 項目（UIの原文） | 必須 | 選択肢・補足 |
|:--|:--|:--|
| 名前 * | 必須 | テキストボックス。注文や商品などの画面で表示される名前 |
| 表示名 | 任意 | テキストボックス |
| コード * | 必須 | テキストボックス |
| 場所種別 * | 必須 | セレクトボックス（**倉庫 / 店舗** の2種のみ）。2026-06-28再確認では内部値は `WAREHOUSE` / `RETAIL`、初期選択は `倉庫` |

**所在地・連絡先セクション**

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| ロケーションの住所を登録する | 任意 | チェックボックス。オンにすると住所欄（国/地域・郵便番号・都道府県・市区町村・住所・建物名）が展開される |
| マップ | 任意 | URLを入力 |
| 電話番号 | 任意 | — |
| メールアドレス | 任意 | — |

**販売設定セクション（右サイドバー）**

| 項目（UIの原文） | 必須 | 選択肢・補足 |
|:--|:--|:--|
| 店舗受け取りを有効にする | 任意 | チェックボックス（デフォルト: オフ）。店舗受取用の設定。2026-06-28再確認では、店舗受取ルール作成フォームの候補は場所種別=店舗で絞られ、このチェックがOFFの店舗も候補に表示される。ストアフロントでの受取可否や連携後の実効差は接続環境で要確認 |
| 在庫依頼を受け付ける | 任意 | チェックボックス（デフォルト: オフ）。EC注文起点などの自動在庫リクエストで、このロケーションを送付対象にするための設定とされる。手動の在庫依頼作成画面では、OFFでもリクエスト先に選択・保存できる（2026-06-21実機確認） |
| ポイント利用種別 * | 必須 | セレクトボックス（**値引き（デフォルト）/ 金種** の2種のみ）。2026-06-28確認では `金種` を選択して保存でき、詳細画面でも `金種` が残る |

### ロケーション詳細画面

保存後の詳細画面には以下のセクションが表示されます。

| セクション | 主な表示項目 |
|:--|:--|
| 基本情報 | ロケーションID / 外部ID / コード / 表示名 / 場所種別 / 営業状態（営業中バッジ） |
| 所在地・連絡先 | 登録した住所・マップ・電話番号・メールアドレス |
| 所属ロケーショングループ | このロケーションが属するグループの件数（n個のグループに属しています） |
| 公開設定 | 公開状態（公開中バッジ） |
| 販売設定 | 店舗受取（有効・無効）/ 在庫依頼（有効・無効）/ ポイント利用種別 |
| 備考欄 | フリーテキスト |
| タグ | タグの追加・管理 |
| アーカイブ操作 | 詳細画面の `アーカイブする` からアーカイブ済みにする。アーカイブ後は「このロケーションはアーカイブされています。」と表示され、`アーカイブを解除する` で復帰可能 |

2026-06-27再確認では、ロケーションをアーカイブしても一覧の `すべて` タブには行が残り、公開列が `非公開`、アーカイブ列が `アーカイブ済み` になりました。解除後はアーカイブ列だけ空に戻り、公開設定は `非公開` のままでした。また、店舗受取ルール・ロケーショングループ・販売員のロケーション選択モーダルではアーカイブ済みロケーションも候補に表示され、選択可能でした。

## 販売員（/admin/settings/retail_staff_members）

リテールポータルで、どの販売員が顧客情報を閲覧したか、どの販売員が客注注文を作成したかを記録するための販売員を管理します。

### 販売員作成フォーム（/admin/settings/retail_staff_members/create）

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| 姓 * | 必須 | 姓・名は同じ行に横並びで表示 |
| 名 * | 必須 | — |
| コード * | 必須 | — |
| ロケーション | 必須 | 「検索する」ボタンをクリックするとロケーション選択モーダルが開く。作成フォームでは1件のみ選択可 |

- ロケーション選択モーダルには「すべて / 店舗 / 倉庫」の3タブがあります。
- 空保存時は `名前を入力してください`, `苗字を入力してください`, `コードを入力してください`, `ロケーションを選択してください` が表示されます。
- 作成フォームでは、1件にチェックを入れると他の行はdisabledになり、1件のみ選択する仕様です。
- 作成後の販売員詳細画面では、「店舗を追加する」から複数店舗へ紐づけできます。

## 翻訳（/admin/settings/translation）

### 翻訳ルール作成フォーム（/admin/settings/translation/translation_rules/create）

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| 名前 * | 必須 | テキストボックス |
| 言語 * | 必須 | コンボボックス（上記10言語から選択） |
| 翻訳データを自動で作成する | — | チェックボックス。オン/オフどちらでも保存でき、保存後の詳細画面でも状態が保持される。2026-06-28再確認では翻訳トップ・一覧・作成・詳細に手動実行/再実行/翻訳/生成ボタンはなく、実際の生成タイミング・反映結果は生成済みデータまたは連携環境待ち |

## メタフィールド定義（/admin/settings/metafield_definitions）

各オブジェクトにカスタムフィールドを追加するための定義を管理します。

### メタフィールド定義の対象とUI反映（16種）

組織 / ロケーション / 会社 / 仕入れ先ベンダー / 商品 / バリエーション / 顧客 / 注文 / 下書き注文 / ディスカウント / 在庫移動伝票 / 在庫調整伝票 / 在庫取置伝票 / 発注伝票 / 入荷指示 / 出荷指示

2026-06-28〜2026-07-01実機では、対象一覧には16種が表示されますが、同条件で定義作成できたのは9種でした。

| 対象 | 定義作成 | 代表詳細画面への表示 |
|:--|:--|:--|
| 組織 | 可 | 専用詳細画面なし。設定トップには検証用定義名は出ない |
| ロケーション | 可 | ロケーション詳細に表示 |
| 会社 | 可 | 会社詳細に表示 |
| 仕入れ先ベンダー | 失敗 | `metafieldDefinitionCreate` がサーバーエラー |
| 商品 | 可 | 商品詳細に表示 |
| バリエーション | 可 | SKU/バリエーション詳細に表示 |
| 顧客 | 可 | 顧客詳細に表示 |
| 注文 | 可 | 注文詳細に表示 |
| 下書き注文 | 可 | 定義作成・一覧表示・削除は可。定義一覧では `0個の下書き注文`。現行UIでは下書き注文一覧が空、作成ボタンは遷移せず、直URL `/admin/draft_orders/create` は予期せぬエラーのため値入力画面は確認できない（2026-07-01再確認） |
| ディスカウント | 可 | ディスカウント詳細に表示 |
| 在庫移動伝票 / 在庫調整伝票 / 在庫取置伝票 / 発注伝票 / 入荷指示 / 出荷指示 | 失敗 | `metafieldDefinitionCreate` がサーバーエラー |

### メタフィールド定義 作成フォーム

作成は `/admin/settings/metafield_definitions` の対象オブジェクト一覧で対象を選び、`定義を追加する` から行います。オブジェクト指定なしの `/admin/settings/metafield_definitions/create` は存在しないページです（2026-06-24実機確認）。

対象別一覧は `/admin/settings/metafield_definitions?ownerType=<対象オブジェクト>` です。2026-06-28実機では `/admin/settings/metafield_definitions/<対象オブジェクト>` は存在しないページです。

| 項目（UIの原文） | 必須 | 制約・補足 |
|:--|:--|:--|
| 名前 * | 必須 | 最大50文字 |
| 説明 | 任意 | 最大100文字 |
| ネームスペース * | 必須 | 最大50文字。2026-06-27実機では3文字の `abc` でも保存可 |
| キー * | 必須 | 最大50文字。2026-06-27実機では3文字の `def` でも保存可 |
| メタフィールドのタイプ * | 必須 | コンボボックス（下記12種から選択）。未選択で保存すると `タイプを選択してください` |

**タイプの選択肢（12種）:**
単一行のテキスト / 単一行のテキスト(リスト) / 複数行のテキスト / 日付と時刻 / 日付 / trueまたはfalse / バーコード(NW7) / 整数 / 小数 / JSON / ID / リッチテキスト

- 識別子は「ネームスペース.キー」形式で管理されます（例: `custom.note`）。
- 商品・バリエーション・ロケーション・会社・顧客・注文・ディスカウントでは、保存後に代表詳細画面へ「メタフィールド」セクションと定義名、`未設定` が表示されます。バリエーションはSKU/バリエーション詳細画面で表示確認済みです（2026-06-28）。下書き注文は定義作成自体は可能ですが、値入力先となる下書き注文詳細を現行UIで開けないため、詳細画面表示は案内しません。
- 商品詳細で値を保存すると、表示は `未設定` から入力値に変わり、定義一覧の「使用箇所」は `0個の商品` から `1個の商品` に増えます。
- 作成済み定義は一覧で行選択後に `定義を削除` できます。確認ダイアログには「関連するメタフィールドの値は削除されません」と表示されます。削除後は定義一覧から消え、商品詳細側のメタフィールド欄にも表示されません。

## アプリ（/admin/settings/apps）

SQ Admin APIにアクセスするためのAPIキーを管理します。

- 画面の説明文（UIの原文）: 「APIアクセスを管理する / アプリを作成して、SQのアドミンAPIにアクセスしましょう。」
- この「アプリ」はSQに対するアクセストークンを権限付きで発行する開発者向け機能です。外部アプリをインストール/接続する画面ではありません。
- 「アプリを作成する」リンクから作成フォームへ遷移します。

### アプリ作成フォーム（/admin/settings/apps/create）

| 項目（UIの原文） | 必須 | 補足 |
|:--|:--|:--|
| アプリ名 * | 必須 | テキストボックス（プレースホルダー: 「モバイルアプリ」） |
| 権限（チェックボックス群） | — | 76権限から付与するスコープを個別に選択。各権限には説明文が付いている（例: 「商品情報を閲覧することができます。」） |

空保存時は `アプリ名を入力してください` と `権限が選択されていません` が表示されます。権限未選択の補足文は「アプリが持つ権限は、必ず1つ以上選択されている必要があります。」です（2026-06-27実機確認）。

### アプリ詳細画面（保存後）

アプリを作成すると詳細画面に遷移し、以下の情報とセクションが表示されます。

#### Admin API

作成と同時にアクセストークンとシークレットが自動発行されます。アクセストークン/シークレットは秘密情報のため、FAQやログに出力しないでください。2026-07-01の再確認では、作成済みアプリの詳細画面・上部「その他の操作」・一覧カードとも、アプリ削除・トークン失効・アクセストークン/シークレット再発行の導線は確認できませんでした。誤作成時の削除・失効方法は開発元確認が必要です。

| 項目 | 内容 |
|:--|:--|
| アクセストークン | Admin APIへの接続に使用するトークン（自動発行） |
| シークレット | Webhookのリクエスト検証等に使用するシークレット（自動発行） |

#### 検証方法

「**Playgroundを開く**」ボタンから GraphQL Playground を開いてAdmin APIを試すことができます。2026-06-28再確認では、stagingのアプリ詳細からでもリンク先は `https://sq.stackservice.com/api/admin/v1/playground` で、ページ表示は可能でした。URLクエリにトークン/アプリIDらしき値は付与されません。秘密値露出を避けるためクエリ実行・スキーマ取得は行っていません。

#### リクエストログ

「**リクエストログを見る**」から `/admin/settings/apps/{id}/admin_api` へ遷移します。2026-07-01再確認では、Admin APIログは `Admin API - 対象アプリ名`、Storefront APIログの直URL `/admin/settings/apps/{id}/storefront_api` は `Storefront API - 対象アプリ名` と表示され、どちらも本文は `TODO` でした。ログ行・保持期間は未確認です。

#### Storefront API

「**トークンを発行する**」ボタンからStorefront API用トークンの発行モーダルを開けます。2026-07-01実機確認では、項目は `カタログ` と `ロケーショングループ` です。空保存では `カタログを選択してください` / `ロケーショングループを選択してください` が表示されます。`UNIQLO` と `ユニクログループ` を選択後に `発行する` を押しても通信は発生せず、モーダルは閉じませんでした。画面上にテナント選択欄はありませんが、実装上は未表示の `tenantID` が必須のため、現行UIでの発行完了は未確認です。

#### Webhook

「**Webhookを作成する**」ボタンからWebhookを追加できます。

**Webhook作成ダイアログの入力項目:**

| 項目（UIの原文） | 説明 | 選択肢・補足 |
|:--|:--|:--|
| イベント | Webhookを送信するトリガーとなるイベント | 「注文の作成」 / 「注文の更新」 / 「在庫の更新」の3種 |
| エンドポイント | Webhookの送信先URL | URLを入力する |

未入力で保存すると、`イベントを選択してください` / `エンドポイントを入力してください` がダイアログ内に表示され、ページ遷移せずWebhookも作成されません。2026-07-01追加確認では、イベント `注文の作成`、検証用 `.invalid` エンドポイントでWebhookを実作成でき、保存後はWebhookセクションにイベント名・エンドポイント・形式 `JSON` が表示されました。作成済み行のゴミ箱アイコンから `Webhookを削除する` ダイアログを開け、`削除する` 後はトースト `Webhookを削除しました` が出て行が消えます。編集/停止導線と送信成功/失敗ステータスは確認できませんでした。

## 補足・注意点

- 特権管理者グループの権限内容は変更できません。グループ名のみ変更可能です。
