# 注文管理

> 対象画面: 注文管理 / `/admin/orders`　|　最終確認: 2026-06-28

## この機能でできること

- 販売チャネルや連携経路から流入した注文の一覧表示・確認・追跡
- 下書き注文の一覧確認（現行空状態では作成ボタンはdisabled相当）
- 返品の一覧確認（返品の受領・完了は注文詳細内で扱う）
- 注文・返品から顧客詳細と売上実績へ反映される状態を確認する

> **注文の変更はどこで行う？（開発元の説明）**: Shopify経由の注文は、受注内容の変更などを**Shopify側の管理画面で行い、変更内容がSQへ自動反映**される運用です。SQ側から操作するのは、ECモール（楽天など）経由の注文の返品処理などが中心です。

### 注文詳細

内部GraphQLで投入した検証注文では、注文詳細に `返金`、`返品・交換を作成`、`返金履歴`、`出荷指示`、`在庫依頼`、ポイント/会員ランク算出対象、タグが表示されました。`inventoryBehavior=BYPASS` の注文では、画面には `この注文による在庫変動は無視されます` と表示され、在庫引当・手持ち変動は発生しません。

一方、`FAQ-REMAINING-DECREMENT-20260628135953` を `inventoryBehavior=DECREMENT`、`requiresFulfillment=true` で作成すると、注文は `支払い済み` / `未発送` になり、出荷指示 `#IO-1034` が `ORDER` / `出荷待ち` で生成されました。対象SKU `TEST_E2E_20260622_GU_1905_NAVY_M` は対象倉庫で `販売可能 1→0`、`引当済み 0→1`、`手持ち 1→1`、`売可 1→0` に変化しました。外部チャネル流入注文でも同じタイミング・同じ条件で動くかは接続環境で確認が必要です。

顧客詳細には直近の注文、初回/最終購入日、初回/最終購入場所が反映されました。売上実績には注文作成時に `数量 1 / 売上 ￥1,990` の明細が作成されました。

### 返品（/admin/order_returns）

注文に紐づく返品処理を一覧で確認するための画面です。

2026-06-28に空状態と、内部GraphQLで投入した返品データあり状態を確認しました。

| 確認項目 | 確認した表示 |
|:--|:--|
| 見出し | `返品` |
| 空状態メッセージ | `アイテムが見つかりませんでした` / `絞り込みや検索ワードを変更してみてください` |
| データあり状態の列 | `返品 / 返品状況 / 交換出荷状況 / 作成日時` |
| 返品ステータス | 作成直後 `情報 返送待ち`、受領後 `注意 返品受取済み`、完了後 `成功 返品処理済み` |
| 検索・絞り込みUI | 空状態では `検索と絞り込みの結果` / `絞り込みを追加` / `キャンセル` / `交換出荷` は表示なし。データあり状態では `絞り込みを追加` が表示され、条件は `キャンセル` / `交換出荷`。検索ボックスと保存ビューはなし |
| 新規作成ボタン | 表示なし |
| 直URL `/admin/order_returns/create` | `このページは存在しないようです` |
| 直URL `/admin/order_returns/{id}` | `このページは存在しないようです` |

返品単体の詳細ページはなく、返品の受領・完了は注文詳細内の返品ブロックで扱います。返品完了後、売上実績には `数量 -1 / 売上 -￥1,990` の明細が追加されました。

交換明細付き返品では、作成直後と商品受取後は交換出荷指示がまだ生成されず、返品完了後に `ORDER_EXCHANGE` の交換出荷指示 `#IO-1040` が `出荷待ち` で生成されました。

<!-- 連携実接続対象外: Recustomerや外部チャネル経由で作成された返品が同じ流れで反映されるか。 -->

直接 `/admin/order_returns/create` にアクセスすると404画面になります。返品は少なくとも返品一覧画面や直URLからは新規起票できません。

## サイドバーの未完了バッジ

注文管理は左サイドバーのメニュー項目に「未完了件数」を示すバッジが表示されます。

| 仕様 | 内容 |
|:--|:--|
| 表示タイミング | 未完了の出荷指示を持つ注文が1件以上ある場合に表示 |
| 0件のとき | バッジは表示されない |
| 100件超のとき | 「100+」と表示（正確な件数は表示されない） |

2026-06-28追加検証では、`OrderCountFilter` が `inventoryOutboundOrderWorkingStatuses` / `retailLocationID` / `isCancelled` のみを受け付け、`paymentStatus` 条件を持たないことを確認しました。DECREMENT注文で `WAITING` の出荷指示 `#IO-1034` ができると未完了件数が増え、交換返品完了で `ORDER_EXCHANGE` の `#IO-1040` ができた後はサイドバーにも `未完了の注文 2件` / `未完了の出荷指示 2件` が表示されました。

## 補足・注意点

- 注文は販売チャネル（Shopify等）から自動的に流入する設計です。今回の検証注文は内部GraphQLで投入したものなので、外部チャネル由来の注文流入・同期挙動とは分けて扱ってください。
- `/admin/orders/create` に直接アクセスすると「予期せぬエラーが発生しました」画面になります。注文管理一覧からの手動作成入口も確認できていません。
- 返品の起票は注文管理一覧や返品一覧画面からは行えません（作成ボタンが存在しない）。返品データがある場合、受領・完了は注文詳細内の返品ブロックで確認します。
- 内部GraphQLのDECREMENT注文では、在庫引当・販売可能数減少・出荷指示生成まで確認済みです。外部チャネル注文での同条件再現、実ポイント付与・返品時減算、外部チャネル返品の反映は未確認です。
- 下書き注文の「注文を作成する」ボタンに `href` がないまま変わらない場合、購入顧客データの有無だけでは判断できません。会社データ・担当者データ・検証用購入顧客が存在しても有効化されないことを確認済みです。OmnibusCore/リテールポータル/権限/チャネル接続など別条件は未確認です。
