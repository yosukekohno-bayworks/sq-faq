## 2026-06-27時点で実機確認済みの範囲

| 領域 | 確認状況 |
|:--|:--|
| 発注伝票 | 作成、発注済み、入荷指示作成、入荷待ち、入荷完了、入荷指示未作成時のキャンセルを確認。発注だけでは入荷指示・入荷予定は増えず、発注済み後の「入荷指示を作成する」で入荷先を指定すると反映される |
| 移動伝票 | 作成、同一ロケーション移動、在庫0からの移動/出荷、出荷前キャンセル、出荷後入荷、入荷後出荷を確認 |

| 調整伝票 | +1、-1、0の作成/実行、キャンセル、実施済み後キャンセル不可を確認。2026-06-18再確認では理由未選択は「理由を選択してください」で保存不可 |

| 在庫管理 | 9区分、履歴、販売可能数の直接編集、絶対値更新、負数保存を確認 |

## まだ断定しない範囲

| 領域 | 理由 |
|:--|:--|
| 実注文、返品、売上実績、顧客、ポイント付与、会員ランク反映 | 実データや外部チャネル連携が必要 |
| Shopify/OmnibusCore/スマレジ/リテールポータルの本番同期結果 | 連携先の実設定・実データが必要 |
| 発注起点の入荷指示を作成後、入荷完了前に発注伝票をキャンセルできるか | 2026-06-27時点で、入荷指示未作成ならキャンセル可能、入荷完了後はキャンセル不可まで確認済み。入荷待ち状態でのキャンセル可否は未確認 |
| 破損・検品・予備への在庫振替経路 | 管理画面上の操作UIを確認できていない |
| 取置伝票詳細の商品明細テーブル | 2026-06-28時点で複数伝票に同じ34行が表示されるため、伝票固有明細として扱わない。作成フォーム保存前の明細と在庫推移を優先 |

## FAQを書くときのルール

- 実機で確認した内容は、確認日と対象画面を残す。
- 画面文言と実際の結果が違う場合は、画面文言をそのまま信じず、実測結果を優先する。
- 「確保済み」は2種類あるため必ず分ける。
  - 在庫依頼ステータスの「確保済み」
  - 在庫区分の旧「確保済み」= 現「取置中」
- 在庫数の説明では、販売可能、取置中、手持ち、積送中、入荷予定を分ける。
- 入荷/出荷/調整/直接編集は巻き戻し不可または実質戻せないため、手順には確認ポイントを必ず入れる。
- 未確認の内容は `<!-- 残課題: 要確認 -->` を残し、FAQ本文で断定しない。

## FAQに必ず入れるべき注意点

| 注意点 | 書き方 |
|:--|:--|
| 発注しただけでは入荷予定は増えない | 「発注済み後の『入荷指示を作成する』で入荷先ロケーションを指定した時点で入荷待ちに出て、入荷予定が増える」と書く |
| 入荷は出荷前でも登録できる | 「出荷と入荷の順序は固定ではない。ただし移動伝票の完了には両方が必要」と書く |
| 部分入荷・過入荷ができる | 「予定数と実績数が一致しない場合は、必要に応じて完了に更新する」と書く |
| 在庫0でも移動/出荷できる | 「SQは不足をブロックしないため、マイナス在庫になり得る」と書く |
| 同じ配送元/配送先でも移動伝票を保存できる | 「同一ロケーション不可とは案内しない」と書く |
| 販売可能数の直接編集は絶対値 | 「数量は差分ではなく、変更後の販売可能数」と書く |
| 調整伝票の理由 | 「理由は `廃棄 / 紛失 / 在庫差異 / その他` の4択で選択推奨。2026-06-27の実機では未選択表示でも保存・実行できたため必須断定しない」と書く |
| 在庫依頼から作る移動伝票 | 「外部システム連携扱いになり、商品編集が制限される」と書く |
