# SQをはじめる — 全機能ガイド

> 対象読者: SQを初めて触る方
> 最終確認: 2026-06-11
>
> このガイドは二部構成です。**前半（1〜4章）は読み物**として通して読むと全体像がつかめます。**後半（5章以降）は辞書**として、必要なときに引いてください。

---

## 1. SQとは？ — まず3行で

1. SQは、ネット通販（Shopify）と実店舗（POSレジ）の**商品・在庫・注文を、裏側でまとめて管理する**システムです。
2. お客様が見る購入画面ではなく、**運営スタッフが使う管理画面**です。
3. いちばんの特徴は、**倉庫と店舗のあいだで在庫を取り寄せる流れ**を伝票でしっかり管理できることです。

![SQのホーム画面。左のサイドナビと、商品管理・在庫管理・注文管理などへのクイックリンク](https://raw.githubusercontent.com/yosukekohno-bayworks/sq-faq/main/_analysis/screenshots/01-home.png)

### たとえで言うと

お店にたとえると、SQは「売り場」ではなく**バックヤード**です。

| | 担当するもの | 例 |
|:--|:--|:--|
| 売り場（お客様が見る） | 商品を見せる・売る | ShopifyのECサイト、店舗のレジ（スマレジ） |
| **バックヤード（SQ）** | 商品台帳・在庫表・注文の控え・仕入れ記録 | **SQの管理画面** |

売り場で何かが売れたら、その情報は自動でバックヤード（SQ）に流れ込みます。スタッフはSQを見て「どこに何個あるか」「何を出荷すべきか」を判断します。

### SQがあると何がうれしい？

ネットと店舗の両方で販売していると、こんな悩みが出てきます。

- ネットでは「在庫あり」なのに、実際は店舗で売り切れていた
- 店舗にない商品を倉庫から取り寄せたいが、電話とメモで管理していて漏れる
- 注文・在庫・顧客の情報がチャネルごとにバラバラで、全体が見えない

SQはこれらを**1か所に集めて**解決します。

## 2. まず覚える言葉は4つだけ

最初からすべての用語を覚える必要はありません。この4つだけ押さえれば画面が読めるようになります。

| 言葉 | 意味 | 例 |
|:--|:--|:--|
| **商品とバリエーション** | 「商品」は品物のまとまり。「バリエーション」は色・サイズの組み合わせ1つ1つ | 商品「オーバーサイズスウェットシャツ」の中の「BEIGE / L」が1バリエーション |
| **SKU** | バリエーション1つ1つに付く管理番号。在庫はこの単位で数える | `486125-31-L` |
| **ロケーション** | 在庫の置き場所。倉庫と店舗の2種類 | 物流倉庫 W0001 / ユニクロ-銀座店 R0001 |
| **マスタ** | 最初に登録しておく基本情報のこと | テナント（事業部）・ロケーション・取引先（仕入先）・決済方法 など |

> SQの在庫はすべて「**どのSKUが、どのロケーションに、何個あるか**」という形で記録されます。この1文を覚えておくと、どの画面も読み解けます。

## 3. いちばん大事な流れ — 「取り寄せ販売」をストーリーで

SQの目玉機能です。実際の場面で見てみましょう。

> **場面**: 銀座店にお客様。欲しいスウェットのLサイズが店頭にない。でも倉庫には在庫がある。「お取り寄せしますね」と言ったあと、裏側では何が起きるのか？

| ステップ | 操作する画面 | やること | ステータス |
|:--|:--|:--|:--|
| ① 依頼 | 在庫依頼 | 「倉庫からLサイズを1枚、銀座店へ」という**在庫依頼**を作る | 確認待ち |
| ② 取り置き | 在庫依頼 詳細 | 倉庫の在庫を「**在庫を引当てる**」で確保する（お店の取り置き予約のイメージ） | 確保済み |
| ③ 伝票発行 | 確保済み 一覧 | 「**移動伝票を作成する**」を実行。この瞬間に**出荷指示と入荷指示が自動で生まれる** | 出荷待ち / 入荷待ち |
| ④ 倉庫が発送 | 出荷管理 | 発送したら「**出荷実績を登録する**」 | 出荷完了 |
| ⑤ 店舗に到着 | 入荷管理 | 受け取ったら「**入荷実績を登録する**」。銀座店の在庫が増える | 入荷完了 |

3つだけ注意点があります。

- **④と⑤は登録すると巻き戻せません。** 商品・数量・ロケーションをよく確認してから登録してください。
- 途中でやめたいときは、**①の段階（確認待ち）でしか取り消せません**。②で確保した後は最後まで進めるのが基本です。
- 在庫の数字が動くのは**③の伝票発行のタイミング**です。②の引当てでは、まだ画面上の在庫数は変わりません（詳しくは5章）。

> 図で見る: [SQのデータの流れ（図解）](データの流れ-図解.md)
> 操作手順の詳細: [取り寄せ販売の処理手順](../02-by-task/取り寄せ販売の処理手順.md)

| 分析 / 卸売 | **準備中**（画面はあるが未実装） |

### 商品の「ステータス」だけ先に覚える

商品には公開状態があります。つまずきやすいので先に説明します。

- ステータスは「**下書き**」「**公開中**」「**アーカイブ済み**」の3種類
- 作成フォームでは「下書き / 公開 / 非公開」の3択で、**「非公開」を選ぶと「アーカイブ済み」として作成されます**（同じ状態の別名です）
- 既存商品をアーカイブするには: 商品詳細 > 「その他の操作」 > 「商品をアーカイブする」
- アーカイブ済みの商品は、商品一覧の「アーカイブ済み」タブで確認できます

## 5. 在庫の数字の読み方（最重要・つまずきポイント）

### まずは2つだけ

在庫管理の画面には数字の列がたくさん並びますが、最初は2つだけ見れば大丈夫です。

- **手持ち** … 実際にそこにある数
- **販売可能** … 売ってもいい数

「手持ちはあるのに販売可能が少ない」ときは、誰かが取り置き（確保）していたり、出荷待ちだったりします。その内訳が残りの区分です。

### 画面によって区分の数が違う

> **2026-06-14更新**: 在庫区分の名称が変わりました（確定済み→引当済み、確保済み→取置中）。SKU詳細に「積送中」「入荷予定」も新設。

| 画面 | 区分 |
|:--|:--|
| 在庫一覧 | 販売可能 / 引当済み / 取置中 / 手持ち（旧: 販売不可 / 確定済み / 販売可能 / 手持ち） |
| SKU詳細（ロケーション別・9列） | 販売可能 / 引当済み / 取置中 / 破損 / 検品 / 予備 / 手持ち / 積送中 / 入荷予定 |

「破損」「検品」「予備」が販売不可の在庫です（旧一覧では「販売不可」に合算表示されていました）。

| 区分 | 意味 |
|:--|:--|
| **販売可能** | 販売に使える在庫数 |
| **引当済み** | 注文に引き当て済みの在庫数（旧「確定済み」） |
| **取置中** | 取置・移動用に確保された在庫数（旧「確保済み」。移動伝票作成時に「販売可能」から振替わる） |
| **破損** | 品質問題により販売不可の在庫数 |
| **検品** | 検品中で販売不可の在庫数 |
| **予備** | 予備として確保された在庫数 |
| **手持ち** | 上記6区分の合計 |
| **積送中** | 出荷済み・入荷前の輸送中の数（移動先で計上・手持ちに含まれない） |
| **入荷予定** | 移動伝票作成〜出荷前の入荷予定数（移動先で計上・手持ちに含まれない） |

#### 在庫数の計算式

手持ち = 販売可能 + 引当済み + 取置中 + 破損 + 検品 + 予備（積送中・入荷予定は含まない）

#### 在庫数の反映タイミング

取り寄せ販売の各操作で、在庫の数字がいつ動くかの対応表です。

| 操作 | 移動元ロケーションの在庫変化 |
|:--|:--|
| 在庫依頼を作成する（確認待ち） | 変化なし |
| 在庫を引当てる（在庫依頼が確保済みへ） | 変化なし |
| 移動伝票を作成する | 「販売可能」が減り、「取置中」が同数増える（手持ちは変わらない）。移動先には「入荷予定」が立つ |
| 出荷実績を登録する | 移動元の取置中・手持ちが減り、移動先の入荷予定が「積送中」へ移る |
| 入荷実績を登録する | 移動先の積送中が「販売可能・手持ち」へ移る |
| 移動伝票をキャンセルする | 「取置中」が減り、「販売可能」が同数戻る |
| 出荷実績を登録する | 移動元の手持ちが減る |
| 入荷実績を登録する | 移動先の手持ちが増える |

> 注意: 在庫数は**マイナスになることがあります**。在庫が0のロケーションから出荷実績を登録しても処理はブロックされないためです。マイナスになった場合は調整伝票で修正してください。

## 6. 基礎用語の補足

2章の「4つの言葉」以外で、画面に出てくる用語です。

### テナント
**事業部のような区分け単位**です。発注・ディスカウント・外部連携などの設定は、テナントに紐付いて管理されます。

> 例: 「ユニクロ」が1テナントとして設定されている状態

### 論理ロケーション（仮想店舗）
物理的には存在しない「仮想の店舗ロケーション」をSQ上に作成し、店舗在庫のうちECで販売してよい数だけをそこで管理する使い方があります。

作成は通常のロケーション作成と同じ操作です（専用機能はなし）。設定 > ロケーション > 「ロケーションを作成」で、場所種別「**店舗**」を選んで保存するだけで作れます。作成後は在庫一覧・SKU詳細・手動在庫依頼の選択肢に自動で表示されます。EC販売用の在庫数は、SKU詳細の「販売可能」列にある鉛筆アイコン（「販売可能数を編集」）から設定します。

- 実在しない入れ物なので、物理的な出荷・入荷はここでは行わない
- このロケーションをShopifyロケーションと連携させると、Shopifyはここの在庫数を見て販売可否を制御する

> 取り寄せ販売でこのロケーションをどう使うか: [取り寄せ販売（EC注文起点）の流れ](../02-by-task/取り寄せ販売（EC注文起点）の流れ.md)

### ロケーショングループ
**複数のロケーションをまとめたグループ**です。Shopify連携の設定でロケーショングループを指定すると、仕様上はグループ内で「在庫依頼を受け付ける」が有効な店舗に在庫リクエストが送られる想定です。取り寄せ販売（EC注文起点）の運用では、論理ロケーションと対象店舗をひとつのグループにまとめてShopify連携に設定します。2026-06-29にShopify接続済みレコードでロケーショングループ設定値は確認済みですが、実注文時の送付先絞り込みは正式連携SKUで再検証が必要です。手動の在庫依頼作成では、このフラグがOFFのロケーションもリクエスト先に選択・保存できます（2026-06-21実機確認）。

<!-- 連携実接続対象外: OmnibusCore連携が接続されている環境での在庫振り分けへの具体的な使われ方を確認する -->

### カタログ
**商品をグループ化する入れ物**です。外部連携、店舗受取、外部在庫同期などの設定側で「どの商品を対象にするか」を指定するために使います。

### マスタ一覧

マスタ（基本情報）は「設定」メニューで管理します。事前に登録しておくと、各フォームの選択肢に表示されます。

| マスタ名 | 用途 |
|:--|:--|
| テナント | 事業部の区分け |
| ロケーション | 在庫の置き場所（倉庫・店舗） |
| ロケーショングループ | ロケーションのグループ化 |
| ブランド | 商品のブランド分類 |
| 取引先 | 仕入先（発注時に選択） |
| 決済方法 | 注文時の支払い手段 |
| 販売員 | 店舗スタッフの情報 |

| 外部連携 | API（アプリ）・ロジザードZERO（倉庫管理）・Recustomer（返品・交換）との連携 |
| カスタムデータ | 翻訳（10言語対応）・メタフィールド・採寸ルール |

また、「設定」には**権限グループ**の管理機能があります。初期状態では「特権管理者」グループのみです。店舗スタッフなど権限を制限したいメンバー向けに、カスタムの権限グループを追加作成できます。

## 9. 次のステップ

SQを実際に使い始めるには、以下の順番で進めることを推奨します。

> **設定の順番と依存関係の完全版は [セットアップガイド](セットアップガイド.md) にまとめてあります。**「何から設定すればいいか分からない」場合はまずそちらを読んでください。

### ステップ1: 初期設定

まず「設定」メニューでマスタデータを整備します。

1. テナントを確認する
2. ロケーション（倉庫・店舗）を登録する
3. 取引先（仕入先）を登録する
4. 決済方法を登録する
5. 管理メンバーと権限グループを設定する

> 詳細: [初期設定の手順](../02-by-task/初期設定の手順.md)

### ステップ2: 商品・在庫を登録する

1. 商品を作成する（商品名・バリエーション・画像）
2. 在庫数を設定する
3. ステータスを「公開中」にする

> 詳細: [商品を作成する](../02-by-task/商品を作成する.md)

### ステップ3: 販売チャネルを接続する

Shopifyやスマレジとの連携を設定します。連携後、注文データが自動的にSQに取り込まれます。

> 詳細（Shopify）: [Shopify連携](../01-by-feature/Shopify連携.md)
> 詳細（スマレジ）: [スマレジ連携](../01-by-feature/スマレジ連携.md)

### ステップ4: 日々の運用

- 注文確認 → 出荷管理
- 在庫依頼 → 取り寄せ対応（倉庫↔店舗）
- 定期的な在庫調整（調整伝票）

> 詳細（取り寄せ販売）: [取り寄せ販売の処理手順](../02-by-task/取り寄せ販売の処理手順.md)
