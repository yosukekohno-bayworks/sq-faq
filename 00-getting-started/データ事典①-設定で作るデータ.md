# データ事典① — 設定で作るデータ

> 「設定」で作る各データについて、**項目・設定値ごとに「設定すると何が起きるか」**を表で引ける事典です。
> 読み方: 見出し＝データ。表の左列＝そのデータの項目や値。右列＝それがどの画面で・どう効くか。
> 商品・在庫・伝票・注文側は [データ事典②](データ事典②-商品・在庫・運用のデータとステータス.md)。

## テナント（設定 > テナント）

ブランド・部門などの運用単位を分けるための管理単位。ポイント、会員ランク、チャネル連携など複数機能の「紐づけ先」になる。

| 項目 / 値 | 効果 |
|:--|:--|
| テナント（データそのもの） | 発注伝票・ディスカウント・Shopify/OmnibusCore/スマレジ/リテールポータル連携・CSVエクスポート・ポイント一括加算の各フォームで**必須の選択肢**になる |
| 〃 | 注文一覧・顧客一覧の**絞り込み条件**になる |
| 〃 | 管理メンバー詳細「ユーザーテナント」で、メンバーごとのアクセス可否を制御する単位になる |
| 0件のとき | 上記フォームの選択肢が空になり、保存できない |

## ロケーション（設定 > ロケーション）

在庫を持つ場所（倉庫・店舗）。在庫はすべて「SKU × ロケーション」で記録される。

| 項目 / 値 | 効果 |
|:--|:--|
| 名前 | 「注文や商品などのエリアでこの名前が表示されます」（フォーム原文）。在庫・伝票の全画面に出る表示名 |
| コード | リテールポータル関連画面の「場所コードで検索する」の検索キー。ロケーション選択ダイアログの「場所コード」列に表示 |
| **場所種別 = 店舗** | リテールポータル連携の「店舗ロケーション」、店舗受取の「ロケーション」の選択肢に**出る**（倉庫種別は出ない）。各画面の「店舗」タブに表示 |
| **場所種別 = 倉庫** | リテールポータル連携の「在庫ロケーション」の選択肢に**出る**（店舗種別は出ない）。「倉庫」タブに表示 |
| **在庫依頼を受け付ける = ON** | ①在庫依頼の依頼先（リクエスト先）として選べるようになる ②取り寄せ販売で、ロケーショングループ内のこの店舗に**在庫リクエストが届くようになる** |
| 在庫依頼を受け付ける = OFF（初期値） | 在庫リクエストの送付対象にならない |
| **店舗受け取りを有効にする = ON** | 店舗受取（BOPIS）の受取店舗にできる |
| ポイント利用種別 = 値引き / 金種 | 店舗でのポイントの扱いを切り替える<!-- TODO: 要確認（値引きと金種の動作の違い） --> |
| ロケーション作成そのもの | 作成した瞬間から、在庫一覧のロケーション切替・SKU詳細のロケーション別行・全伝票のロケーション選択肢に**自動で出る** |

**論理ロケーション（仮想店舗）という使い方**: 物理の場所ではなく「店舗在庫のうちECで売ってよい数」を入れる器として、種別「店舗」のロケーションを1つ作る運用。SKU詳細の「販売可能数を編集」でEC枠を入れ、Shopifyのロケーションと連携すると、Shopifyはこの数で販売可否を決める。
> ⚠️ **未検証**: 「倉庫に実在庫がないと、ここへの出荷指示が在庫リクエストに変換されて店舗取り寄せが動く」という挙動は、Shopify実接続での確認ができていません（推測）。取り寄せ販売フローの根幹なので、実接続後に検証予定です。<!-- TODO: 要確認（Shopify接続後の引当・在庫リクエスト変換の実動作） -->

## ロケーショングループ（設定 > ロケーショングループ）

ロケーションの束。Shopify連携の必須項目。

| 項目 / 値 | 効果 |
|:--|:--|
| グループ（データそのもの） | **Shopify連携・OmnibusCore連携の「ロケーショングループ」欄の選択肢**になる |
| **デフォルトのロケーション** | Shopify注文の在庫は、ロケーション指定がない限り**ここから引き当てられる**（フォームヒント原文）。グループから外せない（外すには先にデフォルトを変更） |
| グループ内の「在庫依頼を受け付ける」ON店舗 | Shopify連携にこのグループを指定すると、**その店舗にだけ在庫リクエストが送られる** |
| 新しいロケーションを自動的に含める = ON | 以後作成したロケーションが自動でこのグループに入る（**作成後は変更不可**） |

## ブランド（設定 > ブランド）

| 項目 / 値 | 効果 |
|:--|:--|
| ブランド（データそのもの） | 商品作成・編集の「ブランド」欄（「ブランドを選択する」ダイアログ）の選択肢になる |
| **コード** | カタログの自動追加ルールや販売閾値ルールの自動追加ルールで、「ブランドコード」の**完全一致判定**に使われることがある |
| 外部ID | 外部システム連携用の識別子。**作成時のみ設定可・保存後は変更不可**（変更したい場合は削除→再作成） |
| 削除 | 取り消し不可（「この処理は巻き戻すことができません。」） |

ブランドはカタログのためだけに作るデータではありません。主目的は商品マスタ上の分類属性です。カタログ側で自動追加ルールに「ブランドコード」を指定した場合に限り、商品を集める条件として使われます。

## 権限グループ（設定 > 管理メンバー >「権限グループ一覧」）

38リソース ×「閲覧 / 編集」= **76権限**のセット。

| 項目 / 値 | 効果 |
|:--|:--|
| グループ（データそのもの） | 管理メンバーの「権限」欄の選択肢になる。**作っただけでは効かず、メンバーへの割り当てが必要** |
| ◯◯:read（閲覧権限） | その画面の閲覧可否 |
| ◯◯:write（編集権限） | その画面の作成・編集・削除の可否 |
| 特権管理者（初期グループ） | 全76権限。名前以外は変更・削除不可 |
| デフォルト全OFF | 新規グループは何も見えない状態から始まる（最小権限） |

**38リソース**: お知らせ / アプリ / ブランド / カタログ / 取引先企業 / 顧客マイル / 顧客ポイント / 顧客会員ランクルール / 顧客会員ランク / 顧客 / ディスカウント / 下書き注文 / 在庫 / 在庫リクエスト / 発注返品伝票 / 発注伝票 / 店舗受取ロケーションルール / ロケーショングループ / ロケーション / マイル商品 / ネクストエンジン連携 / OmnibusCore連携 / 注文 / ポイントキャンペーン / ポイントルール / 販売価格ルール / 商品 / 店舗ポータル連携 / 店舗スタッフ / 売上データ / Shopify連携 / スマレジ連携 / テナント / 翻訳ルール / 翻訳 / ユーザー権限 / ユーザーテナント / ユーザー

> この76権限は**アプリ（APIキー）の権限スコープとまったく同じ体系**。画面の権限とAPIの権限が1対1で対応する。

## 販売設定の4ルール（販売設定）

「設定で作るデータ」だが、チャネルへの価格・販売可否の制御に使う。**ここでは作成時の必須項目だけ**を示す。ステータス・在庫への効果・配下データ（通常/セール価格、SKUごとの上限・閾値など）の詳細は [データ事典②の「販売ルール」](データ事典②-商品・在庫・運用のデータとステータス.md) を参照。

| ルール（作成場所） | 作成時の必須項目 | 効果・注意 |
|:--|:--|:--|
| 販売価格ルール（販売設定 > 販売価格） | ルール名・**通貨**（5通貨。**作成後変更不可**） | Shopify連携の「販売価格ルール」に指定すると、そのルールでShopifyの販売価格が決まる。配下に通常価格・セール価格をSKU単位で登録 |
| 予約販売ルール（販売設定 > 予約販売） | タイトルのみ | 在庫0でも注文を受け続ける枠。OmnibusCore連携の「在庫予約ルール」に指定。配下にSKU＋販売数 |
| 販売上限ルール（販売設定 > 販売上限） | ルール名・**チャネル**（必須・1件以上） | **接続済みチャネルしか選択肢に出ない＝チャネル接続前は作成を完了できない**。配下にSKU＋上限数＋集計開始日時 |
| 販売閾値ルール（販売設定 > 販売閾値） | ルール名 | リテールポータル連携の「販売閾値ルール」に指定すると販売可否判断に使われる。配下にSKU＋閾値、ブランドコード一致の自動追加ルール |

> ⚠️ 各ルールの「上限到達時 / 閾値割れ時 / 在庫0時」の実際の挙動（売り止めが効くか等）は、チャネル未接続のため未検証です。詳細は事典②を参照。

## スマレジ連携（左メニュー「スマレジ」）

> 前提: スマレジ側にSQアプリのインストールが必要。

| 項目 / 値 | 効果 |
|:--|:--|
| テナント・契約ID・プラン | 必須の接続情報（プラン5択。初期値: リテールビジネス） |
| カタログ | スマレジに出す商品の範囲 |
| **在庫同期の方向 = スマレジからSQへ**（初期値） | 店舗POSの在庫数をSQへ反映する向きの設定 |
| **在庫同期の方向 = SQからスマレジへ** | SQの在庫数を店舗POSへ反映する向きの設定 |
| **在庫同期の方向 = 在庫を同期しない** | 在庫同期なし |
| 注文による在庫変動を起こさない = ON | スマレジ経由の注文でSQの在庫を動かさない |

> ⚠️ **未検証**: 在庫同期の「実際の反映タイミング・頻度・エラー時の動き」はスマレジ未接続のため確認できていません（設定項目とその方向の意味までは確認済み）。<!-- TODO: 要確認（スマレジ接続後の同期挙動） -->

## リテールポータル連携（左メニュー「リテールポータル」）

店舗スタッフ用画面の利用設定。**1接続 = 1店舗**。

| 項目 / 値 | 効果 |
|:--|:--|
| 店舗ロケーション | この店舗のポータルになる（選択肢は**店舗種別のみ**） |
| 在庫ロケーション | **在庫の引当元**になる（選択肢は**倉庫種別のみ**） |
| テナント・カタログ | 必須。カタログ＝店舗で扱う商品の範囲 |
| 販売閾値ルール（任意） | 在庫閾値による**販売可否の判断**に使われる |
| リテールポータルで販売員の選択を必須にする = ON | 店舗側の操作で販売員選択が必須になる |
| 配送先住所の編集を許可する = ON | 店舗スタッフが配送先を編集できる |
| 下書き注文の送料明細の編集を許可する = ON | 店舗スタッフが送料を編集できる |
| 下書き注文の注文明細の価格の編集を許可する = ON | 店舗スタッフが価格を編集できる |
| 下書き注文の完了を許可する = ON | 店舗スタッフが下書き注文を完了できる |
| **ユーザーを追加する**（作成後の詳細画面） | 追加した管理メンバーがこのポータルを使えるようになる（メールアドレス検索） |

### 店舗受取（リテールポータル > 店舗受取）

| 項目 / 値 | 効果 |
|:--|:--|
| ロケーション（店舗種別のみ） | 店舗受取を行う店舗として登録される |

## ロジザード連携（設定 > ロジザード）

WMS「ロジザードZERO」との接続。

| 項目 / 値 | 効果 |
|:--|:--|
| グループ番号・接続番号 | ロジザードZEROのAPIエンドポイントURLに使われる |
| オーナーID・ユーザーID・パスワード・アプリケーションキー | API認証情報 |
| 入荷予定登録ファイルID / パターンID | 入荷予定の連携APIに使われる |
| 出荷予定登録(通販/卸)ファイルID / パターンID | 出荷予定の連携APIに使われる |
| 出荷箱明細実績エクスポートファイルID / パターンID | 出荷実績の取得APIに使われる |
| **商品バリエーションを特定するキー** = SKUコード / JANコード / EANコード / UPCコード | SQとロジザードの**商品の突き合わせキー**になる |

## 横断早見表

### 作成後に変更できない項目（設定編）

| データ | 変更できない項目 | 対処 |
|:--|:--|:--|
| ブランド | 外部ID | 削除して再作成 |
| ロケーショングループ | 「新しいロケーションを自動的に含める」 | 新規グループを作成 |
| 翻訳ルール | 言語 | 新規ルールを作成 |
| ディスカウント | クーポンコード・テナント | 新規作成（事典②） |
| 販売価格ルール | 通貨 | 新規ルールを作成（事典②） |

### マスタ未登録のときの共通挙動

必要なマスタが0件だと、参照側フォームのドロップダウンには**何も表示されない**（エラーも出ない）。選択肢が空のときは、このページで「どこで作るデータか」を逆引きする。
