# データ事典① — 設定で作るデータ

> 「設定」で作る各データについて、**項目・設定値ごとに「設定すると何が起きるか」**を表で引ける事典です。
> 読み方: 見出し＝データ。表の左列＝そのデータの項目や値。右列＝それがどの画面で・どう効くか。
> 商品・在庫・伝票・注文側は [データ事典②](データ事典②-商品・在庫・運用のデータとステータス.md)。

---

## 候補表示の確認ルール（2026-06-27追加）

設定データは業務画面の選択肢・表示項目・権限制御として参照されますが、空表示の理由は一律ではありません。

| 空表示の種類 | 実機確認例 |
|:--|:--|
| 0件 / テナント不一致 | ディスカウントの顧客追加は顧客0件またはテナント不一致で空、`選択する` disabled。同一テナントの購入顧客は追加・削除可 |
| 検索/種別不一致 | 店舗ロケーション欄で倉庫コード、在庫ロケーション欄で店舗コードを検索すると0件 |
| 連携・チャネル未接続 | 販売上限ルールはチャネル候補がなく、保存時に `チャネルを選択してください` |
| 参照UIなし | 採寸ルール、通知用メール、APIアプリなどは「候補として選ぶ」業務画面を管理画面単体で確認できない |

詳細なマトリクスは `_analysis/MASTER-CHOICE-LIVE-MATRIX-2026-06-27.md` に残しています。既存のテナント・ブランド・取引先などを全削除して0件化する検証は、環境破壊になるため実施していません。

## テナント（設定 > テナント）

ブランド・部門などの運用単位を分けるための管理単位。ポイント、会員ランク、チャネル連携など複数機能の「紐づけ先」になる。

| 項目 / 値 | 効果 |
|:--|:--|
| テナント（データそのもの） | 発注伝票・ディスカウント・Shopify/OmnibusCore/スマレジ/リテールポータル連携・CSVエクスポート・ポイント一括加算の各フォームで**必須の選択肢**になる |
| 〃 | 注文一覧・顧客一覧の**絞り込み条件**になる |
| 〃 | 管理メンバー詳細「ユーザーテナント」で、メンバーごとのアクセス可否を制御する単位になる |
| 0件のとき | 既存テナント全削除による直接再現は未実施。発注伝票・各連携フォームではテナントselectに既存候補が表示され、空保存では必須エラーになる |

## ロケーション（設定 > ロケーション）

在庫を持つ場所（倉庫・店舗）。在庫はすべて「SKU × ロケーション」で記録される。

| 項目 / 値 | 効果 |
|:--|:--|
| 名前 | 「注文や商品などのエリアでこの名前が表示されます」（フォーム原文）。在庫・伝票の全画面に出る表示名 |
| コード | リテールポータル関連画面の「場所コードで検索する」の検索キー。ロケーション選択ダイアログの「場所コード」列に表示 |
| **場所種別 = 店舗** | リテールポータル連携の「店舗ロケーション」、店舗受取の「ロケーション」の選択肢に**出る**（倉庫種別は出ない）。店舗ロケーション欄で倉庫コード `W0001` を検索すると0件 |
| **場所種別 = 倉庫** | リテールポータル連携の「在庫ロケーション」の選択肢に**出る**（店舗種別は出ない）。在庫ロケーション欄で店舗コード `R0001` を検索すると0件 |
| **在庫依頼を受け付ける = ON** | EC注文起点などの自動在庫リクエストで、この店舗を送付対象にするための設定とされる。**手動の在庫依頼作成画面では、このフラグでリクエスト先候補は絞られない**（2026-06-21実機確認） |
| 在庫依頼を受け付ける = OFF（初期値） | 手動作成画面ではリクエスト先として選択・保存可能（2026-06-21実機確認）。自動在庫リクエストの送付対象から外れるかは、EC/リテールポータル接続環境で要確認 |
| **店舗受け取りを有効にする = ON** | 店舗受取用の設定。2026-06-27再確認では、店舗受取ルール作成フォームのロケーション選択肢は場所種別=店舗で絞られることを確認済み。ただし、このON/OFFが候補表示やストアフロントでの受取可否にどう影響するかは接続環境で要確認 |
| ポイント利用種別 = 値引き / 金種 | 店舗でのポイントの扱いを切り替える。2026-06-27再確認ではロケーション作成フォーム上で金種も選択肢として有効表示（disabledではない）。値引きとの動作差は開発元確認事項 |
| ロケーション作成そのもの | 作成後、在庫一覧のロケーション切替・SKU詳細のロケーション別行・伝票/設定系のロケーション選択肢に出る |
| ロケーションをアーカイブする | 詳細画面の `アーカイブする` でアーカイブ状態にする。アーカイブ後も一覧の `すべて` には残り、`公開=非公開`、`アーカイブ=アーカイブ済み` になる。解除後も公開は `非公開` のまま。店舗受取ルール、ロケーショングループ、販売員の候補にはアーカイブ済みロケーションが出る画面がある（2026-06-27実機確認） |

**論理ロケーション（仮想店舗）という使い方**: 物理の場所ではなく「店舗在庫のうちECで売ってよい数」を入れる器として、種別「店舗」のロケーションを1つ作る運用。SKU詳細の「販売可能数を編集」でEC枠を入れ、Shopifyのロケーションと連携すると、Shopifyはこの数で販売可否を決める。
> ⚠️ **未検証**: 「倉庫に実在庫がないと、ここへの出荷指示が在庫リクエストに変換されて店舗取り寄せが動く」という挙動は、Shopify実接続での確認ができていません（推測）。取り寄せ販売フローの根幹なので、実接続後に検証予定です。<!-- 連携実接続対象外: Shopify接続後の引当・在庫リクエスト変換の実動作 -->

## ロケーショングループ（設定 > ロケーショングループ）

ロケーションの束。Shopify連携の必須項目。

| 項目 / 値 | 効果 |
|:--|:--|
| グループ（データそのもの） | **Shopify連携・OmnibusCore連携の「ロケーショングループ」欄の選択肢**になる |
| **デフォルトのロケーション** | Shopify注文の在庫は、ロケーション指定がない限り**ここから引き当てられる**（フォームヒント原文）。グループから外せない（外すには先にデフォルトを変更） |
| グループ内の「在庫依頼を受け付ける」ON店舗 | Shopify連携にこのグループを指定したときの自動在庫リクエスト送付先を制御する設定とされる。実接続での送付先絞り込みは要確認 |
| 新しいロケーションを自動的に含める | 作成フォームには表示されない。既存グループ詳細ではチェック欄が disabled 表示で、標準UIから変更できない（2026-06-27実機確認） |

## ブランド（設定 > ブランド）

| 項目 / 値 | 効果 |
|:--|:--|
| ブランド（データそのもの） | 商品作成・編集の「ブランド」欄（「ブランドを選択する」ダイアログ）の選択肢になる |
| **コード** | カタログの自動追加ルールや販売閾値ルールの自動追加ルールで、「ブランドコード」の**完全一致判定**に使われることがある |
| 外部ID | 外部システム連携用の識別子。**作成時のみ設定可・保存後は変更不可**（変更したい場合は削除→再作成） |
| 削除 | 取り消し不可（「この処理は巻き戻すことができません。」） |

ブランドはカタログのためだけに作るデータではありません。主目的は商品マスタ上の分類属性です。カタログ側で自動追加ルールに「ブランドコード」を指定した場合に限り、商品を集める条件として使われます。

## 権限グループ（設定 > 管理メンバー >「権限グループ一覧」）

38リソース ×「閲覧 / 編集」= **76権限**のセット。

| 項目 / 値 | 効果 |
|:--|:--|
| グループ（データそのもの） | 管理メンバー作成の「権限グループ」ラジオ候補、メンバー詳細の「権限」選択肢になる。**作っただけでは効かず、メンバーへの割り当てが必要** |

## 販売設定の4ルール（販売設定）

「設定で作るデータ」だが、チャネルへの価格・販売可否の制御に使う。**ここでは作成時の必須項目だけ**を示す。ステータス・在庫への効果・配下データ（通常/セール価格、SKUごとの上限・閾値など）の詳細は [データ事典②の「販売ルール」](データ事典②-商品・在庫・運用のデータとステータス.md) を参照。

| ルール（作成場所） | 作成時の必須項目 | 効果・注意 |
|:--|:--|:--|
| 販売価格ルール（販売設定 > 販売価格） | ルール名・**通貨**（5通貨。**作成後変更不可**） | Shopify連携の「販売価格ルール」に指定すると、そのルールでShopifyの販売価格が決まる。配下に通常価格・セール価格をSKU単位で登録 |
| 予約販売ルール（販売設定 > 予約販売） | タイトルのみ | 在庫0でも注文を受け続ける枠。OmnibusCore連携の「在庫予約ルール」に指定。配下にSKU＋販売数 |
| 販売上限ルール（販売設定 > 販売上限） | ルール名・**チャネル**（必須・1件以上） | **接続済みチャネルしか選択肢に出ない＝チャネル接続前は作成を完了できない**。配下にSKU＋上限数＋集計開始日時 |
| 販売閾値ルール（販売設定 > 販売閾値） | ルール名 | リテールポータル連携の「販売閾値ルール」に指定すると販売可否判断に使われる。配下にSKU＋閾値、ブランドコード一致の自動追加ルール |

> ⚠️ 各ルールの「上限到達時 / 閾値割れ時 / 在庫0時」の実際の挙動（売り止めが効くか等）は、チャネル未接続のため未検証です。詳細は事典②を参照。

## 横断早見表

### 作成後に変更できない項目（設定編）

| データ | 変更できない項目 | 対処 |
|:--|:--|:--|
| ブランド | 外部ID | 削除して再作成 |
| ロケーショングループ | 「新しいロケーションを自動的に含める」 | 作成フォームにはなく、既存詳細では disabled 表示。標準UIから変更不可として扱う |
| 翻訳ルール | 言語 | 新規ルールを作成 |
| ディスカウント | クーポンコード・テナント | 新規作成（事典②） |
| 販売価格ルール | 通貨 | 新規ルールを作成（事典②） |

### マスタ候補が出ないときの見方

候補が出ない原因は、未登録・0件だけではありません。2026-06-27の確認では、ロケーションの種別不一致、チャネル未接続、参照UI自体が存在しないケースもありました。選択肢が空のときは、このページで「どこで作るデータか」と、対象フォームの種別条件・連携状態を逆引きしてください。
