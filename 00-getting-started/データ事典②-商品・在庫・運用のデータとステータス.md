# データ事典② — 商品・在庫・運用のデータとステータス

> 商品・在庫・伝票・販売ルール・注文・CRMの各データについて、**項目ごとの効果**と**ステータスごとの「入る条件 → 次の操作 → 在庫への効果」**を表で引ける事典です。
> 設定で作るマスタは [データ事典①](データ事典①-設定で作るデータ.md)。

---

## 商品（商品管理）

| 項目 / 値 | 効果 |
|:--|:--|
| **商品コード**（必須） | 商品一覧・カタログでの**唯一の検索キー**（商品名・タグでは検索できない）。**作成後は変更不可**。使える文字は**半角英数字・ハイフン・アンダースコアのみ** |
| 商品名・説明文・メディア | 表示用。画像はローカルファイルのみ（URL指定不可） |
| **製造元** | カタログの自動追加ルール「製造元」の**完全一致判定**に使われる |
| **ブランド** | ブランドマスタ（設定 > ブランド）から選択。コードが自動追加ルール「ブランドコード」の判定に使われる |
| タグ | 商品サイドバーに表示されるだけ。**一覧での絞り込みには使えない** |
| 一覧の「在庫」列 | **バリエーション数**を表示（実在庫数ではない） |
| 一覧の「カタログ」列 | 所属カタログ数 |
| 「保存する」ボタン | 作成フォームでは未入力でも押せる。空保存時は「商品コードを入力してください」「商品名を入力してください」「選択してください」などのインラインエラーが表示される。編集画面では変更が無いと保存ボタンが無効 |
| 削除 | 取り消し不可。確認ダイアログに「この処理は巻き戻すことができません。」と表示される。削除後の詳細URLは「該当するProductが見つかりませんでした。」になる |

### 商品ステータス（画面で表記が変わる）

| 作成フォーム | 編集フォーム | 一覧タブ | この状態の効果 |
|:--|:--|:--|:--|
| 下書き（初期値） | 下書き | 下書き | 作業中。チャネルに出ない |
| 公開 | 公開中 | 公開中 | 販売中 |
| 非公開 | アーカイブ済み | アーカイブ済み | **同じ状態**。「販売チャネルと管理画面上で非表示」 |

| 操作 | 結果 |
|:--|:--|
| 編集画面のステータス変更 | 「公開中 ⇔ 下書き」の2択のみ |
| その他の操作 > 商品をアーカイブする | アーカイブ済みへ（チャネル・管理画面で非表示に） |
| その他の操作 > 商品のアーカイブを解除する | **必ず「下書き」に戻る**（公開中には戻らない） |
| その他の操作 > 商品を削除する | 確認ダイアログ後に完全削除。復元不可 |

## バリエーション（SKU）と オプション軸

| 項目 / 値 | 効果 |
|:--|:--|
| **オプション軸**（オプション名＋種別: サイズ / カラー / その他） | バリエーションの組み合わせの軸になる。**商品作成時にしか設定できない**（後から軸の追加・種別変更は不可） |
| オプション値 | バリエーション追加フォームの選択肢になる（新しい値は先に商品編集側で追加） |
| **上代**（必須） | 販売価格のベース |
| **SKU (最小管理単位)**（必須） | 在庫管理・各種ルール登録・CSV・検索の**キー** |
| **在庫を追跡する** | SKUの在庫追跡フラグ（CSV項目: `is_tracked`）。作成フォーム初期値はOFF。2026-06-21の実機確認では、OFFでも在庫管理一覧・SKU詳細・ロケーション別在庫テーブルに表示され、管理画面内の在庫編集・伝票・販売設定の候補からも除外されない |
| **在庫切れの場合でも販売を続ける = ON** | 在庫0でも販売を止めない。作成フォーム初期値はOFF |
| 配送を必須にする（初期値ON） | 配送が必要な商品として扱う |
| 重量・単位（グラム/キログラム/オンス/ポンド） | 配送情報 |
| 関税情報（原産国コード・HSコード） | 越境・関税用 |
| バーコード / JAN / EAN / UPC | ロジザード連携の商品突き合わせキーの候補（UPCは編集画面のみ） |
| メーカーSKU | 任意の別SKU |
| 原価（**編集画面の原価セクション**） | バリエーション作成画面には「仕入価格」「原価」の入力欄はない。原価は作成後にバリエーション編集画面の「原価を登録する」から設定する。CSVインポート「原価」で一括設定可 |
| 在庫管理リンク（編集画面上部） | 該当SKUの在庫詳細画面へ遷移し、実在庫数を確認できる |

> 作成フォームと編集フォームで項目が一部違う（編集: 原価・UPCあり）。送信ボタンは作成「作成する」・編集「更新する」で、**上代・SKUが未入力だと押せない**。

#### 「在庫を追跡する」の実効範囲（実機確認: 2026-06-21）

- **管理画面内ではOFFでも対象に残る**: 在庫一覧・SKU詳細・販売可能数の直接編集・移動伝票・調整伝票・取置伝票・在庫依頼・発注伝票・店舗受取・販売価格・予約販売・販売閾値のSKU候補で、OFF SKUが表示/選択できることを確認した。
- **在庫数は編集できる**: OFF状態のSKUでも、在庫詳細の鉛筆編集でロケーション別の販売可能数を変更できた。
- **CSVでは独立項目**: 商品バリエーションCSVでは `is_tracked`（TRUE/FALSE）として管理される。`inventory_policy`（在庫切れの場合でも販売を続ける、CONTINUE/DENY）とは別項目。作成フォームの初期値は `is_tracked` 相当がOFF、`inventory_policy` 相当もOFFです。
- **未確認**: Shopify / OmnibusCore / リテールポータルなど外部チャネル連携時の在庫同期・販売可否・注文引当への影響は未確認。

## カタログ（商品管理 > カタログ）

商品の入れ物。**連携・店舗受取・外部在庫同期などで「どの商品を対象にするか」を決めるための商品グループ**。

| 項目 / 値 | 効果 |
|:--|:--|
| カタログ（データそのもの） | Shopify連携（必須）・OmnibusCore・スマレジ・リテールポータル連携の「カタログ」欄の選択肢になり、**その連携で扱う対象商品グループ**を決める |
| 商品の追加（手動 / 自動追加ルール / CSV） | カタログに入った商品が、参照先の連携・業務設定で対象商品として扱われる |
| 一覧の「販売先」列 | このカタログを参照しているチャネル連携の数 |

### 自動追加ルール（カタログ詳細 > 自動追加ルール）

| 項目 / 値 | 効果 |
|:--|:--|
| 項目 = 製造元 / ブランドコード、条件 = 一致する、値 | **完全一致**した商品が自動でカタログに追加される（大文字小文字・スペースの違いも別物扱い） |

## 店舗受取バリエーション（商品管理 > 店舗受取）

| 項目 / 値 | 効果 |
|:--|:--|
| 店舗受取対象SKU | そのSKUが**店舗受取可能**になる。受取の「店舗」側は事典①の[リテールポータル > 店舗受取（ロケーションルール）](データ事典①-設定で作るデータ.md)で登録 — 店舗受取は「対象バリエーション（ここ）」と「受取店舗（リテールポータル側）」の対で成立する。2026-06-27実機では `/admin/local_pickup_product_variants` の「バリエーションを追加する」からSKU選択ダイアログを開き、`TEST_E2E_20260622_GU_1905_NAVY_M` を追加すると一覧20→21件へ即時反映。追加済みSKUを再度選択しても重複行は増えず、行チェック後の `削除する` で対象解除でき、削除後は同じSKUが追加候補に戻る |

## 在庫（在庫管理）— 区分

> **2026-06-14 更新**: 在庫区分の名称が変わりました（「確定済み」→**「引当済み」**、「確保済み」→**「取置中」**）。さらに **「積送中」「入荷予定」** の2列が新設されています。一覧の列構成も変わりました（下記）。

**手持ち = 販売可能 + 引当済み + 取置中 + 破損 + 検品 + 予備**（6区分の合計）。
**「積送中」「入荷予定」は手持ちには含まれない別枠**（移動の輸送過程で移動先に立つ数。手持ちより後ろの列）。

- **一覧（在庫管理トップ）の列**: 商品 / 品番(SKU) / **販売可能 / 引当済み / 取置中 / 手持ち**（旧: 販売不可 / 確定済み / 販売可能 / 手持ち。「販売不可」合算列は廃止）
- **SKU詳細の列（9列）**: ロケーション / 販売可能 / 引当済み / 取置中 / 破損 / 検品 / 予備 / 手持ち / 積送中 / 入荷予定

### 区分ごとの「何で増えて、何で減るか」

| 区分 | 意味 | 増える操作 | 減る操作 |
|:--|:--|:--|:--|
| 販売可能 | 売ってよい数 | 鉛筆編集／CSV「販売可能在庫」／移動伝票キャンセル（戻り）／入荷実績（移動先に到着） | **移動伝票の作成**・**取置伝票の作成**（取置中へ振替） |
| 引当済み | 注文に引き当て済み（旧: 確定済み） | 注文の引き当て | 出荷の進行 |
| 取置中 | 取置・移動用に確保（旧: 確保済み） | **取置伝票の作成**／**移動伝票の作成**（販売可能から振替） | 取置の処理済みマーク（販売可能へ戻る）／移動の出荷実績（手持ちごと減る）／移動伝票キャンセル |
| 破損 / 検品 / 予備 | 販売不可の内訳 | 2026-06-19の実機確認では、SQ管理画面上にこれらの区分へ直接振り替える操作UIは確認できない | 同左 |
| 手持ち | 実在庫の合計（上記6区分の和） | **入荷実績の登録**（移動先）／鉛筆編集（販売可能と同時に） | **出荷実績の登録**（移動元） |
| 積送中 | **出荷済み・入荷前の輸送中数**（移動先で計上・手持ち外） | 出荷実績の登録（移動先の入荷予定が積送中へ） | 入荷実績の登録（販売可能・手持ちへ） |
| 入荷予定 | **移動伝票作成〜出荷前の入荷予定数**（移動先で計上・手持ち外） | 移動伝票の作成（移動先） | 出荷実績の登録（積送中へ移る） |

### 操作 → 在庫の動き（時系列の早見表・実機検証済み 2026-06-14）

| 操作 | 移動元の変化 | 移動先の変化 |
|:--|:--|:--|
| 在庫依頼を作成（確認待ち） | **変化なし** | — |
| 在庫を引当てる（在庫依頼ステータスが確保済みへ） | **変化なし**（数字はまだ動かない） | — |
| 移動伝票を作成 | 販売可能 −n、取置中 +n（手持ち不変） | **入荷予定 +n** |
| 出荷実績を登録 | 取置中 −n、**手持ち −n** | 入荷予定 −n → **積送中 +n** |
| 入荷実績を登録 | — | 積送中または入荷予定 −n → **販売可能 +n・手持ち +n**。出荷前でも入荷登録は可能 |
| 移動伝票をキャンセル | 取置中 → 販売可能に戻る | — |
| 取置伝票を作成 | 販売可能 −n、取置中 +n | — |
| 取置を処理済みにマーク | 取置中 −n → 販売可能 +n（解放） | — |
| 調整伝票を実行 | 増減数がそのまま反映 | — |
| 鉛筆で販売可能数を編集 | 「販売可能」を**入力値そのものに設定**（絶対値）。手持ちは差分だけ増減 | — |

> 「在庫依頼」のステータス**「確保済み」は名称変更なし**（在庫の区分列だった「確保済み」だけが「取置中」に改名）。混同しないこと。

### 在庫まわりのその他のデータ

| データ / 項目 | 効果 |
|:--|:--|
| **マイナス在庫** | 手持ち0から移動伝票作成・出荷実績登録をしてもブロックされず、**在庫がマイナスになる**。修正は調整伝票 |
| 調整履歴（在庫詳細 >「調整履歴」） | 在庫区分の変更を差分付きで記録（例: 5 (+5)）。鉛筆編集の「理由」もアクティビティとして残る。**在庫がいつ・なぜ変わったかはここで監査** |
| 鉛筆編集（販売可能数を編集） | 理由（必須）＋数量。**数量は絶対値**（入力値が新しい販売可能数になる。加算ではない）。手持ちは差分だけ増減。**負数も保存できる**。調整伝票は作られない別経路。論理ロケーションのEC枠設定もこれ |
| 在庫振替（その他のアクション） | **同一商品内の別バリエーション**へ在庫を振り替える（別商品へは不可） |

## 調整伝票（在庫管理 > 調整伝票）

理由付きで在庫を増減する伝票。マイナス在庫の修正手段。

| 項目 / 値 | 効果 |
|:--|:--|
| ロケーション・商品 | 必須。空保存時に「ロケーションを選択してください」「商品を1つ以上追加してください」が出る |
| 理由 | 選択推奨。4択: 廃棄 / 紛失 / 在庫差異 / その他（見本・棚卸差異・破損・検品・予備は無い）。2026-06-27の実機では未選択表示でも保存・実行できたため、必須断定しない |
| 増減数 | プラスで増・マイナスで減。「実行」で反映。0でも伝票作成可能 |

| ステータス | 入る条件 | 次にできる操作 | 在庫への効果 |
|:--|:--|:--|:--|
| 未実施 | 保存 | 「実行」「キャンセル」。項目の編集も可 | まだ無し |
| 実施済み | 「実行」→ 確認ダイアログ | キャンセルボタンが残る場合があるが実行不可 | 増減数が反映。**巻き戻し不可**（「この処理は巻き戻すことができません。」） |
| キャンセル | 未実施詳細の「キャンセル」 | 実行ボタンが残る場合があるが実行不可 | 無し。確認ダイアログなしで即時にキャンセルへ変わる |

> 未保存の変更があると「実行」が押せない（先に保存）。

## 取置伝票（在庫管理 > 取置伝票）

在庫を確保しておく伝票。**保存した瞬間に在庫が動く**。

| ステータス | 入る条件 | 次にできる操作 | 在庫への効果 |
|:--|:--|:--|:--|
| 未処理 | 作成・保存 | 「処理済みとしてマークする」 | 対象在庫が**「取置中」区分になる**（確保が続く） |
| 処理済み | 「処理済みとしてマークする」→「実行する」 | — | **取置中の在庫が解放される**（販売可能へ戻る）。確認ダイアログ原文: 「取置伝票を処理済みにすることで、取置中の在庫が解放されます。」 |

未処理詳細に独立した「キャンセル」ボタンは表示されない。取置を終了する操作は「処理済みとしてマークする」。

2026-06-28の実機確認では、作成フォーム上で12明細・数量入力12件を保存し、処理済み化まで完了できた。一方で、取置伝票詳細の商品明細テーブルは複数伝票に同じ34行が表示されたため、伝票固有明細としては扱わず、保存前フォームの明細・一覧ステータス・在庫推移を優先して確認する。

## 移動伝票（在庫管理 > 移動伝票）

ロケーション間の在庫移動。**作成した瞬間に出荷指示・入荷指示が自動生成される**。

| 項目 / 値 | 効果 |
|:--|:--|
| 配送元・配送先（必須） | 配送元を選ぶまで商品欄は無効 |
| 配送元 = 配送先 | 同じロケーションでも保存できる。誤作成時は詳細の「その他の操作」>「キャンセル」で取り消す |
| 配送元在庫0 | 在庫0のSKUでも移動伝票を作成でき、出荷実績登録もブロックされない |
| 数量0 | 保存不可。「quantityは1以上で入力してください」 |
| 作成（保存）した瞬間 | ①移動元: 販売可能→取置中へ振替（手持ち不変）②移動先: 入荷予定 +n ③**出荷指示（#IO）と入荷指示（#II）が自動生成** ④出荷管理・入荷管理に未完了バッジ |

| ステータス（一覧タブ） | 入る条件 | 次に起きること | 在庫への効果 |
|:--|:--|:--|:--|
| 出荷作業 | 作成直後 | 出荷指示の出荷実績登録を待つ。入荷を先に登録することも可能 | 取置中のまま |
| 一部受領済み | 一部の入荷実績が登録された状態。今回の標準stagingでは対象一覧データがなく、一覧タブ表示条件は追加データ依存 | 残りの入荷実績登録、または入荷指示の「完了に更新する」 | （移った分だけ手持ちが移動先へ） |
| 受領済み | 入荷が完了した状態。今回の標準stagingでは対象一覧データがなく、一覧タブ表示条件は追加データ依存 | 出荷未完了なら出荷実績登録を待つ | 入荷済み分が移動先へ反映 |
| 完了＋入荷完了（詳細バッジ） | 出荷・入荷の実績登録が完了 | — | 手持ちが移動元→移動先へ移った状態 |
| キャンセル | その他の操作 >「キャンセル」 | 紐づく**出荷指示・入荷指示もすべてキャンセル** | 取置中が**販売可能に全数戻る** |

> **伝票自体にステータスを進めるボタンは無い**。出荷・入荷の実績登録に連動して自動で進む。入荷は出荷前でも登録でき、部分入荷・過入荷も保存できる。一覧タブの「一部受領済み / 受領済み」への正確な表示条件は未検証。

**外部連携由来は編集不可**: 取り寄せ販売フロー（在庫依頼の確保済み）から自動生成された移動伝票は、管理画面で**商品の追加・削除などの編集ができない**。詳細画面に注意文が出る — 「**この移動伝票は外部システムから連携されているため、編集（商品の追加や削除など）は制限されています。連携元システムの管理画面より操作を行ってください。**」。手動編集しようとして詰まらないこと。

**キャンセルの注意（行き止まり）**: 取り寄せフロー経由の移動伝票をキャンセルすると、①出荷指示はステータス別タブには出ず、2026-06-27の実機再確認では「すべて」タブに「キャンセル済み」として表示される行があった（関連する移動伝票詳細のリンクや直接URLから詳細を開ける場合があり、条件指定エクスポートにはキャンセル済み指定もある）②入荷指示は「キャンセル」タブに残る ③**元の在庫依頼は「確保済み」のまま残り、再作成もクローズもできない**。やり直しは在庫依頼の新規作成。

## 在庫依頼（取り寄せ販売の起点）

「この店舗へ、ここから在庫をn個確保して送ってほしい」という依頼。

| 項目 / 値 | 効果 |
|:--|:--|
| 商品バリエーション・希望数・移動先ロケーション（必須） | 移動先＝在庫の届け先（詳細画面では「引当先ロケーション」表記） |
| リクエスト先ロケーション（任意） | 指定すると**引当ダイアログのロケーションがそこだけに絞られる**（詳細では「依頼先」表記） |
| チャネル列 | 依頼の発生元を表示（管理画面で作成した依頼は「web-admin」と表示される） |

| ステータス | 入る条件 | 次にできる操作 | 在庫への効果 |
|:--|:--|:--|:--|
| 確認待ち | 作成 | 「在庫を引当てる」／「在庫依頼をクローズする」 | **無し** |
| 確保済み | 引当て実行 | 確保済み一覧から「移動伝票を作成する」。**クローズは不可** | **まだ無し**（在庫が動くのは移動伝票作成時） |
| 終了 | クローズ（確認待ちのときだけ可能） | — | 無し |

> **確保済み一覧**（在庫依頼 > 確保済み）: 「移動伝票を作成する」で移動伝票＋出荷指示＋入荷指示が一括生成され、行が消える。作成された移動伝票は外部システム連携扱いになり商品編集が制限される。移動完了後も元の在庫依頼は「確保済み」のまま。

### 在庫リクエスト（EC注文起点・自動発生）

| 起きること | 内容 |
|:--|:--|
| 発生条件 | EC注文の連携時、倉庫の実在庫が不足しているSKUに自動発生 |
| 届く先 | 仕様上はロケーショングループ内で「**在庫依頼を受け付ける**」がONの店舗だけとされる。実接続での送付先絞り込みは要確認（手動作成画面ではOFF店舗もリクエスト先に選択・保存可能） |
| 店舗側の操作 | リテールポータルで確認 → 在庫の確保登録 → 移動伝票の発行（ロケーションごとに分割）→ 店舗から倉庫へ出荷<!-- 実機環境提供待ち: ポータル側の画面ラベル原文 --> |

## 入荷指示・入荷実績（入荷管理）

移動伝票から出荷指示と同時に**自動生成**。種別列は「移動」。

| ステータス（一覧タブ） | 入る条件 | 次にできる操作 | 在庫への効果 |
|:--|:--|:--|:--|

## 発注伝票（発注管理）

| 項目 / 値 | 効果 |
|:--|:--|
| 取引先（必須） | 取引先マスタ（設定 > 取引先）の選択肢から |

| 下書き | 発注伝票を保存する | 編集、発注、キャンセル | なし |

| 入荷待ち | 発注済み詳細の「入荷指示を作成する」で入荷先ロケーションを指定して保存する | 入荷実績登録 | 入荷管理の「入荷待ち」タブに表示され、対象SKUの入荷予定が増える |

> 発注ダイアログの文言は2026-06-27時点で「発注後は伝票を編集できません。発注後に入荷指示を作成できます。」です。発注済みにしただけでは入荷管理一覧に出ず、入荷予定も増えません。発注済み詳細の「入荷指示を作成する」で入荷先を指定して保存した時点で、入荷管理の「入荷待ち」タブとSKUの入荷予定に反映されます。

## 販売ルール（販売設定）

> 作成時の必須項目・通貨などは [データ事典①の「販売設定の4ルール」](データ事典①-設定で作るデータ.md) に整理。ここでは配下データとステータス・効果を扱う。

> ⚠️ **共通の対象外事項**: 以下4ルールの「在庫0時に本当に注文を受け続けるか」「上限に達したら売り止まるか」「閾値を割ったら販売停止・通知が走るか」といった**到達時・適用時の実挙動は、チャネル実接続が必要なため今回の検証対象外**です。設定フォームの項目と意味までは確認済みです。

|:--|:--|
| 配下: バリエーション＋**販売数** | **在庫0でも注文を受け続けられる数**（バックオーダー枠）として設定する値 |

| 割引要件 = 最低購入額 / 最低購入数量 | 適用の最低条件 |
| すべての商品を割引対象に設定する = OFF | 作成後に「商品管理」タブで対象SKUを個別追加する運用になる |
| すべての商品を割引対象に設定する = ON | 詳細の適用条件 > 対象商品が「成功 / すべての商品」と表示される |
| お客様1人につき1回のみの使用とする = ON | 1人1回制限 |
| 有効期間（開始・終了とも必須） | 下のステータスを自動で切り替える |
| 詳細「顧客管理 / 店舗管理 / 商品管理」タブ | 対象顧客・対象ロケーション（グループ）・対象SKUの個別追加。顧客追加は同一テナントの購入顧客データが必要。店舗はロケーション追加・適用種別変更・削除まで可能 |
| 詳細「Shopify連携」セクションの「連携する」 | ディスカウントをShopifyへ連携する |
| 詳細「利用履歴」タブ | 利用された注文の履歴（CSVエクスポート可） |

| ステータス | 条件 |
|:--|:--|
| スケジュール済み | 開始日時より前 |
| 有効 | 期間内（注文に適用可能） |
| 期限切れ | 終了日時を過ぎた |

### 注文ポイント付与ルール（ポイント）

| 項目 / 値 | 効果 |
|:--|:--|
| 購入金額＋ポイント | 付与率（例: 100円ごとに1ポイント） |
| ポイント利用分を含める / セール価格の商品を除外する | ポイント利用分やセール価格商品の付与対象扱いを制御する |
| **付与までの日数（オンライン）** | 画面説明上、**出荷完了**からポイントが利用可能になるまでの日数。実付与は未確認 |
| **付与までの日数（店舗）** | 画面説明上、**決済完了**からポイントが利用可能になるまでの日数。実付与は未確認 |
| 利用可能になってから有効な日数 | ポイントが利用可能になってから失効までの日数 |
| 会員ランク算出ルール（紐づけ） | このルールの顧客を指定の算出ルールで管理。**先に算出ルールを作っておく必要がある** |
| ポイントキャンペーン優先設定 | 競合時に開始日時が**新しい / 古い**どちらのキャンペーンを適用するか |
| 詳細「会員ランク倍率」 | `/rank_multipliers` でランクごとの付与倍率を設定。追加ダイアログは確認済み、保存後反映は未確認 |
| 詳細「商品倍率」 | `/point_multiplier_products` で商品ごとの付与倍率を設定。追加ダイアログは確認済み、保存後反映は未確認 |

### ポイントキャンペーン

| 項目 / 値 | 効果 |
|:--|:--|
| 種別 = なし | 全注文が対象。倍率/追加ポイントをフォームで直接入力 |
| 種別 = 会員ランク / 購入金額 / 商品 | 保存後に詳細ページで対象（会員ランク×倍率、購入金額×倍率、商品SKU×倍率）を設定 |
| **ポイント付与方法**（合計倍率 / 追加ポイント） | 計算方法。**作成後変更不可**。例（原文）: 「キャンペーン倍率3 ＋ 通常倍率2倍 ＝ 合計5倍」 |
| 種別 = なし + 追加ポイント | 倍率入力ではなく固定ポイント数の入力欄が表示される |
| **対象の注文ポイント付与ルール**（必須） | このルールが適用される注文だけがキャンペーン対象。**作成後変更不可** |
| チャネル | 適用チャネルの条件。チャネル未接続環境では未入力時の実適用は未確認 |
| 種別「商品」の対象商品 | 「商品を選択して追加する」またはCSVインポート「キャンペーン対象商品」で投入可。「ブランドを指定して追加する」フォームもあるが、大量追加になる可能性があるため実投入は未実施 |

### そのほかのポイント設定

| データ | 効果 |
|:--|:--|
| 誕生日ポイント付与ルール | 誕生日に付与。「表示タイトル」が顧客のポイント履歴に表示される |
| 利用外商品 | 登録した商品が**ポイント利用の対象外**になる。ポイント付与対象外にしたい場合は、商品倍率を0倍にする |
| 失効通知 | 「失効日の◯日前」に顧客へ通知 |
| ポイント一括加算（CSV） | テナント＋顧客IDの種別（SQ発番 / 外部発番）を指定して残高を一括加算 |
| ポイント一括減算（CSV） | テナント＋CSVファイルのみ。顧客IDの種別は表示されない |

### 会員ランク算出ルール（会員ランク）

| 項目 / 値 | 効果 |
|:--|:--|
| 算出方法 | 現在は**購入金額固定** |
| 税抜き価格でランクを算出する = ON / OFF | 税抜 / 税込どちらで集計するか |
| 算出期間 = 1年間（＋開始月）/ 直近365日 / 無期限 | 集計の対象期間（例: 開始月4月 → 4/1〜3/31）。直近365日・無期限では開始月・月初算出・持ち越し設定が非表示になる |
| 会員ランクを月初に算出する = ON | 毎月1日に再計算される |
| 会員ランクを次の算出期間に持ち越す = ON | ランクを最大2年間維持できる |
| 配下: 会員ランク（表示名＋達成条件） | ランクの段階。**初期ランク「Bronze」（画面表示: `0円の獲得で達成`）は条件変更・削除不可**（表示名のみ変更可） |
| 配下: 除外商品 | その商品の購入金額をランク算出から除外 |
| ルールそのもの | 注文ポイント付与ルール・会員ランク倍率・ポイントキャンペーン（種別: 会員ランク）の**前提データ**。ポイントより先に作る |

## CSVインポート / エクスポート

### CSVインポート（22カテゴリ）

商品5（商品 / 商品バリエーション / 商品画像 / 商品バリエーション画像 / カタログ）・価格3（販売価格 (通常) / 販売価格 (セール) / 原価）・販売1（販売閾値）・在庫3（販売可能在庫 / 予約販売 / 在庫移動伝票）・メタフィールド2・出荷2（ヤマトB2クラウド / DHL）・ポイント3（一括加算 / 一括減算 / キャンペーン対象商品）・会員ランク1・各種マスター2

**ステータスは2軸**（実行履歴の各行をクリックすると確認できる）:

| ステータス軸 | 意味 | 失敗時 |
|:--|:--|:--|
| 検証ステータス | アップロードしたCSVの内容チェックの結果 | **検証失敗の行はSQに反映されない**。「検証失敗」件数のリンクから失敗詳細画面で行ごとのエラーを確認 → CSV修正 → 新規インポートからやり直し |
| 実行ステータス | データ反映の結果 | 「成功 完了」になると**「実行する」ボタンが無効化され、同じインポートは再実行できない** |

> ⚠️ 検証ステータス・実行ステータスの全選択肢（成功以外の表記）は、失敗CSV等の追加データ依存のため未検証。

| 項目 / 値 | 効果 |
|:--|:--|
| 検証 → 実行の2段階 | 「保存する」で検証が走り、検証成功後に「実行する」で反映 |
| 販売可能在庫の反映方法 = **絶対値で反映する**（初期値） | CSVの数値で在庫を**上書き**（在庫10でCSV20 → 20） |
| 販売可能在庫の反映方法 = **差分値で反映する** | 現在値に**加算**（在庫10でCSV20 → 30） |
| 商品画像の処理方法 = 画像を追加する / 画像を上書きする | 既存画像を残すか、置き換えるか |
| ポイント一括加算の顧客IDの種別 | SQが発番した顧客ID / 外部システムが発番した顧客ID。一括減算にはこの項目は表示されない |
| ポイントCSVインポートの履歴 | 一括加算・一括減算とも、検証/実行の2軸ステータス＋失敗詳細画面を持つ |
| テンプレート | 列順・ヘッダー名を変えると失敗。テンプレートが無いカテゴリもある（販売価格 (通常) など） |
| 在庫移動伝票 | `/admin/csv_import/csv_import_operation_inventory_movement_orders/create` からファイルをアップロードする。空保存時は「ファイルを選択してください」 |

### CSVエクスポート（CSVエクスポート画面）

| 項目 / 値 | 効果 |
|:--|:--|
| カテゴリ（在庫 / ロケーション / ディスカウント利用履歴 / 商品バリエーション / セール価格 / ヤマトB2クラウド / 売上実績（注文軸）/ 売上実績（明細軸）/ ポイント変動履歴） | カテゴリごとにロケーション・ロケーショングループ・テナント・期間などを指定する。非同期処理で実行され、対象カテゴリの一覧に実行履歴とダウンロードリンクが残る。メール通知の有無・文面は未確認 |

### PDFエクスポート（PDFエクスポート画面）

| 項目 / 値 | 効果 |
|:--|:--|
| PDFエクスポート画面（CSVエクスポートとは別画面） | 納品書PDFのエクスポートカテゴリを確認する画面。トップには「指定された出荷指示の納品書をエクスポートできます。」と表示される。2026-06-21時点の検証環境では新規作成ボタンは表示されない。直接 `/admin/pdf_export/pdf_export_operation_packing_slips/create` にアクセスしても404相当の画面になり、出荷完了済みの出荷指示詳細にも納品書/PDF生成ボタンは確認できなかった。なおヤマトB2条件指定エクスポートフォームには「同梱する納品書PDF」の文言があるため、PDFは同フォームの非同期出力物として生成される可能性があるが、実ファイル確認は未済。納品書のレイアウトは設定 > 納品書（事典①）で別管理 |

## 横断早見表

### 作成後に変更できない項目（運用編）

| データ | 変更できない項目 |
|:--|:--|
| 商品 | 商品コード |
| バリエーション | オプションの軸と種別（商品作成時のみ） |
| 販売価格ルール | 通貨 |
| ディスカウント | クーポンコード・テナント |
| ポイントキャンペーン | ポイント付与方法・対象の注文ポイント付与ルール |
| 会員ランク「Bronze」 | 達成条件（削除も不可） |

### 巻き戻せない操作

| 操作 | 対処 |
|:--|:--|
| 調整伝票の「実行」 | 逆向きの調整伝票で戻す |
| 出荷実績・入荷実績の登録 | 誤登録は調整伝票で修正 |
| 取置伝票の「処理済みとしてマークする」 | （確保が解放される） |
| CSVインポートの「実行」 | 再実行不可。修正版を新規インポート |
| 商品・ブランドの削除 | 復元不可（商品はアーカイブなら解除可）。商品削除後の詳細URLは「該当するProductが見つかりませんでした。」になる |
| 移動伝票のキャンセル | 伝票は戻せるが**元の在庫依頼が行き止まり**になる → 在庫依頼を新規作成 |
